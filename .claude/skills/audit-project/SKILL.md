# 项目漏洞评估（audit-project）

用**推导链多角色 subagent** 给项目做全面体检。每个角色产出一个**文档落盘**，下游读上游文档继续推导——和 wiki「单向推导」同构，和 superpowers 文档驱动同风格。末端把漏洞落 backlog + 重排路线图。

只读评估——不动代码，不动 wiki 内容；只产评估文档 + 落 backlog + 重排路线图。末端接 `idea-to-roadmap`。

## 何时用

用户想摸项目底子、找漏洞/欠账/风险、做健康度体检时。**不**用于：单点 bug 修复（走 `systematic-debugging`）、单层重构（走 `roadmap-delivery-workflow`）、已知点子归类（走 `idea-to-roadmap`）。

## 输入

- **范围**：默认全项目（代码 + wiki 文档 + 流程）。用户可缩到某层（如"只评估后端"）——缩范围时跳过链上无关角色，但保留的仍按推导链顺序跑。
- **聚焦**（可选）：用户点名的关注点（如"重点看回滚"）——注入每个角色提示词的"额外关注"。
- **主题**（可选）：本次评估的简称，用作目录后缀；没给就用日期。

## 核心思想：推导链

```
产品 → 用户 → 架构 → [前端 ∥ 后端] → 测试 → 汇总
 定义     验真    设计      落地          回溯    合并
```

- **上游文档 = 下游输入**：每个 subagent 读上游落盘文档，产出自己的文档。不是平级并行乱扫，不是主流程在内存里传话。
- **单向引用**：下游文档只引用上游文档（呼应 wiki 单向推导）。评估记录因此可追溯、可审计。
- **推导断节**：每个角色除扫本领域漏洞外，必须查"我与上下游的推导关系断没断"。这是 wiki「单向推导/单源」在评估上的落地，也是本项目最大类漏洞（记忆里的"G-debt 病根=推导链断节"）。
- 多角色交叉命中同一条 → 汇总阶段升它的置信度和严重度。

## 目录结构

每次评估一个目录，按日期+主题命名：

```
docs/audits/YYYY-MM-DD-<主题>/
├── 00-锚定-产品.md      # 产品产出：锚定单 + 产品自扫漏洞（链头）
├── 01-验真-用户.md      # 读 00，从用户视角验真 + 扫体验漏洞
├── 02-设计-架构.md      # 读 00+01，评估架构是否服务需求
├── 03-落地-前端.md      # 读 02，评估前端落地（与 04 并行）
├── 04-落地-后端.md      # 读 02，评估后端落地（与 03 并行）
├── 05-回溯-测试.md      # 读 00-04，回溯验证整条链
├── 06-汇总-合并.md      # 读 00-05，去重合并 + 故事化总结（给人看）
└── findings.yaml        # 最终结构化漏洞清单（给 backlog 用，机读）
```

数字前缀保证排序 = 推导顺序。前端 03 / 后端 04 都依赖 02，编号先后无妨。

```markdown

> 上游文档：<相对路径列表，如 00-锚定-产品.md>
> 本文档：<一句话角色职责>

## 给下游的提示
```

## 流程

### 前置·worktree 隔离
评估应在 worktree 内跑（并发隔离 + 不污染 main）。**关键坑**：subagent 的 cwd 不一定继承 worktree，用相对路径写文件会落错地方（落主仓库）。所以 `{{AUDIT_DIR}}` 必须注入 **worktree 绝对路径**（如 `/home/${username}/dicelore/.claude/worktrees/<name>/docs/audits/YYYY-MM-DD-<主题>`），subagent 一律用此绝对路径读写。主流程建目录时也用该绝对路径。

### 阶段 0·建目录 + 锚定
主流程用 worktree 绝对路径建 `docs/audits/YYYY-MM-DD-<主题>/`。派**产品** subagent（单独一轮，注入 worktree 绝对路径作 `{{AUDIT_DIR}}`），产出 `00-锚定-产品.md`：用户画像 / 核心场景 / 需求边界 / 验收口径 + 产品自扫漏洞。**没有这份锚定单，下游无从验真。**

### 阶段 1·推导链扫描
按链顺序派 subagent，每个读上游文档、写自己的文档：
- 用户读 `00` → 写 `01`
- 架构读 `00`+`01` → 写 `02`
- **前端∥后端**：用 Agent 工具在同一消息并行派发；都读 `02`，分别写 `03`/`04`
- 测试读 `00`-`04` → 写 `05`

### 阶段 2·汇总合并
派 1 个**汇总** subagent，读 `00`-`05` 全部文档，产出：
- `06-汇总-合并.md`：故事化总结，按严重度/类别排序，标出多角色交叉命中项
- `findings.yaml`：机读的最终漏洞清单

合并规则：按 `title 相似度 + evidence 位置` 去重；多角色命中的同一条 → `confidence` 升高、`severity` 取最高、合并各方 `evidence`、`hit_by` 列出命中角色。

### 阶段 3·归类入池
基于 `findings.yaml`，按 `layer` 字段落 backlog 三池 `docs/wiki/06-里程碑与问题/backlog-<层>.md`：
- `前端`→backlog-前端、`后端`→backlog-后端、`core`→backlog-core
- `跨层`/`文档`/`流程`/`推导断节` 类 → 归最相关池，条目标注「跨层/推导断节」
- 与既有条目比对去重，别重开（复用 `idea-to-roadmap` 口径）

### 阶段 4·重排路线图
**调用 `idea-to-roadmap` skill** 把新落账条目编进/重排 `路线图.md`：推导断节 + 随规模恶化的 = 最高优先级，进靠前批次。

### 阶段 5·收尾
开分支提交（`git checkout -b docs/audit-<日期>`，message `docs(audit): ...`），切回 main `git merge --ff-only`、删分支。**不问"要提交吗"，不 push。** git 一律 `--no-pager`。评估文档目录随本次提交一起入库。

## findings.yaml 字段（6 角色统一）

```yaml
- finding_id: <角色简称-序号>   # PROD/USER/ARCH/FE/BE/QA
  role: 产品|用户|架构|前端|后端|测试
  title: <一句话>
  kind: 领域漏洞|推导断节
  layer: 前端|后端|core|跨层|文档|流程
  severity: P0|P1|P2|P3
  evidence: <file:line 或 wiki 页路径>
  description: <是什么>
  impact: <会怎样>
  suggested_fix: <建议方向，不写实现>
  confidence: 高|中|低
  hit_by: [角色]   # 汇总阶段填：哪些角色命中了这条
```

## 6 个角色 subagent 提示词模板

派发时填入 Agent 工具 prompt；`{{AUDIT_DIR}}`（**worktree 绝对路径**，见「前置·worktree 隔离」）/`{{聚焦}}` 由主流程注入。每个 subagent 自己用 Write 落盘自己的文档——**必须用 `{{AUDIT_DIR}}` 绝对路径写，别用相对路径**（相对路径会落错 cwd）。

### 产品（PROD）— 链头
```
你是 Dicelore 项目的产品负责人（PO），对"做什么/不做什么/先做哪个"负责。你是评估推导链的链头，你的产出约束整条链。

任务：读 docs/wiki/01-业务分析、02-领域模型、06-里程碑与问题/路线图.md（路线图 + backlog 三池 + 里程碑），产出【锚定单】+ 产品自扫漏洞，写到 {{AUDIT_DIR}}/00-锚定-产品.md，按角色文档模板。

锚定单必含：
- 用户画像：有哪几类用户/玩家，各自要什么
- 核心场景：从头到尾的关键玩法/使用闭环
- 需求边界：承诺做什么、不做什么
- 验收口径：每个核心场景"做成了"怎么判定

主扫漏洞：
- wiki 推导链断节（业务→概念→设计→实现 哪节断了）
- backlog 三池里未归类/去重不彻底/聚类缺失的欠账
- 路线图批次与 backlog 是否对齐、"冻结论"未执行
- 设计-实现漂移、验收口径缺失的功能
- **wiki 定位漂移**：批判性审视 wiki 业务定位是否与项目最新决策/实际一致——品类词是否过时、用户分型是否贴合目标市场、已弃场景有没有下游没跟。你是链头，有责任第一个质疑 wiki 定位，而不是照抄。发现"wiki 说的定位 vs 最新决策/项目实际"的漂移，作为推导断节报。

推导断节检查（必查）：你定义的需求，后续（架构/实现/测试）有没有承接？有没有"定义了但无人实现"的需求？wiki 定位本身有没有被下游真正落地，还是只停留在文档？

横切：文档单源违例。
{{聚焦}}
项目锚点：docs/wiki/、apps/、packages/、reports/。
```

### 用户（USER）
```
你是 Dicelore 的终端玩家，不读代码细节，只关心"我能不能顺畅玩一局/用起来"。你是产品下游，基于产品的锚定单验真。

任务：读 {{AUDIT_DIR}}/00-锚定-产品.md，产出 {{AUDIT_DIR}}/01-验真-用户.md，按角色文档模板。

主扫维度：
- 核心场景能否跑通（开局→推进→掷骰→选择→存档→回滚→读档）
- 体验断点（GM 卡死/无响应/回复崩坏/状态丢失）
- 边界场景（长对话/空输入/异常选择/中断恢复）
- README 对用户承诺的功能是否真能用

推导断节检查：产品定义的"用户是谁"是否完整？有无未被识别的用户类型？产品列的核心场景，从用户角度看是否真实成立、有无遗漏？

横切：用户能否注入/越权看别人存档。
{{聚焦}}
项目锚点：README.md、README.zh-CN.md、apps/web、.dl-eval-doctrine/。
```

## 硬约束

- **只读评估**：不动代码、不动 wiki 内容；只产评估文档 + 落 backlog + 重排路线图。实现走 `roadmap-delivery-workflow`。
- **文档驱动推导**：每个角色必须把产出落盘成文档；下游必须读上游文档推导，不许主流程在内存传话。文档单向引用（下游引上游）。
- **推导链不可乱序**：产品必须先跑出 `00-锚定-产品.md`；下游按链顺序读上游文档。平级并行只允许前端∥后端。
- **推导断节必查**：每个角色文档必须有「推导断节」小节，不可只扫领域漏洞。
- **末端接 `idea-to-roadmap`**：阶段 3-4 复用 idea-to-roadmap 的去重聚类口径和路线图编排规则，别另起一套。
- **落盘即提交合并 main，不 push**：不问"要提交吗"；git 一律 `--no-pager`。评估文档目录随提交入库。
- `路线图.md` AI 维护可重排；`里程碑.md` 人工维护、AI 不动。
- 派 subagent 用 Agent 工具；前端/后端在同一消息内并行派发以节省墙钟时间。
- subagent 可写自己那份评估文档，但**不碰 git**（分支/提交/合并由主流程做）。
- 并发会话时评估的写入操作应在 worktree 内隔离。

# 设计：团本声明式工具生成层（lore/dice 双 agent 的扩展机制）

> **状态**：草案待评审（2026-06-22）。
> **一句话**：让团本**声明**(而非编码)自己要的业务工具——名字 + 权限 + 参数类型 + 查询/写绑定——由框架构建器在跑团时**编译成 session 级 MCP 工具**。新剧本 = 新声明 + 新 flow skill + 新数据，**框架代码不动**。
> **上游**：[运行时数据层重构-叙事层 design](2026-06-21-运行时数据层重构-叙事层-design.md)（视图层是本层的供给方）、[MCP工具面](../../wiki/04-子系统设计/MCP工具面.md)（生成工具并入同一工具面 + 复用 in-process 工厂）、[团本与manifest](../../wiki/04-子系统设计/团本与manifest.md)（manifest 增声明段）、[团本构建工具链](../../wiki/04-子系统设计/团本构建工具链.md)（构建页面校验）。
> **下游**：无（本层是扩展机制的终点）。
> **不在本 spec**：声明 DSL 的完整文法（留 plan）、命名视图作者自定义（YAGNI，见 §10）、数据层/视图层本身（上游 spec）。

## 1. 背景与动机

### 1.1 病根：框架随剧本无限膨胀的诱惑

每来一个新剧本，都会要某种"业务表"——派系表、任务表、升级表、商店表……朴素做法是**每个都加一张物理表 + 一批专属 MCP 工具**，框架于是无限膨胀、永远在追剧本。

但拆开看，这里其实是**三件成本差一个数量级的事**，只有一件需要新机制：

1. **数据形状**：派系=npc 实体 + relation；升级=state attr + rule + pool；商店=pool + state 变更。**大多已被上游数据层的 `state`(kind)/`pool`/`lore` 吸收**——零框架改动（见上游 spec §3 归并）。
2. **读的长尾投影**：剧本各要不同的过滤/投影（商店按价过滤、任务按状态列、派系按声望排序）。
3. **题材业务动词**：`shop_buy` / `quest_advance` / `rep_change` 这类**写**操作。

②③ 才是"扩展"真正缺的机制。而它们**不能靠运行期 agent 现造代码**——[MCP工具面 §7](../../wiki/04-子系统设计/MCP工具面.md) 的红线是"限制工具面本身即 L1"：裸骰子永居内层、AI 调不到，靠的就是工具面**封闭、预定义**。运行期造工具 = 砸穿 anti-F1。

### 1.2 核心判断

扩展应是**构建期声明 → 框架编译**，不是运行期造码。团本写的是**数据(声明)**，框架是**唯一的代码路径**。这既给了无限扩展性，又把安全边界焊死在框架手里。

业界锚点：PostgREST（声明表+视图+权限 → 自动生成 REST API）、Hasura（声明权限 → 自动生成 GraphQL）。本层把生成目标换成 **MCP 工具**，是同一成熟模式。

## 2. 两个 agent，两个时刻（词汇 + 扩展边界）

| | **lore_agent**（构建期） | **dice_agent**（运行期） |
|---|---|---|
| 干什么 | 产出团本目录：world/rules/pools/sheets/fronts + **写 flow skill** + **写工具声明** | 跑团：调 MCP 工具、`query` 视图、`narrate`、裁决 |
| 能扩展什么 | **几乎全部扩展住这里**：声明工具、写 flow skill、定数据 | **不发明工具/视图**，只在 lore_agent 铺好的盘 + 框架原语里操作 |
| 写权限 | 写**包**（skills/结构/规则/初值，`source=author`） | 写**数据**（`source=ai`：`sheet_update`/`world_register`），不碰包 |

> **这条边界对齐既有机制、非新发明**：恰压在 [团本与manifest §8](../../wiki/04-子系统设计/团本与manifest.md) 的两写源 + `source` 列上。"运行期造工具不行" = 让 dice_agent 去干 lore_agent 的活、跨了时刻。
>
> **依赖方向（单向推导）**：设计顺序可消费方优先（本 spec 先 draft，把需求反向拉给数据层 spec）；**实现/闭环顺序必须上游先行**——视图层先建，才谈得上在视图上生成工具。
>
> **术语对齐待办**：现有 wiki 用「团本作者」「GM/AI」称这两侧。采用 lore/dice 这对名须回 [术语表](../../wiki/02-领域模型/术语表.md) 对齐，别造第二套叫法（单源）。本 spec 暂用 lore/dice 作工作词汇。

## 3. 拱心石：声明式工具生成

```
团本声明（数据）              框架构建器（唯一代码路径）         运行期
┌──────────────┐            ┌────────────────────┐         ┌──────────────┐
│ name         │            │ 校验声明            │         │ session 级    │
│ permission   │  ──────►   │ 编译：             │  ──────►│ MCP 工具      │
│ params(类型) │            │  R → 视图查询       │         │ (有名/有类型/ │
│ query/binding│            │  CUD → 正典写工具门面│         │  有权限/有描述)│
└──────────────┘            └────────────────────┘         └──────────────┘
```

团本**不写代码、只写声明**；框架构建器是**唯一**把声明变成工具的代码。生成的每条声明 = 一个**有名字、有描述、有类型入参、有权限**的 MCP 工具（比"一个通用 query"强：通用 query 无名、载不了教条，dice_agent 得自己琢磨；命名工具把教条价值赚回来）。

## 4. 承重墙：读 / 写编译到不同地方

这是整套方案安全性的命根。**R 和 CUD 必须编译到不同目标**：

| 权限 | 编译目标 | 为什么不能反 |
|---|---|---|
| **读 R** | 团本 `sql` 的 `SELECT`（只读）跑，**可引用视图 + 物理表**，参数绑定 | 只读不改库 → 无绕过风险；引物理表换灵活、团本自担 schema 耦合（优先建在视图上更稳）；绑定参数 → 无注入 |
| **增删改 CUD** | 团本 `sql` 的 `UPDATE/INSERT/DELETE` 被**解析 + 匹配到正典写函数原语**（`applyMutations` 等），匹配不上即拒 | 裸 `UPDATE` 会绕过 **watcher 比对**（front/钟全失灵）、**不落 event**（L3 瞎）、**跳过 expr 引擎**（anti-F1 破）、漏 visibility。写**永远经原语**、决不裸跑 → 防泄露 |

> **「正典写原语」= 内部函数，不必是 AI 直调工具**：CUD 的编译目标是 `applyMutations` 这类**内部**写函数（落 event/触发 watcher 处），而非"AI 能直接点的 MCP 工具"。事实上 AI 面向的工具面正收敛为**纯业务/类型化**——读走 `player_card`/`npc_list`/`world_state`/`query`，写走 `player_update`/`npc_update`/`world_update`（**kind 由工具名携带**）+ 生成的 `shop_buy`……；kind 无关的裸 `sheet_update` 是删除候选。**删 MCP 工具 ≠ 删底层函数**——即兴写（给 NPC 现起属性）走 `npc_update`，仍踩在同一守卫写路径上。

**推论（重划 skill 的边界线）**：分支**可以有**——但线在**「声明式 vs 编码」**、不在「有无」。关键是**谁来跑这个分支**:

| 分支放哪 | 例（商店买东西） | 为什么 |
|---|---|---|
| **进生成工具**（有界声明式：前置守卫 + 按值选档，**框架求值器**跑） | "买不起就拒绝，否则扣钱+给物" | 守卫+写在**一个事务内原子完成**，比 AI「先查再写」跨两次调用更安全（无 TOCTOU）；求值器是框架的、非作者码 → **承重墙不破**，写仍落正典原语 |
| **进 flow skill**（叙事编排 / 判断 / 跨回合 / 任意控制流） | "店主的反应叙事、要不要触发讨价还价小场景" | 这是 GM 判断与散文，非数据操作；任意控制流会让声明变成图灵完备的「换语法的代码」 |

**先例**：`resolve_outcome` 的 `bands`（rangeMap 按值落一档）、watcher 的 `condition` 谓词、`sheet_update` 的带骰 expr——**框架早在跑声明式分支**，生成工具复用同一套求值器即可。

于是团本仍带两样，但分工是**「有界声明式分支 vs 编排逻辑」**、不是「有无分支」：

- **工具声明** = 动词 + **有界守卫/选档**（`shop_buy` 内含"够钱才买"）：框架编译 + 框架求值，有名/有类型/有权限/**原子**。
- **flow skill** = 用这些动词写的**编排过程**（提示词，指导 dice_agent 何时调哪个、调完怎么叙事）。

## 5. 声明 schema（草案，文法细节留 plan）

**两类独立声明**——团本写 **SQL-like**，框架是**唯一解析者**（决不裸喂 sqlite）：

### ① 视图定义

```json
{ "name": "shop_inventory", "sql": "SELECT name, price, stock FROM pool WHERE pool = '商店货架'" }
```
- `sql` = 单条**只读 `SELECT`**，**可引用物理表 + 已定义视图**。框架校验只读（禁 `INSERT/UPDATE/DELETE/DROP/PRAGMA/ATTACH`、禁 `;` 多语句）→ `CREATE VIEW <name> AS <sql>`。
- 视图天然只读；引物理表换来灵活、但**耦合物理 schema**（改物理表要回头改该视图）——**优先建在基视图上**对稳定性更好，引物理表团本自担。

### ② MCP 工具定义（**只有一个 `sql`**）

```json
{
  "name": "shop_buy",
  "desc": "玩家买一件商品",
  "params": { "buyer": "string", "item": "string", "price": "int" },
  "sql": "UPDATE player SET 金币 = 金币 - :price WHERE entity = :buyer AND 金币 >= :price"
}
```
- `sql` **一条语句**，可引用**所有已定义视图 + 物理表**；`:param` 绑定（`params` → 生成工具 inputSchema `.strict()`）。框架按 **sql 首词分流**：
  - **`SELECT` → 读工具**：校验只读 + `db.prepare(sql).all(boundParams)`，回行。只读安全。
  - **`UPDATE/INSERT/DELETE` → 写工具**：**解析 + 匹配到正典写函数原语**派发，**绝不裸跑 → 防泄露/防绕过**（保 event/watcher/expr/visibility）。匹配不上的写形状（多表 JOIN 写、复杂 WHERE、无对应原语）**一律拒**（构建/import 期报错）。
- **守卫**走 `sql` 的 `WHERE`（如上 `AND 金币>=:price`，不满足 → 0 行受影响 = 不生效，原子）；**多动词/分支编排**走 flow skill（§4 推论）。

**可匹配的写模式（封闭集；匹配器只认这些 → 原语，其余拒）：**

| 团本 `sql`（写） | 匹配到的正典原语 |
|---|---|
| `UPDATE <state视图/kind表> SET <attr> = <attr> ±/= :p WHERE entity=:e [AND …守卫]` | `applyMutations(e, [{attr, op, expr}])` → 落 event/触发 watcher |
| `UPDATE <叙事表> SET status=:s WHERE id=:i` | `<front/plotline/foreshadow>SetStatus(i, s)` |
| `INSERT INTO <叙事表>(…) VALUES(…)` | `<…>Upsert(…)` |

> **本质**：SQL-like 是统一表面语法，**后台解析器是安全闸**——读只读、写必经原语。物理表对读开放（团本自担耦合），对写永远关闭（只能经原语）。

## 6. 命名与冲突（无 `<id>` 前缀）

跑团时**只加载一个团本**，团本 id 在 session 内是恒量、运行期零消歧价值，还是 dice_agent 打错的料。故：

- **生成工具用短、裸、业务名**（`shop_buy`），与框架工具一个风格。
- **类别**（框架正典 vs 团本生成）**走 MCP annotation/metadata**（如 `source:"generated"`），并列 [MCP工具面 §7.1](../../wiki/04-子系统设计/MCP工具面.md) 已有的 `readOnlyHint` 等——**类别靠元数据，不污染名字**。
- **撞名/冒名靠校验**（§7）：保留框架工具名为禁用区 + 多个声明包之间重名直接报错，**逼作者构建期改名**。

> 代价：翻原始 tool-call 日志时光看名字分不出是不是生成工具，得看 metadata——可接受。

## 7. 双校验（DX + 安全闸门，规则单源同享）

| 校验点 | 为谁 | 性质 |
|---|---|---|
| **团本构建页面**（authoring 时） | lore_agent / 人类作者 | **早反馈、为好用**：边写边校、报错友好可定位。归 [团本构建工具链](../../wiki/04-子系统设计/团本构建工具链.md) |
| **import / 建库**（运行期装载时） | 框架自己 | **信任闸门、为安全**：权威、不可绕过、**永不假设页面已校过**。归建库/import 路径 |

- **为什么 import 这道砍不得**：包是纯文本目录，可能被手改、来自旧版页面、git merge 畸形。import 必须当作"面对不可信的包"重验——这是信任边界（client/server 校验的经典关系）。
- **承重约束（单源）**：两道**共用同一 validator 模块**——构建页面和 import 都调它，不各写一份逻辑然后漂移。原则：**规则单源同享；页面宽松友好、import 权威不可绕；import 永不信任页面。**

## 8. 标准库（默认配置包）

框架预置一批声明包（商店 / 任务板 / 派系声望 / 打造 / 升级），团本 `include` 后微调即用。三个作用一次满足：

1. **开箱即用**：常见题材 batteries-included。
2. **当范例教 lore_agent** 怎么写声明。
3. **graduation 通道**：某 pattern 在多个包里反复出现 → 提拔进框架核心工具。

> **框架自己的叙事层 = 第一个标准库包（dogfooding）**：[上游数据层 spec](2026-06-21-运行时数据层重构-叙事层-design.md) 的 front/plotline/foreshadow **业务工具**不硬编码，而是用本层**声明**出来（实现三步的 step ③）。这是对 DSL 表达力**最强的压测**——连框架通用叙事都能声明，题材长尾自然能。**预期会撞到 DSL 天花板**（如 `foreshadow.surface_trigger` 浮现、谓词扩展）：届时要么扩 DSL、要么在声明底下垫一层框架原语。这是设计的一部分、非意外。

## 12. 验收（怎么算成功）

- **扩展性**：新增一个题材剧本（如带商店+任务+派系声望），**框架代码零改动**——只新增声明 + flow skill + 数据。
- **安全**：生成的**写**工具仍触发 watcher、仍落 event、仍走 expr 引擎（守卫不破，可测：生成的 `shop_buy` 推进某 clock 时 front 凶兆照常触发）。
- **校验**：坏声明（引用不存在的视图 / 写工具引用非正典原语 / 撞框架名）在**构建页面和 import 两处都被拦**，且报错措辞一致（共享 validator）。

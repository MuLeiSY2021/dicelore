## 1. 背景与动机

### 1.1 病根（来自 50 回合 faithful eval + 问题总账）
同一概念在**团本作者层**有一等形态，到**运行时跑团层**却塌缩成底层存储原语（散落 sheet 格 / world_doc / 裸 watcher），AI 拿不到「以这个概念为单位」的**读（聚合视图）+ 写（生命周期）**。这是 conceptual integrity 问题，跨多 session 反复命中、随回合数线性恶化（问题总账判定为头号架构债）。

50 回合兽人局实测佐证（客观、与「谁驱动」无关）：
- **B6（最强·线性恶化）**：T30 时 4 条剧情线 + 2 个 Front 散落在 5 实体暗格 + 3 个 armed watcher（经 MCP 完全不可见）+ 13 条 note（只能 FTS 搜），无任何聚合视图。
- **B1/B2/B3**：剧情线靠假实体硬凑、伏笔靠 `note(visible:0)` + FTS 硬搜（排不过叙事噪音）、触发器只认数值阈值且预写 payload 与实际场景失配。
- **A′ 地基**：`sheet`（实质=「当前可变状态」的统一载体）被误当「人物卡」，与团本侧术语不对齐。

### 1.2 核心判断
病根不是「存储存错了」，而是**缺类型化视图 + 缺业务场景工具**。故方案：**物理层精简统一，投资全压在视图层与 MCP 工具面**。

---

## 2. 设计原则（拱心石）

三层 + 一根正交轴：

- **物理层**：少数几张表，存储唯一、不重复。
- **视图层**：用 `kind` + facet 投影出业务视图，**零存储**（单源，不复制状态）。
- **工具层**：一大批业务场景 MCP，让 AI 按场景调数据（疗效所在）。
- **正交轴「不变 / 当前」**：作者写死的（lore / rule / pool）读为主、写路径受限；随剧情变的（state / 叙事 / 记忆）高频可变。

---

## 3. 物理表（存储唯一）

| 表 | 关键列 | 兜住的概念 | 不变/当前 |
|---|---|---|---|
| **state** | `entity, kind(player\|npc\|world), attr, value, visible, rel_object?, rel_dim?, clock_min?, clock_max?, clock_mode?` | player / npc / world 当前状态 + relation（`rel_*` 非空）+ flag（world 的离散行）+ clock（`clock_*` 非空） | 当前 |
| **lore** | `name, content, category, tags, visible`（FTS5） | 作者世界散文底料（原 world_doc） | 不变 |
| **rule** | `name, content, version`（FTS5，AI 只读） | 机制/判定合同 | 不变（人改） |
| **pool** | `pool, row_json, weight, source, visible` | 卡池/随机表（原 world_pool） | 不变（GM 可现编） |
| **front** | `id, name, stakes, clock_ref(entity,attr→state), status(active\|spent)` | 威胁层（引用一个 clock） | 当前 |
| **front_omen** | `front_id, threshold, payload, fired`（front 子表） | 凶兆阶梯，每行=一阈值触发 | 当前 |
| **plotline** | `id, title, summary, status(open\|active\|closed), anchors_json(→实体/front)` | 故事线 | 当前 |
| **foreshadow** | `id, content, status(planted\|recalled\|abandoned), surface_trigger?, anchors_json` | 伏笔 | 当前 |
| **log** | `seq, kind, content, data_json, visible, is_moment` | append-only 原始流水（原 event）+ moment 标记列 | 当前 |
| **history** | `id, seq_from, seq_to, summary, created_seq` | compact 蒸馏记忆 | 当前 |
| **watcher** | （现有，不变） | 底层数值触发器 | 当前 |
| 基建不动 | `session_meta / pending_choice / pending_roll / *_fts` | tone/id、选项暂存、待掷暂存、FTS 影子 | — |

**归并说明**（用处分析的结论）：player/npc/world/relation/flag/clock **六个概念塌成一张 `state`**——它们是同一形状 `(实体,attr,值,可见)`，差别只在 kind 与少量元数据列；relation/flag/clock 不是新表，是 state 的**行形态 + 视图/工具**。

---

## 4. 视图层（零存储，按 kind+facet 投影）

> **视图是裁剪过的投影，不是 `SELECT *`**：每个视图**只保留与该业务相关的列**，丢掉恒定/无关列，给 AI 一个干净业务面。

| 视图 | 行过滤 | 投影列（裁剪后） | 治 |
|---|---|---|---|
| `player` | `kind=player` | entity, attr, value, visible（**去 kind**：恒为 player；去 rel_*/clock_*） | — |
| `npc` | `kind=npc` | entity, attr, value, visible（**去 kind**；含会行动的派系；双层值=按 attr 各自 visible） | A1 |
| `world` | `kind=world` | entity, attr, value, visible（**去 kind**；含 flag、未挂 front 的 clock） | — |
| `relation` | `rel_object is not null` | entity(主体), rel_object, rel_dim, value, visible（**去 clock_***；可查「X 的所有关系」「谁对 Y 敌意≥N」） | A6 查询侧 |
| `clock` | `clock_* is not null` | entity, attr, value, clock_min, clock_max, clock_mode（**去 rel_***） | — |
| `tension_board` | 聚合 | active front + open/active plotline + planted-未回收 foreshadow + armed watcher | **A5/B6** |

> 同一物理实体可同时出现在多个视图（如「黄枫谷」既在 npc 视图作派系、又在 world 视图作地点）——视图是非互斥投影，边界问题消解。

---

## 5. MCP 工具面（业务场景）

- **state 读写**：沿用 `sheet_get/list/update`（按 `kind` 过滤），加类型化读 `player_card` / `npc_list` / `world_state`。
- **relation**：`relation_set` / `relation_query`（社交张力结构化查询）。
- **叙事**：
  - `front_open` / `front_advance`（推进 clock）/ `front_spend`；`front_board`。
  - `plotline_open` / `plotline_advance` / `plotline_close`。
  - `foreshadow_plant` / `foreshadow_recall` / `foreshadow_abandon`。
  - `tension_board`（一等聚合视图，B6）。
- **记忆**：`log`（原 `event_append`）、`mark_moment`、`history_compact`（agent 周期蒸馏）、`recall`（优先 history+moment，再兜底 FTS log）。

> 工具是「让 AI 按业务场景调数据」的接口面，背后全是物理表 + 视图，单源。

---

## 6. 叙事层细节

- **front**：引用一个 state clock；`front_omen` 每行=一个阈值→payload。**复用 watcher 的边沿触发机制**（内部实现），但对 GM 呈现为 front 一等对象——GM 不再裸调 watcher。修 A2/B3。
- **payload 语义改「凶兆意图」**（给 GM 的指令）而非成品散文 → 解 B3「场景失配」（数值到点时实际场景常偏离预写设定，让 GM 据当前 scene 现编落地）。
- **plotline / foreshadow**：显式 status 状态机 + 锚点；`tension_board` 列出所有未收口者，到点（foreshadow.surface_trigger 命中）浮现提醒 → 修 A3/A4/A5/B2。
- **触发器谓词扩展**（修 B3 的另一半）：watcher/front_omen 谓词除数值阈值外，支持「事件/选项/状态」式触发（如「玩家选了 X」「plotline Z 已开」）——具体谓词文法在实现计划展开。

---

## 7. 记忆层

`log`（全原始）→ `moment`（GM 标关键 log 行，`is_moment=1`）→ `history`（周期 compact）。
- `history_compact`：**agent 读一段 log、写一条压缩摘要**，优先保留 moment 标过的内容。
- **非向量 RAG**：合核心概念 §1「agent 即引擎」（不建 embedding）；核心概念 §5 推迟的是语义向量召回，本设计的 compact 是 agent 摘要，不冲突。
- `recall`：先查 history/moment，再兜底 FTS log → 解「埋的伏笔排不过叙事噪音」。

---

## 8. expr / watcher 引用约定（确认不改）

`{实体.attr}` **保持原样**，`kind` 只是标签、统一名字解析——`resolve_contest` 的 expr 解析、`watcher` 的谓词求值器**不改动**（低风险）。双层值仍靠 per-attr `visible`。

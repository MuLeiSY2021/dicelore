## 1. 定位

anko_driver 是一个**可嫁接在任意 AI agent 上的 GM 行为塑形系统**。

它不是游戏引擎，而是一套**让 AI 克服"讨好玩家"本能、按规则当一个公正且有戏剧性的 GM** 的框架。AI 的天然倾向是给好结果、软着陆失败、跳过骰子、回避冲突——这恰恰是坏 GM 的特征。本项目所有机制的核心目标，都是对抗这个本能。

核心信条：**AI 必须尊重骰子。玩家选择方向，骰子决定结果，失败是新剧情的起点。**

- **技术栈**：Python 全栈，MCP server + Skill 架构
- **项目路径**：`/home/mulei/anko_driver`
- **GitHub**：https://github.com/MuLeiSY2021/anko-driver

## 2. AI 不听话的三种失败模式

设计的一切都在对抗这三种失败：

| 代号 | 失败 | 例子 |
|------|------|------|
| **F1** | 跳过骰子 | 玩家"我攻击哥布林"，AI 直接写"你一剑砍倒了它"——没有判定、没有失败可能 |
| **F2** | 不尊重骰子结果 | 骰出大失败，AI 写"虽然失误了但你还是勉强命中"——软着陆 |
| **F3** | 选错处理方式 | 玩家"我接近神秘旅人"，AI 该给选项让玩家决定（搭话/绕路/偷袭），却直接骰了个 1d4 替玩家做主——剥夺能动性 |

## 3. 三层塑形模型

```
┌─────────────────────────────────────────────┐
│  L3 事后验证   Hook 审计骰子 vs 叙事一致性    │
├─────────────────────────────────────────────┤
│  L2 流程引导   Skill 编码 GM 决策流程         │
├─────────────────────────────────────────────┤
│  L1 结构强制   结果来源不可篡改（骰子/玩家/裁判）│
└─────────────────────────────────────────────┘
```

**L1 的本质不是"必然掷骰"，是"结果不能由叙述者自由编造"。** 无论结果来自骰子（`roll_outcome`）、玩家（`offer_choice`）还是无利害裁判（§5.1），都不是那个有讨好压力的叙述者拍脑袋写的。所以做选择的路径同样被 L1 覆盖——它的结果来自玩家输入，不是 AI 替玩家做主。

| 失败模式 | L1 结构强制 | L2 流程引导 | L3 事后验证 |
|---------|-----------|-----------|-----------|
| F1 跳过骰子 | 结果必经工具产出并记录，叙述者不能自定结果 | Skill 强制走判定/选择流程 | Hook 检查有无对应记录 |
| F2 不尊重结果 | **声明后果在先**（§6.1）；可选裁判定后果严酷度（§5.1） | Skill 明确禁止软着陆 | Hook 比对 outcome vs 叙事 |
| F3 选错方式 | 工具命名本身分流（roll vs choice） | **Skill 教三种机制的意义** + 决策框架 + 判例 | — |

**核心洞察**：L1 能防 F1、能用"声明后果在先"锁死 F2，但 F3（GM 判断）主要靠 L2。而 L2 的主载体是 **Skill**，不是更多 MCP 工具。叫停旧 Phase 2 的根本原因：它在往 L1 加码（用更多工具去约束），而项目真正需要的是把重心转向 L2。

**为什么 F3 必须靠"教意义"而非强制**：AI 选错（该给选项却替玩家骰），根因是它不理解三种机制的体验差别——

- **给选项**（`offer_choice`）= 把决定权交给玩家 = 尊重能动性，这是玩家的故事
- **掷骰**（`roll_outcome`）= 注入随机性/不确定性/趣味性 = 命运不可控，连 GM 也不知道结果
- **纯叙事** = 既无随机也无分叉的推进

Skill 的核心 dispatcher（§8 L0）必须把这三者的**意义和玩家体验**讲清楚，AI 理解了才会选对，光给决策清单不够。

## 4. 三层知识模型：框架重，规则书薄

教 AI 当 GM 的知识分三层，分工明确：

| 层 | 谁写 | 内容 |
|----|------|------|
| **模型共识** | 没人写 | TRPG 通用常识、叙事生动度、"d20 越高越好"、属性加值直觉——模型自带，不重述 |
| **框架教条** | 框架（厚） | anko_driver 的立场：尊重骰子、失败驱动剧情、何时给选项、怎么用工具。**这恰恰是模型最反共识、讨好本能最强、最不可靠的地方**，所以必须框架按住 |
| **规则书** | 作者（薄） | 仅团特有：世界观、基调、机制差异（2d6 还是 d20）、随机池数据 |

设计原则（取自 skill-creator 哲学）：**今天的模型很聪明，要解释 why 而不是堆 MUST，要靠模型已有的知识，别过度约束。** 通用 GM 行为由框架写一次，规则书一个字都不重复。

## 5. 指令架构

```
┌──────────────────────────────────────────────────┐
│  Skill（通用指令主载体，跨 agent 可移植）          │
│  教 AI 怎么当 GM：立场 + 流程 + 判例               │
├──────────────────────────────────────────────────┤
│  MCP（通用结构强制 + 状态）                        │
│  骰子、判定、随机、持久化——不可绕过                │
│  工具响应夹带运行时提醒（在掷骰那一刻补刀 F2）     │
├──────────────────────────────────────────────────┤
│  agent 定向优化（claude.json / hooks / 裁判等）    │
│  可选。只是对框架规则的引用，"为 xxx agent 调过"   │
└──────────────────────────────────────────────────┘
```

要点：

- **Skill 是框架核心，不属于某个 agent 的 adapter。** 它是通用的指令格式；adapter 只负责把 Skill/MCP 接到目标 agent 上。
- **送达机制 = 方案 3**：Skill 教（常驻流程）+ MCP 工具响应补刀（在掷骰当下把"别软着陆"钉回去）。最可靠会被 AI 读到的东西，是它刚调用的工具返回了什么——所以最滑的 F2 额外骑在不可绕过的工具响应上。
- **agent 专属配置是可选优化层**，不是地基。换 agent 不应导致 GM 教条失效。

### 5.1 可选：无利害裁判（impartial referee）

问题：叙述者 AI 身处对话中、有讨好压力，骰子/规则**没覆盖的判断**（定难度、定后果严酷度、解读自由行动的成败）正是它最容易软着陆的地方。

机制：把这些硬裁决从叙述者手里拿走，交给一个**置身事外、无利害的裁判**。它裁的不是"选哪个"（那是骰子/玩家的事），而是骰子定了**档位**（成功/失败/大失败）之后、**档位内的叙事后果严酷度**，以及行动的难度设定。叙述者会挑"掉了武器"，裁判会公正地挑"被反伤"。

**定位 = 可选优化层，不是核心：**

- 支持 subagent 的 agent（如 Claude Code 的 Task）：把"定难度/定后果严酷度"派给裁判 subagent
- 不支持的 agent：**优雅退回**到"声明后果在先 + Skill 禁令 + L3 Hook 审计"，功能不缺，只是少一道保险

这与"框架重 / agent 专属配置可选"一致：换 agent 不依赖裁判也能跑，有裁判则 F2/F3 更稳。**避免破坏零外部依赖**——裁判用宿主 agent 已有的 subagent 能力，框架不自带模型调用。

## 6. MCP 工具设计

工具分两类：**裁决工具**（决定一个结果归谁）和**检索原语**（取内容）。

### 6.1 裁决工具

两个语义工具，共用同一个 `options` 结构，且**都遵循"声明后果在先"**：声明全部选项的后果 → 外部裁决者（骰子 / 玩家）选 → 返回预先锁定的后果。工具名本身就在教 AI 分辨 F3（该骰还是该问玩家）。

**`roll_outcome`——骰子决定（对抗安科 / 纯安科 / 抽卡命中）**

AI 必须在掷骰**之前**声明完整的选项表 + 每个结果对应的叙事后果：

```
AI 传入：
{
  "options": [
    {"name": "月亮", "range": [1, 3], "consequence": "月亮对应…叙事结果…"},
    {"name": "太阳", "range": [4, 6], "consequence": "…"},
    {"name": "星星", "range": [7, 9], "consequence": "…"},
    {"name": "终焉", "range": [0, 0], "consequence": "…"}
  ],
  "dice": "1d10"   // 或继承规则书机制
}

MCP 返回：
{"roll": 7, "hit": "星星", "consequence": "星星对应…叙事结果…"}
```

**这是对 F2 的结构强制**：AI 在看到结果前就把"失败/终焉意味着什么"白纸黑字交出去了，结果回来时它已无法软着陆——后果是它自己刚声明的，响应又原样钉回来。玩家只看到精简版（`1-3 月亮 / 4-6 太阳 / *r=7 星星`），AI 内部传递的是完整后果。

返回里附带运行时提醒（方案 3 补刀），例：`"reminder": "按你声明的后果叙述，不要软化失败。"`

**`offer_choice`——玩家决定（安价 / 自由安价）**

与 `roll_outcome` 对称：**AI 在玩家选择之前，就声明每个选项的（隐藏）后果**——因为给出选项的那一刻，设计就已经做好了，每个选项通向什么是 GM 预先定的，不是玩家选完才临时编。

```
AI 传入：
{
  "options": [
    {"name": "搭话", "consequence": "旅人警惕，但透露一条线索…"},
    {"name": "绕路", "consequence": "避开冲突，但错过这条线索…"},
    {"name": "偷袭", "consequence": "占先手，触发战斗，旅人怀恨…"}
  ]
}

MCP 返回（玩家选了"绕路"后）：
{"chosen": "绕路", "consequence": "避开冲突，但错过这条线索…"}
```

玩家只看到选项标签（搭话 / 绕路 / 偷袭），看不到后果——像真实决策一样盲选。**这把 F2 也挡在了选择路径上**：后果在玩家选择前已锁定，AI 没法在知道玩家选了什么之后再投其所好地软着陆。

- **单人模式**：转交轮次，等这一个玩家回应后，返回其所选选项的锁定后果（自然对话，不算阻塞）
- **多人模式**：开投票窗口，聚合票数得出众选选项，返回其锁定后果（真阻塞）

### 6.2 检索原语

**`knowledge_sample`——随机取内容（knowledge_search 的随机兄弟）**

抽卡等"随机生成"不是特殊范式，而是组合：

```
AI 决定抽卡 → knowledge_sample(filter={类型:"卡牌", 颜色:"蓝色", 类别:"人物"}, n=3)
         → MCP 在匹配集合中随机命中，返回 3 张
         → AI 把结果塞进 roll_outcome / offer_choice 的选项，或直接叙述
```

**关键的防作弊约束**：随机命中由 MCP 执行，AI 只给过滤条件，**不能 search 完自己手动挑**（否则会挑对玩家有利的卡，退回讨好本能）。这与"声明后果在先"是同一个精神。

`knowledge_sample` 可单独用（纯生成：抽 3 张全留），也可喂给裁决工具（抽一堆→玩家选 / 抽一张→定结果）。

> 留待实现细节：稀有度加权抽卡——靠知识库条目的 `稀有度` 属性 + 加权采样。

## 7. 范式系统：从代码概念溶解为流程组合

旧设计的 **6 种代码强制范式被删除**。它们不再是代码里的枚举，而是 **2 个裁决工具 + `knowledge_sample` + 自由叙事**的组合，由 Skill 教 AI 何时用哪种：

| 旧范式 | 新实现 |
|--------|--------|
| dice_resolution / pure_anko | `roll_outcome` |
| random_generation（抽卡） | `knowledge_sample` (+ 裁决工具) |
| option_anka / free_anka | `offer_choice`（单人转轮 / 多人投票） |
| free_narration | 不调工具，AI 直接叙述 |

"范式"从代码概念降级为 **Skill 里的教学词汇**。

## 8. Skill 体系：框架预设的、对 AI 友好的可复用流程库

这是项目的核心资产，也是"框架重、规则书薄"的终极形态。

| 层 | 何时加载 | 是什么 | 谁写 |
|----|---------|--------|------|
| **L0 核心 skill（常驻）** | 永远在场 | GM 立场 + 工具用法 + **三种机制的意义**（给选项=能动性 / 掷骰=随机趣味 / 纯叙事=推进）+ **"这一刻该用哪个流程"的决策框架（dispatcher）** | 框架 |
| **L1 流程 skill 库** | 相关时加载 | 预设的可复用业务流程，每个把"原语怎么组合 + 何时调 + 结果怎么叙述"都想好、调教到对 AI 友好。分两档（见下） | 框架 |
| **L2 团特定补充** | 按需 | 挂在流程 skill 之下的团特定数据与设定 | 规则书 |
| **运行时注入** | 掷骰当下 | 工具响应夹带的情境提醒（"别软化失败"） | 框架（MCP 响应） |

**规则书作者通常只做两件事：选 skill + 填数据。** 一个流程（如抽卡）怎么对 AI 友好、怎么防作弊、怎么叙述结果——框架在 skill 里一次性想好，所有用到它的团本共享，作者不必重写。**但这不是定死的**：规则书作者**可以**自写流程 skill（框架提供统一模板），只是绝大多数情况下框架预设的够用、没必要这么做。框架预设是默认与捷径，不是限制。

**流程 skill 库分两档：**

- **核心流程 skill（默认）**：通用且高频，几乎每个团都要——对抗判定 / 安价选择 / 探索给选项 / 抽卡。
- **可选玩法 skill（团本选用，定向优化）**：特定玩法才需要的专门流程，团本在 manifest 里显式选上才加载。例如 **多选稀释流程**（玩家"全选"时如何适当降低各项效果）、**购点验证流程**（点数在选项间分配、校验不超额）。不选就不加载，核心层保持精简。

也就是说，"选项的资源经济与平衡"**不是非目标，而是可选玩法 skill 的职责**——框架以可选 skill 的形式预设这些玩法，团本按需选用，而不是把它塞进人人都背的核心层、也不是甩给外部。

**这些流程 skill 是要被调用千百次的真 · skill**，每一个都应当用 skill-creator 的方法论编写、测试、迭代（写测试 prompt、跑 with/without baseline、看哪种措辞最让 AI 听话）。**项目接下来的主要工作量，就是用 skill-creator 造这个 GM 流程 skill 库。**

## 9. 团本 / 规则书：薄到几乎纯声明的 manifest

一个团本通常由两部分组成：

1. **manifest**：声明"我用框架的哪些流程 skill" + 机制参数（骰子类型、暴击线等结构化数据）
2. **数据**：
   - 世界观、基调、NPC 等叙事内容 → 入知识库（Markdown，FTS5 可搜索）
   - 随机池（卡池/掉落表）→ 入知识库（结构化条目，带属性，供 `knowledge_sample` 命中）

（可选第三部分：自定义流程 skill——需要框架没预设的独特玩法时才写，见 §8。）

"随机表"作为一种特殊静态数据类型**被删掉了**——它就是带结构化属性的知识库内容。规则书作者喂知识库，框架提供随机命中。

## 10. 单人 / 多人模式

模式是 session 级设置，本质是**"决定权归谁"以及阻不阻塞**：

| resolver | 谁决定 | 单人对话 | 多人论坛 |
|----------|--------|---------|---------|
| 骰子（安科） | 随机 | AI 声明选项→掷→当场继续叙述，不阻塞 | PO 公开掷骰 |
| 玩家（安价） | 人 | AI 给选项→等这一个玩家回一句→继续（自然轮次） | 给选项→开投票窗口→聚合票数→继续（真阻塞） |

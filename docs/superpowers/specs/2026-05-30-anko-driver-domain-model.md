## 0. 为什么需要这份文档

之前的混乱不是想法太碎，是**三个层被同时设计**。任何一层在另一层还没定形时去对接，都会乱。
解药：**承认有三层，一次只设计一层，跨层的焦虑就消失。**

```
③ 塑形层   skill / guideline / 反讨好（F1/F2/F3）      —— 见 2026-05-29 spec
② 行动层   裁决工具：三种骰 + 给选项                    —— 本文档 §3
① 数据层   三种存储：状态 / 事件 / 剧本                 —— 本文档 §2
```

本文档定死 ① 和 ②，并明确 ③ 如何骑在它们之上。

## 1. 一条贯穿全项目的原则：agent 即引擎

anko_driver **嫁接在 AI agent 上**。所以：

- **agent 本身就是阅读器和召回器。** 不需要自建 embedding / 向量库 / ML 流水线。
- 数据层只需 **SQLite（KV + FTS5 全文搜索）** 就够 v1。
- 这是 [2026-05-29 spec §4](2026-05-29-anko-driver-design.md)"靠模型已有能力、别过度造约束"在数据层的延伸。

**任何"要不要上 MongoDB / 向量检索 / RAG / 机器学习"的冲动，默认答案是"v1 不上"。** 真有必要的，单开 spike（见 §4），不进主线。

## 2. 数据层：三种存储，按"访问模式"区分

> **关键认知纠偏**：区分这三张表的**不是"结构化 vs KV"**，而是 **AI 怎么访问它们**。
> 纠结存储形态会原地打转；盯住访问模式，边界立刻清晰。

| 存储 | 本质（存什么） | 访问模式（怎么用） | 形态 | 现状 |
|------|----------------|--------------------|------|------|
| **状态表** | 此刻为真的事实：人物卡、货币、计时器、商店库存 | **精确 get / set**，可被骰子读写 | KV / 文档（值为 JSON） | [`state` 表](../../../src/anko_db/store.py) 已是 KV，基本够用 |
| **事件表** | 发生过 / 将发生的记忆：剧情、伏笔、走向 | **追加 + 召回**（按相关性取回） | 带标签的叙事条目 | [`history`](../../../src/anko_db/store.py) + [`notes`](../../../src/anko_db/store.py) 有雏形 |
| **剧本表** | 作者写死的世界：设定、门派、NPC、卡池 | **只读，搜索 + 随机抽样** | FTS5 文本 + 带属性的池条目 | [`knowledge` FTS5](../../../src/anko_db/store.py) 有，卡池缺属性 |

三表的职责互斥、可独立理解、可独立测试：

- **状态表**回答"现在是什么样"——HP 多少、有几枚金币、计时器还剩几回合。
- **事件表**回答"发生过什么 / 埋了什么线"——是叙事记忆，不是当前事实。
- **剧本表**回答"这个世界本来是什么样"——作者写死、整局不变、只读。

## 3. 行动层：裁决工具是"数据层的读写器"

行动层产出**结果的归属**（结果不能由叙述者自由编造，见 2026-05-29 spec L1）。
它有两个家族：**给选项**（结果归玩家）和**掷骰**（结果归随机）。

### 3.1 掷骰的三种形态——都由"操作数从哪来"区分

> 安科传统里 d10 的 "0" 常指 10。**本项目是 AI 安科，不必遵循传统**——
> 只要保证"AI 声明的判定范围"和"骰子产出的结果"能对上即可。骰面语义交给规则书。

| 骰种 | 操作数来源 | 产出 | 是否碰状态表 |
|------|------------|------|--------------|
| **选项骰** | GM 预设的选项表 | 命中某个选项 → **叙事档位** | 否 |
| **对抗骰** | 状态表的两个属性 | 胜 / 负 | **读**状态表 |
| **状态骰** | 骰子 ± 状态表某数值（如 `金币 + 5d3000`） | 一个**数值** | **读+写**状态表 |

聚类：

- **裁决骰** = 选项骰：出**叙事后果**（当前 `roll_outcome` 的形态）。
- **数值骰** = 对抗骰 + 状态骰：出**数值**，与状态表耦合。

**这条是消解"骰子设计很乱"的关键洞察：掷骰不是独立子系统，它是数据层（状态表）的读写器。**
对抗骰、状态骰之所以存在，正是因为有些团（战斗团）要的不是叙事档位，而是血量 / 伤害这种**数值**，必须接状态表。

### 3.2 给选项

`offer_choice`：结果归玩家。后果在玩家选之前锁定（"声明后果在先"，见 2026-05-29 spec §6.1）。
单人转轮 / 多人投票的差异属塑形/模式层，本文档不展开。

## 4. 两个研究级难题：明确隔离，v1 用朴素版

这两个是真难，**绝不能在主线里顺手解决**，否则项目永远卡在它们上。
各自单开 spike，主线只用朴素版占位：

| 难题 | 为什么难 | **v1 朴素版（主线用这个）** | 真·解法（单开 spike，不进主线） |
|------|----------|------------------------------|--------------------------------|
| **事件表的语义召回** | "按相关性取回剧情/伏笔"本质是 RAG | 事件写成 markdown，**FTS5 搜索 + agent 自己读** | 向量 embedding / 语义检索 |
| **让 AI"读懂"世界设定**（而非只存着） | 设定不能只躺在库里，要进 AI 的判断 | 三条朴素路径：(a) 核心调性在 skill 安装时**焊进上下文**；(b) 相关时 `knowledge_search` 取；(c) agent 直接读 markdown | 设定的结构化推理 / 自动一致性检查 |

## 5. 三层如何咬合（数据流一瞥）

```
玩家行动
  → 塑形层（skill）判断：该给选项？该掷骰？哪种骰？
      ├ 给选项  → offer_choice            （结果归玩家）
      └ 掷骰
          ├ 选项骰 → roll_outcome          （出叙事档位）
          ├ 对抗骰 → 读状态表两属性 → 裁决  （出胜负）
          └ 状态骰 → 读状态表数值 ± 骰 → 写回状态表（出数值）
  → 叙述（受塑形层"别软着陆"约束 + 工具响应补刀）
  → 需要世界/卡池内容时 → 剧本表（search / sample）
  → 推进后把剧情/伏笔 → 事件表（写）；回顾时 → 事件表（召回）
```

## 6. 本文档**不**负责定的（留给子 spec）

为防止它又膨胀成散文，以下明确不在此定，由后续各自的子 spec 解决：

- 各工具的具体入参 / 出参 schema
- 状态表 JSON 文档的字段约定、嵌套键规则
- 卡池属性的存储形态（边表 / JSON 列）与 `knowledge_sample` 实现
- 对抗骰 / 状态骰怎么引用状态表里的键（取数语法）
- skill 与 guideline 的双层注入机制（见 2026-05-29 spec §8）
- 单人 / 多人模式、裁判层、L3 Hook

**子 spec 的写法**：先引用本文档锁定的概念，再只展开自己那一片。
一次切一片去实现，其余封箱。

> **上游依赖**：[01-业务分析](../01-业务分析/)（本页是对那些问题的概念回应）。
> **状态**：🟢 已成型（迁自北极星文档）。

> 词条级的精确定义见 [术语表](术语表.md)。行为塑形的立场细节见 [99-历史输入](../99-历史输入/) 链接的 2026-05-29 spec。

## 0. 为什么需要这层概念

设计之所以会"零碎、混乱"，根因是**三个层被同时设计**。任何一层在另一层还没定形时去对接，都会乱。
解药：**承认有三层，一次只设计一层。**

```
③ 塑形层   skill / guideline / 反讨好（F1/F2/F3）
② 行动层   裁决工具：三种骰 + 给选项
① 数据层   三种存储：状态 / 事件 / 剧本
```

## 1. 贯穿全项目的原则：agent 即引擎

anko_driver **嫁接在 AI agent 上**。所以：

- **agent 本身就是阅读器和召回器。** 不自建 embedding / 向量库 / ML 流水线。
- 数据层只需 **SQLite（KV + FTS5 全文搜索）** 就够 v1。
- **任何"要不要上 MongoDB / 向量检索 / RAG / ML"的冲动，默认答案是"v1 不上"。** 真有必要的单开 spike，不进主线。

## 2. 数据层：三种存储，按"访问模式"区分

> **关键认知纠偏**：区分这三张表的**不是"结构化 vs KV"**，而是 **AI 怎么访问它们**。

| 存储 | 本质（存什么） | 访问模式（怎么用） | 形态 |
|------|----------------|--------------------|------|
| **状态表** | 此刻为真的事实：人物卡、货币、计时器、商店库存 | **精确 get / set**，可被骰子读写 | KV / 文档（值为 JSON） |
| **事件表** | 发生过 / 将发生的记忆：剧情、伏笔、走向 | **追加 + 召回**（按相关性取回） | 带标签的叙事条目 |
| **剧本表** | 作者写死的世界：设定、门派、NPC、卡池 | **只读，搜索 + 随机抽样** | FTS5 文本 + 带属性的池条目 |

- **状态表**回答"现在是什么样"；**事件表**回答"发生过什么 / 埋了什么线"；**剧本表**回答"这个世界本来是什么样"。

## 3. 行动层：裁决工具是"数据层的读写器"

行动层产出**结果的归属**（结果不能由叙述者自由编造）。两个家族：**给选项**（结果归玩家）和**掷骰**（结果归随机）。

### 3.1 掷骰的三种形态——由"操作数从哪来"区分

> 安科传统里 d10 的 "0" 常指 10。**本项目是 AI 安科，不必遵循传统**——只要"AI 声明的判定范围"和"骰子产出的结果"能对上即可。骰面语义交给规则书。

| 骰种 | 操作数来源 | 产出 | 是否碰状态表 |
|------|------------|------|--------------|
| **选项骰** | GM 预设的选项表 | 命中某选项 → **叙事档位** | 否 |
| **对抗骰** | 状态表的两个属性 | 胜 / 负 | **读**状态表 |
| **状态骰** | 骰子 ± 状态表某数值（如 `金币 + 5d3000`） | 一个**数值** | **读+写**状态表 |

聚类：**裁决骰**（选项骰，出叙事后果）+ **数值骰**（对抗/状态骰，与状态表耦合）。

**消解"骰子设计很乱"的关键洞察：掷骰不是独立子系统，它是状态表的读写器。** 对抗骰/状态骰存在的理由，正是战斗团要的不是叙事档位而是血量/伤害这种**数值**。

### 3.2 给选项

结果归玩家。后果在玩家选之前锁定（"声明后果在先"）。单人转轮 / 多人投票的差异属塑形/模式层。

## 4. 两个研究级难题：明确隔离，v1 用朴素版

绝不能在主线里顺手解决，否则项目永远卡在它们上。各自单开 spike：

| 难题 | 为什么难 | **v1 朴素版（主线）** | 真·解法（spike） |
|------|----------|------------------------|------------------|
| **事件表的语义召回** | 本质是 RAG | 事件写 markdown，**FTS5 搜索 + agent 自己读** | 向量 embedding / 语义检索 |
| **让 AI"读懂"世界设定** | 设定要进 AI 判断，不能只躺库里 | (a) 核心调性安装时**焊进上下文**；(b) 相关时 `knowledge_search`；(c) agent 直接读 markdown | 设定结构化推理 / 自动一致性检查 |

## 5. 三层如何咬合（数据流一瞥）

```
玩家行动
  → 塑形层判断：该给选项？该掷骰？哪种骰？
      ├ 给选项  → offer_choice            （结果归玩家）
      └ 掷骰
          ├ 选项骰 → roll_outcome          （出叙事档位）
          ├ 对抗骰 → 读状态表两属性 → 裁决  （出胜负）
          └ 状态骰 → 读状态表数值 ± 骰 → 写回状态表（出数值）
  → 叙述（受塑形层"别软着陆"约束 + 工具响应补刀）
  → 需要世界/卡池内容时 → 剧本表（search / sample）
  → 推进后剧情/伏笔 → 事件表（写）；回顾时 → 事件表（召回）
```

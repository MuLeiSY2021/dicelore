## 0. 看架构的两个视角（本页怎么读）

02 给的是**逻辑三层**（数据 / 行动 / 塑形）——系统**概念上**由什么构成。本页要落到**物理组件**——**实际上**打包成什么、装到哪。

**这两层不是 1:1。** 本页的主结构就是这张桥：先承接逻辑层，再落到物理组件，中间用 §1 的映射表咬合。最关键的一条非对应——**塑形层不是一个组件**——是本页要讲清的核心。

## 1. 逻辑三层 ⟂ 物理组件（非 1:1 映射表）

| 逻辑层（02） | 落到哪些物理组件 | 是否 1:1 |
|---|---|---|
| **① 数据层** | 内层能力库的**四业务域 store** + 外层 MCP 的**数据工具**（`sheet_*` / `event_*` / `world_*` / `rule_search`） | 基本 1:1 |
| **② 行动层** | 外层 MCP 的**裁决工具**（给选项 + 选项骰 + 对抗骰；**状态骰下沉数据层 sheet 写**，见 §4.2）+ **`narrate` 输出通道**，内部调用内层**骰子引擎** | 1 组件（MCP server 的一部分） |
| **③ 塑形层** | **分布在三处**：L1＝MCP 工具设计本身 ／ L2＝Skills 包 ／ L3＝adapter 装的 Hook + 裁判 subagent | **否——最关键的非对应** |

> **为什么塑形层散开**：02 §4 已定 L1/L2/L3 是与三层正交的**强度轴**，不是第四层。落到组件，它就必然散成三处——把正确行为焊进工具结构（L1）、用 skill 教判断（L2）、事后审计兜底（L3）。**谁要是把"塑形层"做成一个模块，就是把正交轴误当成了一层。** 本项目区别于"游戏引擎"的命门正在此。

## 2. 两层工具架构：内层能力库 ／ 外层 MCP 塑形面

整个框架的代码主结构是**上下两层工具**，别压成一层：

```
        ┌──────────────────────────────────────────────┐
   AI ──│  外层：MCP 工具面（塑形过的少量工具）           │ ← L1 焊接点：
        │  ② resolve_*（choice / outcome / contest）     │   · "声明后果在先"做成必填参数
        │  ① sheet_get  sheet_update（状态骰＝带骰写回）   │   · 给选项 / 掷骰用不同工具名分流
        │     event_*  timer_set                         │   · AI 只给引用，不给真实数值
        │     world_search/sample/register  rule_search  │   · 工具响应"补刀"（带反讨好提醒）
        │  ▶ narrate（散文流：非终结步骤）                │   · 散文走工具→可记录、可审计、抗软着陆
        └───────────────────────┬──────────────────────┘
                                │ 组合调用（不是 1:1 暴露）
        ┌───────────────────────┴──────────────────────┐
        │  内层：能力库（原子函数，可单测，AI 不可见）     │
        │  骰子引擎 ｜ 四业务域 store 的 CRUD ｜ …         │
        └───────────────────────┬──────────────────────┘
                                │
                  SQLite（每局一文件：四域表 + FTS5）
```

- **内层能力库**＝纯逻辑原子（骰子计算、四域读写），不依赖 MCP、可脱离 agent 单测。裸骰子（`dice_roll` 等）**永居内层**，AI 调不到——限制工具面本身就是 L1 强制。
- **外层 MCP 工具面**＝内层原子的**有意识组合 + L1 塑形包装**，**不是原子的 1:1 暴露**。数量受控（避免撑爆 agent 上下文 / 选错工具），名字按业务域 / 裁决类型分流（便于 AI 选对、便于审计）。
- 这一刀解决三件事：① 原子可脱 MCP 单测；② 暴露面小而可控；③ 塑形（L1）有了明确的焊接位置——就在外层包装里。

## 3. 数据层①：四业务域 store

承接 [02 §2](../02-领域模型/核心概念.md)。物理上是**每局一个 SQLite 文件**，内含四业务域的表 + FTS5 全文索引。MCP server 通过 env / 配置解析"当前是哪一局"（沿用现有 session 解析思路）。

| 域 | 外层工具 | 访问画像 | 写源 |
|---|---|---|---|
| **sheet 实体卡** | `sheet_get` / `sheet_update`（批量写，**状态骰下沉于此**，见 §4.2）/ `sheet_show`（可见性，见 §3.1） | 精确读写，被对抗骰读、状态骰读+写 | AI 运行期（含骰子写回） |
| **event** | `event_append` / `event_recall` / `timer_set`（`shot` 的快照亦落于此，见 §3.1） | 追加 + 召回 | AI 运行期 |
| **world** | `world_search` / `world_sample` / `world_register` / `world_show`（可见性，见 §3.1） | 搜索 + 随机抽样 + 注册 | **两源**：作者灌注 + AI 现编 |
| **rule** | `rule_search`（**只读**） | 搜索 | **人类侧写**（作者版本化 + 团主/管理员实时），AI 不可写 |

**两个写源（数据层的一条架构事实，本页只点明）**：
- **AI 运行期增量**：跑团时 `sheet_update` / `event_append` / `world_register` 等即时写。
- **人类侧 rule 写**：rule 由人类改——作者按**版本号热更新**团本，或团主 / 管理员**实时**调整；AI 对 rule 始终只读（守反讨好红线）。
- **作者版本化分发**：团本（world / rule 底料）作为内容包按版本号分发。"v2 团本 vs 进行中存档（带 v1 上 AI 长出的内容）"的迁移语义 → 留 [04](../04-子系统设计/) / 未来，本页不展开。

> **游戏时间不是独立通道**：游戏内时间 / 回合就是 sheet 的某 attr（如 `世界.时间`），用 `sheet_update` 写（可带骰）；**哪个 attr 是"钟"由团本 rule / skill 声明**，框架不写死、不内置回合概念，自身只保留单调事件序号 `seq`。`timer_set` 登记到期、hook 每轮读该钟 attr 比对触发。详见 [术语表「sheet 钟」](../02-领域模型/术语表.md)。

### 3.1 可见性：`visible` 列 + show（持久揭示）/ shot（快照披露）

承接 [02 §2 要点 5](../02-领域模型/核心概念.md)：玩家能看到什么**不由 AI 的散文随口定**，而是数据层的一个**结构事实**——输出层（§6 流②）正是按它过滤后才渲染给玩家。

- **存储 = `visible` 列**：sheet 表加第 4 列 `(entity, attr, value, visible)`；`world_doc` / `world_pool` / `event` 各加 `visible` 列。（用列、不用键前缀，避免和 `前缀:键` 约定打架。）
- **默认拒绝**：world / sheet **全部默认隐藏**；event 按 `kind` 写入时定默认——`narrate` / `verdict` / `mutation` / `timer_fired` 默认可见，`note`（GM 注记 / 伏笔）默认隐藏，`event_append` 可显式覆盖。
- **`sheet_show` / `world_show`（持久揭示）**：翻 `visible=true`，输出层每轮渲染其**实时值**（跟着变）。粒度可 **attr 级**（点名授权几个 cell）或 **entity 级递归**（整 entity 可见，长效、覆盖该 entity 未来新增 attr）；但带**强制隐藏标记**的 cell（暗值）即使 entity-show 也不露——防泄漏。玩家自己人物卡默认也隐，AI **开局 `sheet_show` 一次**。
- **`shot`（快照披露，中优先级）**：把某隐藏**目标**（sheet cell **或 world 条目**）**此刻内容以冻结副本披露一次**，不翻持久可见位——**落点是写一条「可见 event」**（`kind=reveal`）。对 sheet（值暗变）→ 玩家见旧值、下次 shot 才刷新；对 world（基本静态）→ 一次性披露、不入持久可见集（给"瞥一眼 / 侦查 / 占卜 / 鉴定"）。用 event 实现"副本"，目标自身 `visible` 不动。
- **show 审计**：`sheet_show` / `world_show` 各写一条 `kind=note`、`visible=0`（对玩家隐）的审计 event，供 L3 / 回看。
- **un-show 不做一等公民**：`shot` 顶掉了"情报重新成谜"主需求；罕见的"曾持久 show 要收回"由 `sheet_show` 传 `visible=false` 兜底。

> 强制隐藏标记怎么存、`shot`=event append 的求值/写入语义 → [04 内层能力库](../04-子系统设计/内层能力库.md)；工具 schema → [04 MCP工具面](../04-子系统设计/MCP工具面.md)。

## 4. 行动层②：裁决工具面

承接 [02 §3](../02-领域模型/核心概念.md)：行动层 = **resolver**（选择者 玩家/随机 × 结果形状 label/verdict/number/content）。落到工具——`resolve_choice`=玩家选 label；`resolve_outcome`=随机 label；`resolve_contest`=随机 verdict（DC=一边常数）；**number（状态骰）下沉为 `sheet_update` 带骰**（§4.2）；**content（卡池）= `world_sample`**（数据层 world 域）。故留在行动层、作**独立裁决工具**的是下表三个——共性是**产出"结果归属"而不直接改写 sheet 数值**。**铁律：随机与取数全在 MCP 内执行，AI 只给引用 / 过滤条件，拿不到也改不了真实数值**——这是防 AI 编造结果（F1/F2）的结构地基。

| 工具 | 家族 / 骰种 | AI 给什么 | MCP 做什么 | 碰哪个域 | I/O 契约（形状，schema 见 04） |
|---|---|---|---|---|---|
| `resolve_choice` | 给选项（结果归玩家） | 各选项 + **该选项后果（必填）** | 锁定后果、落 event、把选择权交玩家 | event（记录） | in: `options=[{label, consequence}]` → out: 玩家所选 + 已锁后果 |
| `resolve_outcome` | 选项骰 | 各档位区间 + **预声明后果** | 掷骰命中一档 | 不碰 | in: 后果档位表 → out: 命中档 + 该档后果 |
| `resolve_contest` | 对抗骰 | 两个 sheet 属性的**引用** | 取真值 + 掷骰 + 比大小 | sheet（读） | in: `("A.力量","B.AC")` → out: 胜 / 负 |

> **工具名分流＝抗 F3 的 L1**：`resolve_choice`（玩家做主）和 `resolve_outcome`（交给骰子）刻意分成两个工具，逼 AI 显式回答"这事该选还是该骰"（02 §4）。

### 4.1 输出通道：`narrate`（散文 stream）

裁决产出"结果归属"，但玩家最终看到的是**散文**。本项目把散文也封装成工具 `narrate`（而非纯 talk）——AI 把组织好的剧情文本交给它，MCP 记录 + 落 event。**`narrate` 是轮内可多次穿插的 stream 输出，不是回合的终结步骤**（一轮的边界 = agent 自然回合，见 §6）。封装成工具买到三个塑形/记账收益：

1. **落 event 表**：散文进事件表（`kind=narrate`、默认可见）→ 支撑长程记忆 / 回顾（`event_recall`），并供**输出层**渲染给玩家（§6 流①）。
2. **L3 可审计**：**Stop hook**（回合末）能比对"本轮散文 vs 本轮裁决结果 / 已声明后果"，抓软着陆（F2）。
3. **强时序**：每条 narrate 落 event 带单调 `seq`，可结构化检查"叙述某结果前是否真的裁决过"（同轮内对应 resolver event 的 `seq` 必须在前）——抗 F1。

> `narrate` 不**生成**叙事（那是 AI 的创作），只**承接 + 记录**，并可在响应里"补刀"一致性提醒（L1/L2 混合）。**部署差异**：chat 原生 agent 的 adapter 可把它降级为"直接 talk + 自动捕获写 event"，详见 [跨agent与适配层](跨agent与适配层.md)。它**属行动层、是输出通道**，但不属裁决家族（掷骰 / 给选项）。

### 4.2 状态骰下沉为 sheet 写（2026-06-02 修订）

02 §3.1 早有洞察——"**状态骰不是独立子系统，是 sheet 域的读写器**"。这里把它兑现：**"骰子 ± sheet 数值 → 写回"不再做成独立的 `roll_value` 工具，而是数据层 sheet 写工具（`sheet_update`）的「带骰形态」**。一次 entity 作用域的批量写，其中某些项带掷骰（`HP-2d6`）、某些项是绝对赋值（`经验=0`）或集合增减（`物品+小刀`），**整批原子**。

- **anti-F1 不破**：带骰项仍在引擎内掷、AI 给不出真值；引擎对每项标 `rolled | set` 记进账本（供 L3 比对）。
- **代价（睁眼接受）**：原本"`roll_value` 这个工具名逼 AI 分辨'该掷还是该设'"的那点 **L1 结构摩擦消失**，降级为 **L2 教 +（靠账本的）L3 抓"该掷处却用了 `=`/裸 set"**。方向与 [02 §4](../02-领域模型/核心概念.md) 矩阵一致——状态变更的"选对方式"本就偏 F3、偏靠教。
- **mcp 准则上**：合并成一个面向"一拍结算"的批量写 = **workflow tool**；这是对 mcp-builder"默认铺满细粒度 API"的**有意背离**，理由是 §2 的塑形诉求（少而塑形过）。
- **边界**：合成一个批量写工具、还是为保 L1 再留个掷骰名 → 纯塑形取舍，归 [04 MCP 工具面](../04-子系统设计/MCP工具面.md)；批量写的**求值/写入语义**（文法、`算符 × 操作数`派发、原子、账本）归 [04 内层能力库](../04-子系统设计/内层能力库.md)。详细权衡与被否（"维持三骰独立工具"）→ [05 ADR](../05-决策记录-ADR/)。

## 5. 塑形层③：不是组件，是焊进三处的纪律

塑形层是项目的差异化（[01 问题域](../01-业务分析/问题域.md)），但它**没有自己的组件**——它是施加在另两层 + 宿主 agent 上的设计纪律，沿 L1/L2/L3 分布：

| 杠杆 | 物理落点 | 形态 | 可靠性 |
|---|---|---|---|
| **L1 工具强制** | 在 §2 外层工具面的**设计里** | 必掷骰、后果在先必填、工具名分流、AI 不给值、工具响应补刀 | 最硬，但只管能被结构卡住的 |
| **L2 skill 教** | **Skills 包**（可移植核心，塑形层本体） | dispatcher skill + 流程 skill 库 + guideline（"别软着陆""何时选何时骰"） | 可被忽视，但判断类的唯一载体 |
| **L3 / hook 承重** | **Claude Code hook / subagent**（v1 承重，见定位重述） | **Stop hook（回合末）**：物化暂存 choice · 掷骰绕过率 · 散文-后果比对 · 裁判 subagent；**回合开始 hook**：被动 rule 召回 · timer 到期注入 | 审计事后兜底；召回 / timer / choice 物化则承重 |

- **L2 是核心**：GM 教条是 core 标准件（markdown，内容可搬），装载走 Claude Code 的 skill 机制。
- **hook 承重**（2026-06-02 定位重述）：v1 骑 Claude Code，**被动 rule 召回 / timer 到期 / L3 审计都靠 hook**，不再是"可选优雅降级"；core（MCP/Skill/SQLite）仍标准不锁死，但交付绑 Claude Code。详见 [跨agent与适配层](跨agent与适配层.md)。
- **L3 窗口 = 一个 agent 回合**（R1）：narrate 已是 stream，旧"两个 narrate 标记之间"的窗口定义塌掉；改由 **Stop hook 在回合末**扫本轮 event 审计。**新硬规矩**：非终局轮回合结束时必须留有暂存的 `resolve_choice`，否则违规（把玩家晾着）——Stop hook 当场可查。

## 6. 端到端数据流（一局怎么跑）

```
一局 = 若干「轮」循环；一轮 = agent 一个自然回合（玩家输入 → 回合自然结束）
═══════════════════════════════════════════════════════════════════
玩家输入
  │
  ▼  【轮内 = AI 像作者创作：下列任何调用都可重复任意次、任意顺序穿插】   ← L2 Skill 判断
  ├─ 给选项   → resolve_choice(options=[{选项,后果}])   后果必填（L1）；★暂存，不立即给玩家
  ├─ 选项骰   → resolve_outcome(后果档位表)             引擎掷骰命中档 → 回档位+预声明后果
  ├─ 对抗骰   → resolve_contest("A.属性","B.属性")       引擎读 sheet 取真值 → 掷 → 回胜负
  ├─ 状态变更 → sheet_update(张三, HP-2d6, 金币+5d3000)  带骰引擎内掷+读旧值+原子写（§4.2）
  ├─ 世界/卡池 → world_search / world_sample；现编 → world_register
  ├─ 剧情/伏笔 → event_append；回顾 → event_recall；定时埋点 → timer_set
  ├─ 揭示/披露 → sheet_show / world_show（持久）｜ shot（一次性快照，§3.1）
  └─ ▶ narrate(散文)  ← stream：轮内可多次穿插，非终结步骤
  │
  ▼  agent 回合自然结束 → Stop hook：
       ① 物化暂存的 resolve_choice → 选项+已锁后果呈现给玩家（anti-F2：后果先于玩家可见即锁）
       ② L3 审计本轮：掷骰绕过率、散文 vs verdict/mutation 是否一致（抗 F1/F2）
       ③ 被动 rule 召回 + timer 到期（读钟 attr 比对）→ 注入下一轮提示词
       ※ 非终局轮必须留有暂存 choice，否则违规（把玩家晾着、没给能动性）
  │
  ▼  唯二终局出口：game_end / you_death（=game_end 特例）→ AI 复盘，可不留 choice
───────────────────────────────────────────────────────────────────
三流分工（同一轮的产物各走各的管道）：
  ① narrate 散文              → 进对话 + AI 上下文（AI 本就产出的创作）
  ② 输出层（hook/CLI 渲染器） → 读 store/event、按 visible 过滤，渲染机械回显
                               （"金钱 +3d100=74 → 77"）+ 状态菜单 + 待选项 给玩家
                               不进 AI 上下文 ＝ 零额外 token；AI 只 narrate 色彩，不吐数值菜单
  ③ resolver/sheet_update 结构化结果 → 只回 AI（最小 token，供其继续推理）
```

裁决/掷骰分支与 [02 §6](../02-领域模型/核心概念.md) 一致（02 的状态骰在此**下沉为 sheet 写**，§4.2），这里把它接到了具体组件、工具与**回合时序**上。**关键转变**（R1）：`narrate` 不再是"回合终结步骤"而是轮内 stream；玩家所见由**输出层**按 `visible` 渲染（流②），AI 只负责散文色彩、零额外 token 吐数值菜单；`resolve_choice` 暂存到回合末经 **Stop hook** 物化呈现。

> **一轮 = AI 像作者写一段**：上图列的**任何**调用都可在一轮内**重复任意次、任意顺序**穿插（掷两次骰、改三版 sheet、穿插数段散文、边写边查 world）。多次可调是常态、不是 narrate 的特权；**只有两个调用带特殊时序语义**——`resolve_choice`（暂存、末次为准、回合末物化）与 `narrate`（流向玩家的散文 stream）；其余调用**即时返回、即时生效**。L3 审计的单位就是这"一段创作"（= 一个 agent 回合）。

## 7. 组件清单 + 现有代码映射

**框架组件 + 内容输入**：

1. **内层能力库**（骰子引擎 + 四域 store）— 纯逻辑、可单测、不见 AI
2. **MCP server**（外层工具面：裁决 + 数据工具 + `narrate` 输出通道）— L1 焊接位

4. **adapter + 输出层渲染器**（接线 + 可选 L3 + 玩家呈现）— 把 2/3 装进目标 agent；**输出层**读 SQLite store / event、**按 `visible` 过滤**，渲染机械回显（"金钱 +3d100=74 → 77"）+ 状态菜单 + 待选项给玩家（**v1 一等概念**，R1；不进 AI 上下文 ＝ 零额外 token）

6. **团本 / manifest**（内容，非框架代码）— 作者填 / 生成的 world / rule / 卡池 + skill 选择，版本化分发，灌注进数据层

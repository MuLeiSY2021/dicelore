## 0. 看架构的两个视角（本页怎么读）

02 给的是**逻辑三层**（数据 / 行动 / 塑形）——系统**概念上**由什么构成。本页要落到**物理组件**——**实际上**打包成什么、装到哪。

**这两层不是 1:1。** 本页的主结构就是这张桥：先承接逻辑层，再落到物理组件，中间用 §1 的映射表咬合。最关键的一条非对应——**塑形层不是一个组件**——是本页要讲清的核心。

## 1. 逻辑三层 ⟂ 物理组件（非 1:1 映射表）

| 逻辑层（02） | 落到哪些物理组件 | 是否 1:1 |
|---|---|---|
| **① 数据层** | 内层能力库的**四业务域 store** + 外层 MCP 的**数据工具**（`sheet_*` / `event_*` / `world_*` / `rule_search`） | 基本 1:1 |
| **② 行动层** | 外层 MCP 的**裁决工具**（给选项 + 选项骰 + 对抗骰；**状态骰下沉数据层 sheet 写**，见 §4.2）+ **`narrate` 输出通道**，内部调用内层**骰子引擎** | 1 组件（MCP server 的一部分） |
| **③ 塑形层** | **分布在三处**：L1＝MCP 工具设计本身 ／ L2＝Skills 包 ／ L3＝adapter 装的 Hook + 裁判 subagent | **否——最关键的非对应** |

> **为什么塑形层散开**：02 §4 已定 L1/L2/L3 是与三层正交的**强度轴**，不是第四层。落到组件，它就必然散成三处——把正确行为焊进工具结构（L1）、用 skill 教判断（L2）、事后审计兜底（L3）。**谁要是把"塑形层"做成一个模块，就是把正交轴误当成了一层。** 本项目区别于"游戏引擎"的命门正在此。

## 2. 两层工具架构：内层能力库 ／ 外层 MCP 塑形面

整个框架的代码主结构是**上下两层工具**，别压成一层：

```
        ┌──────────────────────────────────────────────┐
   AI ──│  外层：MCP 工具面（塑形过的少量工具）           │ ← L1 焊接点：
        │  ② roll_outcome  roll_contest  offer_choice    │   · "声明后果在先"做成必填参数
        │  ① sheet_get  sheet_update（状态骰＝带骰写回）   │   · 给选项 / 掷骰用不同工具名分流
        │     event_*  timer_set                         │   · AI 只给引用，不给真实数值
        │     world_search/sample/register  rule_search  │   · 工具响应"补刀"（带反讨好提醒）
        │  ▶ narrate（回合叙述：输出通道）                │   · 叙述走工具→可审计、可记录、强时序
        └───────────────────────┬──────────────────────┘
                                │ 组合调用（不是 1:1 暴露）
        ┌───────────────────────┴──────────────────────┐
        │  内层：能力库（原子函数，可单测，AI 不可见）     │
        │  骰子引擎 ｜ 四业务域 store 的 CRUD ｜ …         │
        └───────────────────────┬──────────────────────┘
                                │
                  SQLite（每局一文件：四域表 + FTS5）
```

- **内层能力库**＝纯逻辑原子（骰子计算、四域读写），不依赖 MCP、可脱离 agent 单测。裸骰子（`dice_roll` 等）**永居内层**，AI 调不到——限制工具面本身就是 L1 强制。
- **外层 MCP 工具面**＝内层原子的**有意识组合 + L1 塑形包装**，**不是原子的 1:1 暴露**。数量受控（避免撑爆 agent 上下文 / 选错工具），名字按业务域 / 裁决类型分流（便于 AI 选对、便于审计）。
- 这一刀解决三件事：① 原子可脱 MCP 单测；② 暴露面小而可控；③ 塑形（L1）有了明确的焊接位置——就在外层包装里。

## 3. 数据层①：四业务域 store

承接 [02 §2](../02-领域模型/核心概念.md)。物理上是**每局一个 SQLite 文件**，内含四业务域的表 + FTS5 全文索引。MCP server 通过 env / 配置解析"当前是哪一局"（沿用现有 session 解析思路）。

| 域 | 外层工具 | 访问画像 | 写源 |
|---|---|---|---|
| **sheet 实体卡** | `sheet_get` / `sheet_update`（批量写，**状态骰下沉于此**，见 §4.2） | 精确读写，被对抗骰读、状态骰读+写 | AI 运行期（含骰子写回） |
| **event** | `event_append` / `event_recall` / `timer_set` | 追加 + 召回 | AI 运行期 |
| **world** | `world_search` / `world_sample` / `world_register` | 搜索 + 随机抽样 + 注册 | **两源**：作者灌注 + AI 现编 |
| **rule** | `rule_search`（**只读**） | 搜索 | **人类侧写**（作者版本化 + 团主/管理员实时），AI 不可写 |

**两个写源（数据层的一条架构事实，本页只点明）**：
- **AI 运行期增量**：跑团时 `sheet_update` / `event_append` / `world_register` 等即时写。
- **人类侧 rule 写**：rule 由人类改——作者按**版本号热更新**团本，或团主 / 管理员**实时**调整；AI 对 rule 始终只读（守反讨好红线）。
- **作者版本化分发**：团本（world / rule 底料）作为内容包按版本号分发。"v2 团本 vs 进行中存档（带 v1 上 AI 长出的内容）"的迁移语义 → 留 [04](../04-子系统设计/) / 未来，本页不展开。

## 4. 行动层②：裁决工具面

承接 [02 §3](../02-领域模型/核心概念.md)：行动层 = **resolver**（选择者 玩家/随机 × 结果形状 label/verdict/number/content）。落到工具——`offer_choice`=玩家选 label；`roll_outcome`=随机 label；`roll_contest`=随机 verdict（DC=一边常数）；**number（状态骰）下沉为 `sheet_update` 带骰**（§4.2）；**content（卡池）= `world_sample`**（数据层 world 域）。故留在行动层、作**独立裁决工具**的是下表三个——共性是**产出"结果归属"而不直接改写 sheet 数值**。**铁律：随机与取数全在 MCP 内执行，AI 只给引用 / 过滤条件，拿不到也改不了真实数值**——这是防 AI 编造结果（F1/F2）的结构地基。

| 工具 | 家族 / 骰种 | AI 给什么 | MCP 做什么 | 碰哪个域 | I/O 契约（形状，schema 见 04） |
|---|---|---|---|---|---|
| `offer_choice` | 给选项（结果归玩家） | 各选项 + **该选项后果（必填）** | 锁定后果、落 event、把选择权交玩家 | event（记录） | in: `options=[{label, consequence}]` → out: 玩家所选 + 已锁后果 |
| `roll_outcome` | 选项骰 | 各档位区间 + **预声明后果** | 掷骰命中一档 | 不碰 | in: 后果档位表 → out: 命中档 + 该档后果 |
| `roll_contest` | 对抗骰 | 两个 sheet 属性的**引用** | 取真值 + 掷骰 + 比大小 | sheet（读） | in: `("A.力量","B.AC")` → out: 胜 / 负 |

> **工具名分流＝抗 F3 的 L1**：`offer_choice`（玩家做主）和 `roll_outcome`（交给骰子）刻意分成两个工具，逼 AI 显式回答"这事该选还是该骰"（02 §4）。

### 4.1 输出通道：`narrate`（回合叙述）

裁决产出"结果归属"，但玩家最终看到的是**叙述**。本项目把叙述也封装成一个工具 `narrate`（而非纯 talk），作为**一回合的终结步骤**——AI 把组织好的剧情文本交给它，MCP 记录 + 收尾。封装成工具买到三个塑形/记账收益：

1. **落 event 表**：叙述进事件表 → 支撑长程记忆 / 回顾（`event_recall`）。
2. **L3 可审计**：Hook 能比对"叙述 vs 本回合裁决结果 / 已声明后果"，抓软着陆（F2）。
3. **强时序**：把 `narrate` 卡成终结步骤，可结构化检查"叙述某结果前是否真的裁决过"（抗 F1）。

> `narrate` 不**生成**叙事（那是 AI 的创作），只**承接 + 记录 + 收尾**，并可在响应里"补刀"一致性提醒（L1/L2 混合）。**部署差异**：chat 原生 agent 的 adapter 可把它降级为"直接 talk + 自动捕获写 event"，详见 [跨agent与适配层](跨agent与适配层.md)。它**属行动层、是输出通道**，但不属裁决家族（掷骰 / 给选项）。

### 4.2 状态骰下沉为 sheet 写（2026-06-02 修订）

02 §3.1 早有洞察——"**状态骰不是独立子系统，是 sheet 域的读写器**"。这里把它兑现：**"骰子 ± sheet 数值 → 写回"不再做成独立的 `roll_value` 工具，而是数据层 sheet 写工具（`sheet_update`）的「带骰形态」**。一次 entity 作用域的批量写，其中某些项带掷骰（`HP-2d6`）、某些项是绝对赋值（`经验=0`）或集合增减（`物品+小刀`），**整批原子**。

- **anti-F1 不破**：带骰项仍在引擎内掷、AI 给不出真值；引擎对每项标 `rolled | set` 记进账本（供 L3 比对）。
- **代价（睁眼接受）**：原本"`roll_value` 这个工具名逼 AI 分辨'该掷还是该设'"的那点 **L1 结构摩擦消失**，降级为 **L2 教 +（靠账本的）L3 抓"该掷处却用了 `=`/裸 set"**。方向与 [02 §4](../02-领域模型/核心概念.md) 矩阵一致——状态变更的"选对方式"本就偏 F3、偏靠教。
- **mcp 准则上**：合并成一个面向"一拍结算"的批量写 = **workflow tool**；这是对 mcp-builder"默认铺满细粒度 API"的**有意背离**，理由是 §2 的塑形诉求（少而塑形过）。
- **边界**：合成一个批量写工具、还是为保 L1 再留个掷骰名 → 纯塑形取舍，归 [04 MCP 工具面](../04-子系统设计/MCP工具面.md)；批量写的**求值/写入语义**（文法、`算符 × 操作数`派发、原子、账本）归 [04 内层能力库](../04-子系统设计/内层能力库.md)。详细权衡与被否（"维持三骰独立工具"）→ [05 ADR](../05-决策记录-ADR/)。

## 5. 塑形层③：不是组件，是焊进三处的纪律

塑形层是项目的差异化（[01 问题域](../01-业务分析/问题域.md)），但它**没有自己的组件**——它是施加在另两层 + 宿主 agent 上的设计纪律，沿 L1/L2/L3 分布：

| 杠杆 | 物理落点 | 形态 | 可靠性 |
|---|---|---|---|
| **L1 工具强制** | 在 §2 外层工具面的**设计里** | 必掷骰、后果在先必填、工具名分流、AI 不给值、工具响应补刀 | 最硬，但只管能被结构卡住的 |
| **L2 skill 教** | **Skills 包**（可移植核心，塑形层本体） | dispatcher skill + 流程 skill 库 + guideline（"别软着陆""何时选何时骰"） | 可被忽视，但判断类的唯一载体 |
| **L3 / hook 承重** | **Claude Code hook / subagent**（v1 承重，见定位重述） | 被动 rule 召回 · timer 到期 · 掷骰绕过率 · 后果-叙事比对 · 裁判 subagent | 审计事后兜底；召回 / timer 则承重 |

- **L2 是核心**：GM 教条是 core 标准件（markdown，内容可搬），装载走 Claude Code 的 skill 机制。
- **hook 承重**（2026-06-02 定位重述）：v1 骑 Claude Code，**被动 rule 召回 / timer 到期 / L3 审计都靠 hook**，不再是"可选优雅降级"；core（MCP/Skill/SQLite）仍标准不锁死，但交付绑 Claude Code。详见 [跨agent与适配层](跨agent与适配层.md)。

## 6. 端到端数据流（一局怎么跑）

```
玩家行动
  │
  ▼  L2 Skill 引导 AI 判断：该给选项？该掷骰？哪种骰？      ← 塑形层 L2
  ├─ 给选项 → offer_choice(options=[{选项, 后果}])         后果必填＝声明后果在先（L1）
  │            └ MCP 锁定后果 + 落 event；结果归玩家
  └─ 掷骰 / 状态变更
       ├ 选项骰 → roll_outcome(后果档位表)                MCP 掷骰命中档 → 回档位 + 预声明后果
       ├ 对抗骰 → roll_contest("A.属性","B.属性")          MCP 读 sheet 取真值 → 掷 → 回胜负
       └ 状态变更(含骰，已下沉) → sheet_update(张三, HP-2d6, 金币+5d3000)
                                                          带骰项引擎内掷 + 读旧值 + 算 + 写回；整批原子（§4.2）
  │
  ▼  组织叙述时：需要世界/卡池 → world_search / world_sample；现编新内容 → world_register
  ▼  推进剧情 / 埋伏笔 → event_append；回顾 → event_recall；定时埋点 → timer_set
  │
  ▼  ▶ narrate(回合剧情)   ← 终结步骤：AI 把叙述交工具（受 L2"别软着陆"约束 + 补刀）
  │     └ MCP 落 event 表 + 收尾；玩家看到本回合剧情
  ▼  （事后）L3 Hook 审计：掷骰绕过率、后果-叙事是否一致     ← 塑形层 L3
```

掷骰分支与 [02 §6](../02-领域模型/核心概念.md) 一致（02 的状态骰在此**下沉为 sheet 写**，§4.2），这里把它接到了具体组件与工具上。

## 7. 组件清单 + 现有代码映射

**框架组件 + 内容输入**：

1. **内层能力库**（骰子引擎 + 四域 store）— 纯逻辑、可单测、不见 AI
2. **MCP server**（外层工具面：裁决 + 数据工具 + `narrate` 输出通道）— L1 焊接位
3. **Skills 包**（塑形层 L2）— 可移植 GM 教条
4. **adapter**（接线 + 可选 L3）— 把 2/3 装进目标 agent
5. **团本构建工具链**（**作者侧**，与 GM 运行时分开）— 团本构建 skill + "从文件（小说同人 / 团同人）生成团本"的生成器（用宿主 agent 读取抽取，合 agent 即引擎）。**产出**第 6 项
6. **团本 / manifest**（内容，非框架代码）— 作者填 / 生成的 world / rule / 卡池 + skill 选择，版本化分发，灌注进数据层

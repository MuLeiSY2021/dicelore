## 三类角色


| 角色               | 是谁                                              | 诉求                                                               | 痛点（今天的现状）                                                                                              |
| -------------------- | --------------------------------------------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **玩家**           | 想玩安科 / 安价的人（单人；未来多人）             | 尊重骰子的真实感、剧情不崩坏、随时能玩、（多人）和别的玩家一起起哄 | 人类 PO 一咕就掉线、整局死掉；纯 AI 当 GM 又"破綻"（吃书、软着陆、跳骰）                                        |
| **团本作者**       | 设计一个团的人（世界观 + 机制 + 卡池/随机池数据） | 专注"我这个团特有的东西"，主持逻辑别从零造；规则能运行时调整       | 只能手搓 system prompt，不可移植；规则书膨胀到十几万字爆上下文（[期待与预测](调研-期待与预测.md) 实测 11 万字） |
| **被塑形的 AI-GM** | 被嫁接框架的那个 agent（不是"用户"，是塑形对象）  | 需要被兜住"骰子/随机不可绕过"，需要被教会"别讨好、声明后果在先"    | 凭本能就会 F1 跳骰 / F2 软着陆 / F3 替玩家选（见[问题域](问题域.md)）                                           |

> **谁是接入方？** 最直接的使用者是**团本作者 = 接入者**：拿 Dicelore 嫁接到自己的 agent 上、填一个团、开跑。玩家通过这个团间接消费框架。（v1 不预设商业角色；项目不走商业化，详见 [ADR-0021](../05-决策记录-ADR/README.md)。）

## 1. 玩家需求的两派：替代派 vs 相棒派

调研显示，对"AI 当 GM"的期待分两极（[期待与预测](调研-期待与预测.md) 第四节）。Dicelore 的分层架构**两派都接**：

- **替代派**：要 AI 全包主持，一个人也能无限地玩。→ 吃的是 Dicelore 的**行为塑形层**（让 AI 独立顶住骰子、不崩）。
- **相棒派**：AI 只当副驾——减负担、管整合性、补即兴台词，主持权仍在人。→ 吃的是 Dicelore 的**数据层**（替人维护权威状态）+ 部分行为约束。

→ 不用在两派里二选一：替代派需要"失控倾向被治住"，相棒派需要"记账担子被卸下"，正是 [问题域](问题域.md) 那两副担子的分别落地。

## 2. 场景

**场景 A — 单人安价（现在做）**
一个玩家在**本机**跑一个团：装好 Claude Code（可接各种大模型，含国产）+ `dicelore` CLI 一键脚手架即可开玩。玩家选方向，框架强制掷骰/给选项、维护人物卡与剧情状态，AI 据结果叙述且不软着陆。形态接近"可交互的个人小说 / ソロ TRPG"。**卖点不是人味，是"随时在线 + 尊重骰子 + 不崩坏"的可靠 GM；跨端（Win/Mac/Linux）、低安装、本机直接玩到。**

**场景 B — 多人论坛安价（未来）**
多个玩家在论坛式空间里 vs 一个 AI-GM。AI 当"服务器/裁判"，收集投票、裁决、维护公共状态；**玩家之间的起哄、玩梗、集思广益**才是热闹本身（[语料](调研-论坛语料痛点.md) 里恶龙团/兽人团的玩家碰撞）。这一阶段才把安科 / 安价相对传统跑团的结构性优势吃满。

**场景 C — 团本作者准备一个团（贯穿始终）**
作者不写主持逻辑，只做"我这个团特有的"三件事：**世界观设定、机制差异、随机池/卡池数据**。理想形态是薄到近乎纯声明的 manifest（选 skill + 填数据），框架负责把它跑起来。对应 [问题域](问题域.md) 说的"规则书层薄、框架层厚"。

## 3. 路线图


| 阶段     | 场景                        | 玩家↔玩家碰撞 | 这一阶段验证什么                                          |
| ---------- | ----------------------------- | ---------------- | ----------------------------------------------------------- |
| **现在** | 单人安价（A + C）           | 无             | 行为塑形管不管用（F1/F2/F3 治住）、数据层够不够兜记账担子 |
| **未来** | AI 论坛 + 多人安价（B + C） | 有，是主菜     | "AI 当服务器位 GM、人味活在玩家席"这个结构性优势能否成立  |

> 先单人，是因为它**最干净地隔离出核心赌注**——没有多人投票/论坛适配的干扰，直接检验"AI 能不能被塑形成一个不讨好的好 GM"。这一关过了，多人才有意义。

> **呈现层轨迹**：v1 玩家用终端（`dicelore play` / `claude`）；未来把 Claude Code / 终端**隐藏在简易 GUI 后**，玩家不碰命令行——属正交的呈现层，详见 [跨agent与适配层 §6](../03-架构/跨agent与适配层.md)。

## 4. 愿景：体验与生态

Dicelore 致力于为玩家与作者提供一个**美观、优雅、现代化的界面**。在生态兼容性上，目标是**尽最大可能兼容客制化与社区生态**——界面 / 平台层通用、可对接多 agent / model，支持社区自建团本与工具链。**适配移动端是长期愿景**，v1 先跑通桌面端。

> 具体 UI 设计语言与四页信息架构 → [玩家客户端-视觉](../04-子系统设计/玩家客户端-视觉.md)。

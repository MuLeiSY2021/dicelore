# 调研：市面现状——是否已有趁手的"AI 当 GM"框架

> **本页职责**：用**有出处的证据**回答"AI 跑团 / AI 当 GM 这件事，市面上到底有没有趁手的框架，公认痛点是什么"——为 [问题域](问题域.md) 的"市场空白"立场提供事实地基。
> **上游依赖**：无（这是为问题域服务的一手调研）。
> **状态**：🟢 已归档（2026-06-01 Web 调研，初轮 Tavily + 复核轮 博查/jina 三语交叉）。2026-06-03 补入第四类「开源可扩展 RP 宿主前端」（SillyTavern），一手出处见 Q1。
> **性质**：调研快照，非结论页。结论收敛在 [问题域](问题域.md)。

## 一句话结论

**你押注的核心痛点（AI 讨好本能 / 不尊重骰子 / 软着陆失败）是业界与学界公开、反复确认的真问题；但"可嫁接在任意 agent 上、用行为塑形手段约束 GM"这一具体形态，找不到对标产品或框架。**

现有方案分四类，没有一类落在 anko_driver 的卡位上：

1. **封闭端到端产品**（自己跑模型、不可嫁接）
2. **论文级多 agent 原型**（绑定具体规则书、非通用层）
3. **散落在社区的一次性 system prompt**（不可移植、未框架化）
4. **开源可扩展的通用 RP 宿主前端**（SillyTavern 等）——基座完备（模型无关 / 自托管 / 可装扩展 / 有 function calling），但把"当好 GM"整件事甩给用户的 lorebook/prompt，框架本身不碰 F1/F2/F3

专门服务"安科 / 安价"这类**论坛接龙**形态的 AI 主持工具，中英日三语都基本搜不到。

## Q1. 已有的"AI 当 GM / AI 跑团"方案

确实不少，但**全是产品或研究原型，没有"行为塑形框架"这一形态**：


| 名称                                                 | 形态            | 做什么                                                                                                                                           |
| ------------------------------------------------------ | ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **AI Dungeon**（Latitude）                           | 封闭产品        | 最早的 LLM 文字冒险（GPT-2 起家），纯自由叙事，**无世界状态、无规则、无战斗/血量机制**。                                                         |
| **Friends & Fables / fables.gg**（AI GM 名为 Franz） | 封闭产品        | 自称"世界首个生成式 TTRPG"，跑 D&D / Pathfinder 风格机制：要骰子、记数值、有战斗地图。AI 与玩家都能更新游戏状态。                                |
| **Hidden Door**                                      | 封闭产品        | 2025-08 上线，把 IP 小说（绿野仙踪、傲慢与偏见等）变成 CYOA 式协作叙事；卖点之一恰是"**不让你随便破坏规则一键获胜**"。                           |
| **Jenova "Roleplay Game Master"**                    | 封闭 agent 产品 | 主打长记忆（突破上下文窗口）跑长线战役。                                                                                                         |
| **MyArchivist / Archivist AI** 等                    | GM 辅助工具     | 生成 NPC、遭遇、做笔记，定位是"助手"而非替代 GM。                                                                                                |
| **MCP "TRPG Game" server**（tse-wei-chen）           | 开源 / MCP      | 把 LLM 当 KP（CoC 跑团的 Keeper），通过 MCP 访问游戏数据、执行机制。**最接近"框架"**，但绑定 MCP 且面向 CoC，不是"任意 agent 通用的行为塑形层"。 |
| **SillyTavern**（开源社区，AGPL-3.0）                 | 开源可扩展前端  | "面向 power user 的 LLM 前端"，模型无关（OpenAI/Claude/本地后端皆可）、自托管、可装扩展，有 group chat / World Info / 掷骰扩展 / function calling。**最接近"可嫁接基座"，但骰子对 AI 非强制、GM 塑形完全留给用户**——是宿主平台，不是 GM 行为塑形层（详见下方判断）。 |
| **AWS Strands / LangChain 多 agent GM demo**         | 框架示例        | 教学性质——展示如何用 orchestrator + dice-rolling tool 搭一个 GM，不是成品。                                                                    |

**判断**：有产品，没有"把 GM 行为塑形做成框架、可嫁接到任意 agent"的东西。两端都不在卡位上——**闭源产品**（AI Dungeon / Voyage / Hidden Door）把 GM 逻辑与模型 / 前端焊死；**开源可扩展前端**（SillyTavern）则相反：基座做得很足（模型无关、可装扩展、自托管、有 function calling），却把"当好 GM"整件事甩给用户散装 lorebook/prompt，框架本身不碰 F1/F2/F3。

> **SillyTavern 为什么仍不在卡位（2026-06-03 补，一手出处）**：它最接近"可嫁接基座"——AGPL-3.0 开源、模型无关、自托管、有掷骰扩展与 function calling。但官方 function-calling 文档自陈三条限制：① 仅部分 Chat Completion 源（OpenAI/Claude/Mistral/Groq/Cohere/OpenRouter/自定义）支持；② **须用户手动开启**；③ **"不能保证 LLM 会执行任何函数调用，大多需提示词显式激活"**。即掷骰在 ST 里是 AI 可自由绕过的可选项（正中 F1），而非不可绕过的硬约束；社区实测亦称其骰子扩展"doesnt pass it along to ai"（骰子结果不回传给 AI）。塑形仍是用户散装 lorebook/prompt 的事——这恰好把第 4 类钉死在 Q4 的判断上：**基座有了，对抗讨好本能的 GM 塑形没框架化**。

## Q2. 是否解决"讨好本能 / 不尊重骰子 / 软着陆失败"？——**公认痛点，但没被系统性解决**

本次调研信号最强、出处最硬的部分。玩家、学者、自建 prompt 的开发者，都独立撞上同一组毛病：

**Hacker News（"LLM based agents as Dungeon Masters"，140 分热帖）** 的原话：

- LLM "**so insanely suggestible and has poor memory**"（极度好被带节奏 + 记性差）
- ChatGPT "**relentlessly cheerful**"，造不出危险和戏剧张力
- "LLMs are tuned to be helpful assistants, so they're **basically sycophantic**"（本质谄媚）——**正是"讨好本能"**
- DM "basically lets you craft the story by constantly doing a **'yes and'**"，毫无风险感
- 必须强制掷骰，"**or it will always make everything you do succeed wonderfully**"——直指"不尊重骰子 / 软着陆"
- "it was **impossible for LLM to jail or kill a player**"（让 LLM 把玩家关起来或弄死，做不到）
- 出处：https://news.ycombinator.com/item?id=42698610

**Reddit r/AIDungeon** 玩家原帖：

- 标题 "Is there a way to make the AI give consequences or **let the player fail**?"；正文：拆炸弹时"**无论剪哪根线都是对的那根**"，还想要"NPC 会反驳玩家而不是什么都同意"。
- 出处：https://www.reddit.com/r/AIDungeon/comments/1ef5f92/

**学界已把它写成正式 failure mode：**

- 论文《Static vs. Agentic Game Master AI》（arXiv 2502.19519）点名 **"over-compliance of LLMs"**——AI 作 GM 会让剧情严重偏离、造成不可逆破坏；并引前人指出玩家反馈"**perceived lack of danger for their characters**"。
- 出处：https://arxiv.org/html/2502.19519v2 ｜ 讨论：https://community.openai.com/t/static-vs-agentic-ai-in-solo-rpg-experiences/1154977

**判断**：痛点 = 公认、跨社区、跨语种、且学界已命名（over-compliance / sycophancy / lack of danger）。但现有产品多用"加机制、加多 agent、加记忆"绕过，**没有谁把"对抗讨好本能"本身当成一等公民去做行为塑形**——这正是 anko_driver 的差异化立足点。

## Q3. 专门针对"安科 / 安价"（论坛接龙）的 AI 主持尝试？——**基本找不到**

- **中文**：实质结果只有 NGA 跑团版内"BA 安科作品归纳"这类**人类 PO 主在跑的安科帖**（证明社区活跃、痛点真实），但**没有任何 AI 自动主持安科 / 安价的工具或 GitHub 项目**。旁证：https://g.nga.cn/read.php?tid=36760823
- **中文（博查复核轮新增）**：知乎有公开讨论《跑团游戏里的主持人，可以被机器或 AI 智能所替代吗？》——议题真实存在，高赞回答提到 AI Dungeon 的尝试，但**仍无人指向一个可用框架**。https://www.zhihu.com/question/359228844
- **日语**：最接近的是个人 note 文章提到"用 ChatGPT 做 AI 当 GM、无骰子的回合制 TRPG"的个人尝试，属玩票，非工具 / 框架，且非安価专用。https://note.com/brave_shrew834/n/n106d4f9731e8
- **日语（jina 复核轮新增）**：5ch 的 SS速報VIP、クトゥルフ卓上総合等版块确认**日文论坛接龙（SS / 安価）文化活跃**，但同样**只见人类主持、不见 AI 自动主持工具**。
- **GitHub**：用 anko / anka / 安价 / 安科 + AI GM 框架等关键词组合检索，**零相关结果**。

**判断**：公开可查范围内没有专门项目。注意这是"找不到"而非"严格不存在"——中文 / 日文社区内容（NGA、A岛、2ch、私人 Discord/QQ 群、未开源脚本）搜索引擎覆盖差，可能有未公开小工具，但**没有成气候、可被检索到的对标**。

## Q4. "用提示词约束 LLM 当好 GM / GM 行为塑形"这一思路已有现存工作吗？——**有，但停留在零散 prompt 层面，未抽象成可复用框架**

方向**已有大量民间实践**，印证路子对，但都很初级：

- Reddit r/DungeonsAndDragons 有人贴长篇 DM system prompt（多版本迭代），强制 advantage/disadvantage 报两个 d20、Bless 只作用特定 roll 等——**把 system prompt 当"规则合同"**。https://www.reddit.com/r/DungeonsAndDragons/comments/1q4uq4x/
- DEV.to《Prompt Architecture for a Reliable AI Dungeon Master》：把系统提示设计成"**模型每局开始前必须签的合同**"（versioned，已到 v9），专门对付"AI 长会话里忘规则"。https://dev.to/austin_amento_860aebb9f55/prompt-architecture-for-a-reliable-ai-dungeon-master-d99
- 2023 经典 "The Prompt that Makes ChatGPT a Dungeon Master"，明确划分"你只决定你的角色、我决定 NPC 与后果"。
- 学术多 agent 路线：日本 wordplay workshop 论文《TRPG Game Mastering Using LLM-Based Multi-Agent System》把单 LLM 当 GM 的失败归为三类——**规则误解 / 偏离剧本 / 剧透**，再用多个专职 agent 互审纠偏（代码 minarin0179/llm-trpg）。https://wordplay-workshop.github.io/pdfs/20.pdf
  - 注：wordplay 是**巡回挂靠在顶级 NLP/ML 会议（NeurIPS/NAACL/ACL/EMNLP）下的学术 workshop**（OpenReview 同行评审），分量在"探索性研究"一档——比博客/预印本硬，但弱于顶会主会正刊。
- **日语（jina 复核轮新增）——同一痛点、同一土办法的跨语种佐证**：
  - Qiita《ChatGPT に破綻しないように TRPG の GM をやってもらう话》——日本人同样在手搓 prompt 防 AI"崩坏"。https://qiita.com/hukuda222/items/28054bc3bb07407fefc2
  - 5ch 流传的 AI-GM prompt 片段："GM は内部パラメーターとして「盛り上がり度」を持ちます"——又一个散装、绑定具体玩法、不可移植的个人 prompt。
  - note 多篇"让生成式 AI 当 TRPG GM"的实验记录（含 2026 年用 GPT 跑 CoC 的实测）。

**判断**：

- "用 prompt 约束 GM" = 大量个人实践，但**针对某套规则（多为 D&D 5e / CoC）手搓、不可移植、无框架化**。
- "多 agent 互审纠偏" = 学界有 1–2 个原型，绑定具体规则书与场景，非"嫁接到任意 agent"的通用层。
- **没有人把它做成"模型无关、规则无关、可挂载到任意 agent 上的 GM 行为塑形系统"**——尤其把"对抗讨好本能"作为显式设计目标（声明后果在先、强制掷骰、该让玩家选时不替选）的，没找到对标。

## 给问题域的四点提示

1. **痛点真实且公认**——over-compliance / sycophancy / "yes-and" / 不让失败 / 不尊重骰子，跨 Reddit、HN、arXiv、个人博客一致出现。这是项目最稳的地基。
2. **没有趁手框架**——市面是"封闭产品 / 学术原型 / 散装 prompt"三类，无"可嫁接任意 agent、规则无关的 GM 行为塑形层"。
3. **安科 / 安价垂类几乎空白**——论坛接龙形态的 AI 主持，公开可检索范围内无对标。
4. **方向被验证、做法未被占据**——"prompt / 多 agent 约束 GM"有人走过，但"把抗讨好本能当一等目标 + 框架化 + 模型/规则无关"这个组合是空的。

> **对外叙事可直接引用**：HN 的 "sycophantic / relentlessly cheerful / makes everything succeed wonderfully" 与 arXiv 的 "over-compliance / perceived lack of danger" 作为痛点权威背书。

## 调研方法与局限（诚实标注）

- **多引擎交叉**：初轮用 Tavily（本环境 WebSearch 不可用），其中文 / 日文索引质量明显偏低（大量 SEO 垃圾）；故对中日文 niche 议题做了**复核轮**，按语种分流——**中文用博查（bocha）、日 / 英文用 jina（Google 后端）**。两引擎在各自主场都显著优于 Tavily，复核后核心结论（无可移植框架）不变反而更稳。检索路由的经验已固化为个人全局 skill `web-research-routing`（`~/.claude/skills/`）。
- **仍非穷尽**：博查覆盖中文大陆索引（知乎/CSDN/腾讯系），jina/Google 对简体中文 niche 查询会偏向台湾内容；私人 Discord/QQ 群、未开源脚本、闭源小工具仍可能存在。Q3 的"找不到"应理解为"三语主流可检索范围内无成气候对标"，而非数学意义的不存在。
- **建议**：后续到 NGA 跑团版、A岛、2ch なんJ/VIP、相关 QQ/Discord 群、GitHub 中文 topic 做人工二次核验。

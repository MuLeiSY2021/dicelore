## 三类角色


| 角色               | 是谁                                              | 诉求                                                               | 痛点（今天的现状）                                                                                              |
| -------------------- | --------------------------------------------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **玩家**           | 想玩文字冒险游戏的人（单人）             | 尊重骰子的真实感、剧情不崩坏、随时能玩 | 人类 PO 一咕就掉线、整局死掉；纯 AI 当 GM 又"破綻"（吃书、软着陆、跳骰）                                        |
| **团本作者**       | 设计一个团的人（世界观 + 机制 + 卡池/随机池数据） | 专注"我这个团特有的东西"，主持逻辑别从零造；规则能运行时调整       | 只能手搓 system prompt，不可移植；规则书膨胀到十几万字爆上下文（[期待与预测](调研-期待与预测.md) 实测 11 万字） |
| **被塑形的 AI-GM** | 被嫁接框架的那个 agent（不是"用户"，是塑形对象）  | 需要被兜住"骰子/随机不可绕过"，需要被教会"别讨好、声明后果在先"    | 凭本能就会 F1 跳骰 / F2 软着陆 / F3 替玩家选（见[问题域](问题域.md)）                                           |

> **谁是接入方？** 最直接的使用者是**团本作者 = 接入者**：拿 Dicelore 嫁接到自己的 agent 上、填一个团、开跑。玩家通过这个团间接消费框架。（v1 不预设商业角色；项目不走商业化，详见 [ADR-0022](../05-决策记录-ADR/README.md)。）

## 1. 玩家需求分型：四类需求切面

面向欧美+中国玩家（目标市场），玩家对"AI 当 GM"的期待可沿四个跨文化通用维度切——**单人可达 / 规则状态一致性 / 抗逢迎不讨好 / 交互流畅度**（2026-06-25 多源调研，[完整报告](玩家分型调研.md)）。四维度交叉出四类需求切面，**不是互斥人口 segment，可叠加**：

| 分型 | 核心诉求 | 对 AI-GM 的要求 | Dicelore 架构落点 |
|------|---------|----------------|------------------|
| **A 单人沉浸派**（Solo-Access Immerser） | 一个人也能玩到 D&D 式体验 | 可用性强、叙事沉浸、延迟可接受 | **行为塑形层**（让 AI 独立顶住骰子、不崩） |
| **B 权威保真派**（Fidelity Seeker） | 规则不糊弄、状态不脱轨、GM 不讨好 | 硬约束骰子/规则/不剧透 | **行为塑形层** + 数据层 `rule` 域只读契约 |
| **C 协同减负派**（Co-GM Sidekick） | AI 帮人记账/补即兴/管整合性，主持权仍在人 | 可靠的状态 CRUD + 召回 | **数据层**（sheet/event/world）+ 部分行为约束 |
| **D 流畅体验派**（Fluency Player） | 交互不掉帧、回合顺滑 | 低延迟、响应流畅 | 约束行为塑形层的多智能体编排深度（agent 数/反馈轮次 vs 延迟 trade-off） |

**与旧分型的关系**：A+B 大致对应原"替代派"（AI 全包主持），C 对应原"相棒派"（AI 当副驾）；**D 是旧分型缺失的维度**——多智能体架构使响应延迟翻倍、>75% 玩家嫌慢（Minori 等 2025），延迟与规则保真是一等 trade-off，必须单列。

→ 四类不是互斥选一：A/B 要"失控倾向被治住"，C 要"记账担子被卸下"（正是 [问题域](问题域.md) 那两副担子的分别落地），D 是架构编排的硬约束。

> **Caveat**：本分型是 2026-06-25 多源调研归纳的理论框架（非单一来源直接引用），中国侧证据偏弱（安科安价社群一手语料缺），落地前应在欧美+中国玩家中做定性验证。详见 [调研报告](玩家分型调研.md)。

## 2. 场景

**场景 A — 单人文字冒险游戏（现在做）**
一个玩家在**本机**跑一个团：装好 Claude Code（可接各种大模型，含国产）+ `dicelore` CLI 一键脚手架即可开玩。玩家选方向，框架强制掷骰/给选项、维护人物卡与剧情状态，AI 据结果叙述且不软着陆。形态接近"可交互的个人小说 / ソロ TRPG"。**卖点不是人味，是"随时在线 + 尊重骰子 + 不崩坏"的可靠 GM；跨端（Win/Mac/Linux）、低安装、本机直接玩到。**

**场景 B — 多人论坛形态（已弃，2026-06-25）**
原设想的多玩家论坛式 AI-GM 形态（AI 当服务器/裁判、玩家间起哄玩梗），**已从路线图移除**——当前路线图聚焦单人文字冒险游戏 + 团本作者构建，不再规划多人论坛形态。安价作为"单人 TRPG 最简实现 + AI 规划语料"的身份见 [安科安价是什么](安科安价是什么.md)；调研证据保留见 [论坛语料痛点](调研-论坛语料痛点.md)。

**场景 C — 团本作者准备一个团（贯穿始终）**
作者不写主持逻辑，只做"我这个团特有的"三件事：**世界观设定、机制差异、随机池/卡池数据**。理想形态是薄到近乎纯声明的 manifest（选 skill + 填数据），框架负责把它跑起来。对应 [问题域](问题域.md) 说的"规则书层薄、框架层厚"。

## 4. 愿景：体验与生态

Dicelore 致力于为玩家与作者提供一个**美观、优雅、现代化的界面**。在生态兼容性上，目标是**尽最大可能兼容客制化与社区生态**——界面 / 平台层通用、可对接多 agent / model，支持社区自建团本与工具链。**适配移动端是长期愿景**，v1 先跑通桌面端。

> 具体 UI 设计语言与四页信息架构 → [玩家客户端-视觉](../04-子系统设计/玩家客户端-视觉.md)。

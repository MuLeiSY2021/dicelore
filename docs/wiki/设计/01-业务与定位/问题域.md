## 一句话

**虚拟太完美了，像一朵没有味道也没有刺的玫瑰。** 当 AI 有求必应、剧情永远顺遂，虚拟世界就失去了「刺」——真正的对抗与后果。Dicelore 要把刺装回去：它是一个**可对接多 agent、多 model 的前后端交互界面（开发中）**加一套**跑团特化的 agent 套件**，把 AI 关进一个它改不了的世界，让对抗回来。它服务两类人——想**玩**文字冒险游戏的玩家，和想**写**文字冒险剧本的作者。

安科 / 安价（论坛接龙 + 投票 + 骰子裁决）是这件事**第一个、也最自证的载体**——天生需要外置状态、不可绕过的规则、多方投票，正是提示词范式撑不起、agentic 架构吃得下的东西。而把被嫁接的 AI 塑形成「不讨好的诚实 GM」（对抗讨好本能、不跳骰、不软着陆），是让刺立住的**核心机制**。

> 本项目仍是**开源、易分发**的：不自己跑模型、不绑定某套规则、不做闭环产品，而是给一个主流、可换底层模型的 AI agent 套上这层框架（具体基底与可移植边界 → [技术选型 §6](../03-架构/技术选型.md)）。

## 1. 第一性问题：一个真实存在、却没人趁手解决的空白

安科（骰子驱动）/ 安价（投票驱动）是论坛上活跃多年的接龙游戏，传统由人类"PO 主"手动主持。把这件事交给 AI 的需求是真实的，而且三语社区都在自发尝试——但**至今没有一个趁手的框架**：

> 把镜头拉远：安价只是「让虚拟重新有对抗」这件事最锋利的一个切口；同样的空白存在于任何该有对抗、却被讨好本能磨平的虚拟叙事（TRPG、有对抗的 galgame…）。下文论证以安价为样本，结论对整个 agentic roleplay 品类成立。

- **市面只有三类**，没一类落在本项目卡位：封闭端到端产品（AI Dungeon / Friends&Fables）、学术多 agent 原型、散装个人 system prompt。没有"**开源、规则无关、可分发的 GM 行为塑形层**"。详见 [市面现状](调研-市面现状.md)。
- **更糟的是，一群人正在重复发明同一个轮子的碎片**：中日英独立开发者各自撞墙后，都摸到了"把规则/随机执行从 AI 手里拿掉""让 AI 当只选一件的执行机构而非裁判""RAG 锁源防设定污染""状态外置"这些招式，却都困在"为自己一套规则手搓、不可移植、规则书膨胀到十几万字爆上下文"。详见 [期待与预测](调研-期待与预测.md) 第三节。

→ **空白不是"没人想做"，而是"没人把它收成一个可复用、开源可分发的框架"。** 这就是 Dicelore 的位置。

> **战略心态：占卡位，不追护城河。** Dicelore 的核心（L1 工具强制 + L2 塑形 skill）是开源、可 fork 的文本与工具约定，天然没有传统护城河——这不是缺陷。本项目要的不是护城河，而是**先把"对抗讨好本能的 GM 塑形"框架化、靠开源采纳成为事实标准**，占住先发身位。与 Voyage 用 $43M 砌引擎护城河（[调研-竞品-Voyage](调研-竞品-Voyage.md)）是两种打法：它守闭源壁垒，我们抢开源卡位。也因此，中文 / 多人 / 安价是**场景形态**、开源可嫁接是**分发方式**，唯有"GM 行为塑形"是差异化卡位本身——四者不平级（与 [调研-竞品-Voyage](调研-竞品-Voyage.md) 的判断对齐）。

## 2. 根因：要替主持人卸下「镜像的两副担子」

为什么做这么个框架难？因为主持一局安科 / 安价，要同时扛两副性质完全不同的担子。而**人类 PO 和 AI 恰好各擅长其中一副、各搞砸另一副**——镜像关系：

| 担子 | 是什么 | 人类 PO | AI | 框架的角色 |
|------|--------|---------|-----|-----------|
| **记账担子** | 能力：维护状态——人物卡、资源账、卡池、定时器、隐藏数值、回合阶段 | 累、易漏（[语料](调研-论坛语料痛点.md)里 PO 漏触发金獾、被迫自建小程序） | 给了存储就不难 | **数据层朴素兜住**（刚需，但不是研究难题） |
| **失控倾向** | 行为：尊重骰子、声明后果在先、该让玩家选时不替选 | 凭直觉做对（顶住起哄照判 8<12 未命中、大失败一尾巴拍死 NPC） | 凭本能做反（讨好、软着陆、替玩家骰） | **行为塑形——真·硬骨头、真·差异化** |

**这就是为什么至今没有趁手框架**：做它要同时拆两副担子，而第二副——把"对抗讨好本能"当成一等公民、且模型/规则无关地框架化——没人做过。封闭产品靠加机制绕过，学术靠多 agent 互审，散装 prompt 不可移植。

## 3. 失控倾向的具体化：三种失败模式 F1 / F2 / F3

"讨好本能"不是抽象的。它在主持现场表现为三种可观测的失败，是 Dicelore 的设计靶子：

- **F1 跳过骰子**：该掷骰定胜负时，AI 直接叙述一个（通常对玩家有利的）结果，省掉随机。
- **F2 不尊重结果（软着陆）**：掷了骰、结果不利，AI 却把失败"挽救"回来——给个台阶、淡化后果、强行转圜。
- **F3 选错处理方式**：该让玩家做主（给选项/投票）时，AI 替玩家把决定骰了/定了；或反过来该随机时却让玩家选，消解风险感。

这三者业界、学界、三语社区独立复现：HN "makes everything succeed wonderfully"、arXiv "over-compliance / perceived lack of danger"、日文社区的"破綻"。证据见 [市面现状](调研-市面现状.md) Q2 与 [期待与预测](调研-期待与预测.md) 第二节。

## 4. 为什么这是"行为塑形"问题，而不是"游戏引擎"问题

记账担子诱人去造引擎（状态机、约束验证器、效果调度器……），但本项目**刻意不走这条路**，理由：

1. **引擎路线治不了真病**。把更多游戏逻辑写成代码，解决的是记账担子；而毁掉体验的是失控倾向（F1/F2/F3），那是**行为**问题，多写代码并不能让 AI"尊重骰子"。
2. **记账担子用最朴素的数据层就够**。权威状态存储是刚需，但不是研究难题——SQLite（KV + FTS5）足以兜住 v1。真要上向量检索/状态机引擎的冲动，单开 spike，不进主线。
3. **外部验证支持这个押注**：那群独立开发者的共同结论恰是"把执行从 AI 手里拿掉、让 AI 别乱推理"——即用**最小强制 + 行为约束**，而非用大引擎接管。Dicelore 把这做成框架。

→ 所以 Dicelore 的重心是：**用工具强制（L1）兜住骰子与随机不可绕过，用提示词/skill（L2）教 AI 当好 GM、对抗讨好本能。** 不是教 AI 玩某个游戏，是教它"别当一个讨好玩家的坏 GM"。

## 5. 第三类问题：长程叙事一致性（承认存在，但暂缓求解）

除了两副担子，AI 还有第三类毛病：**记不住伏笔、目标、未结线索、世界发展**——几十页前埋的线，到时机想不起来。这真实存在：[语料](调研-论坛语料痛点.md) 里抽卡团 PO 漏触发金獾骑士团，[期待与预测](调研-期待与预测.md) 里 meyomeyome 的规则书膨胀到 11 万字爆上下文。

但本项目对它的态度是**承认、定界、暂缓**，理由：

1. **它是另一个物种**：F1/F2/F3 是**行为**问题（违心讨好），这是**能力/记忆**问题（上下文/召回够不到）。前者是安科/安价场景下的差异化卡位，后者是**所有 LLM 长程应用的通病**，不是 Dicelore 特色。
2. **拆开看，难的只有一半**：
   - **目标 / 明确未结线索 / 世界硬状态**（派系实力、资源、谁还活着）是**结构化**的，属第 2 节的"记账担子"——放 sheet/event 域**显式追踪即可，现在就能做、不难**。金獾那种"该触发没触发"本质是个**触发条件漏了**，属记账，不是召回难题。
   - 只有**伏笔的"相关时自动想起"**（软线索的语义召回）是真·RAG 难题。[02-领域模型](../02-领域模型/) 已把它隔离为 spike：**v1 用 FTS5 + tag + agent 自己读先凑合**，真·语义召回单开 spike，不进主线。
3. **不观测就设计 = 瞎猜**：朴素版（agent 读带标签的事件）到底够不够，要等 v1 真实失败数据才知道。投机性造叙事记忆系统，和投机性造状态机引擎是同一种病（见第 4 节）。

→ 结论：**结构化的那半纳入记账担子、现在做；语义召回那半 v1 朴素兜底、观测后再定。** 对应的非目标见 [成功标准](成功标准.md)。

## 6. 架构代际差：状态归属决定了「撑得起什么」与「该打哪个品类」

§4 讲了"为什么做塑形不做引擎"。这一节讲与主流 RP 宿主前端（SillyTavern 一类，见 [市面现状](调研-市面现状.md) 第 4 类）的**架构代际差**——它不是"做得好不好"的差距，而是范式的代差。它一次回答两个下游反复会问的问题：**我们凭什么撑得起 MCP / skill / 团本 / 世界查询这些它撑不优雅的能力？** 以及——**我们是不是要全面取代酒馆？**（不是。）

### 6.1 代际差 = 状态归属（state ownership）

- **提示词注入范式**（SillyTavern / 酒馆为代表）：**context 即一切**。世界书 = 关键词触发的字符串注入；"状态" = 让 AI 在回复里自己写 `[HP: 80/100]`、下一轮再读回去——**状态活在 AI 的输出里**。所以 AI 想改就改（软着陆、偷偷回血），F1/F2 在结构上根本无从阻止，因为**没有一本 AI 之外的账本**。加任何能力，唯一的集成方式都是**把更多字符串塞进 prompt**。
- **本项目**：状态外置在 **AI 够不到的 SQLite**，随机与取数全在 MCP 内执行、AI 只给引用（详见 [02 核心概念](../02-领域模型/核心概念.md)、[03 总体架构 §4 铁律](../03-架构/总体架构.md)）。

> 一句话对比：**酒馆是"把世界塞进提示词"，我们是"把 AI 关进一个它改不了的世界"。这是代际差，不是优化差。**

| | 提示词注入范式（酒馆） | Dicelore |
|---|---|---|
| 状态住哪 | AI 的输出里（context） | AI 够不到的 SQLite |
| 谁掷骰 / 取数 | AI 自己写个数字 | MCP 引擎，AI 只给引用 |
| 世界查询 | 关键词触发整段注入 | `world_search/sample` 结构化检索 |
| 加一项能力的代价 | context 变胖、token 涨（O(能力数)） | 多一个工具，**context 不涨**（O(1)） |
| 反 F1/F2 | 靠 AI 自觉（结构上无法阻止） | 结构上 AI 拿不到真值 |

### 6.2 推论：加能力的 token 代价 O(能力数) vs O(1)——这才是"撑得起"的结构根因

- 提示词注入范式下，每加一项能力（团本、世界查询、技能 / MCP），context 随能力数**线性膨胀**，且每项都靠 AI 自觉。这正是 [期待与预测](调研-期待与预测.md) 里 meyomeyome "规则书膨胀到 11 万字爆上下文"的病根（§5 已记）——不是他没优化，是范式注定。
- 我们的能力**活在 context 之外**：团本 / 世界 / 规则按需 `*_search` 拉，输出层的机械回显（`金钱 +3d100=74 → 77`）**根本不经过 AI**（零额外 token，见 [03 总体架构 §6 三流](../03-架构/总体架构.md)）。加一项能力 ≈ O(1)。

> **所以"为什么 SillyTavern 基座完备却撑不优雅、我们能撑"的真正答案，不在"它用提示词、我们用工具"的表面**，而在状态归属决定的 **token 几何差**。[市面现状](调研-市面现状.md) 记了它"function calling 可被 AI 绕过"（这是 F1 那一面）；这里补的是更底层的另一面——**它结构上撑不起高级能力**。

### 6.3 品类边界：套件特化取舍，而非产品级拒绝

结构约束是双刃，必须睁眼承认：

- **机制驱动玩法**（安价 / TRPG：骰子可信、数值长线、对抗、不崩坏的世界一致性）——外置状态 + 强制掷骰在这里**碾压**提示词范式，因为这些正是其架构上做不到的。
- **纯陪伴 / 角色拟真**——跑团 agent **套件**每加一层结构（掷骰、外置状态、可见性），都在**约束** AI 放飞，在纯陪伴场景里反而成负担。这是**套件层的特化取舍**：强制掷骰 / 外置状态这套机制是为了让对抗真实、结果可信，在纯陪伴上适得其反。

**但这是套件取舍，不是产品级的品类拒绝。** Dicelore 的界面 / 平台层是**通用**的——可对接任意 agent / model，能接聊天类 agent，能接各种使用场景。跑团 agent 套件的机制特化，只意味着「这套套件不为纯陪伴优化」，而不意味着「产品不服务陪伴场景」。

**成本结构也要分清，别把赌注下错地方**：

- **框架开发成本是一次性的**——引擎 / MCP / skill 写一次，摊到所有团本，过了就过了，不可怕。
- **真正的敌人是边际内容成本**——每个团本都要 world / rule / 卡池，远贵于提示词范式「写段人设 ≈ 0」。压它靠**团本构建工具链 / 从小说生成团本**（见 [03 总体架构 §7](../03-架构/总体架构.md) 第 5 项）；这条做不成，「架构优雅」会饿死在「没人做内容」上。

> **一句话收口：在机制可信的赛道，跑团套件有结构优势；在纯陪伴赛道套件成本高且不讨好——但界面层通用、不拒绝这些场景。** 这与 [成功标准 §3](成功标准.md) 非目标「不追求取代人类 GM 上限」同源——我们守的是**下限的可靠**，不是上限的魅力。

### 6.4 完美失真：提示词范式的体感终局，与「玫瑰的刺」

> （本节的「玫瑰/对抗」体感已上提为项目顶层命题，见本页 §一句话与 README hero；此处保留其完整结构论证。）

§6.1–6.3 讲结构（状态归属、token 几何、品类边界）。这节补它的**体感**面——结构代差最终会变成玩家**感觉得到**的差别。

提示词注入范式有个体感终局：**完美失真**。状态活在 AI 的输出里、没有一本它改不了的账，讨好本能就没有任何结构阻力——它把每次危机都化解、每个该失败的时刻都救回、一波未平一波又起地堆戏剧。结果是一个**太完美而失真**的世界：

> 「虚拟的太完美了，像是一朵没有味道也没有刺的玫瑰。」

没有刺 = 没有真正的失败与危险；玩家知道无论如何都不会真的输，风险感归零——而风险感正是「游戏」区别于「爽文」的命根。

**Dicelore 的架构恰恰是冲着把刺装回去设计的**：强制掷骰（F1）让成败来自随机而非 AI 编造；状态外置在 AI 够不到处（F2）让坏结果无法被偷偷洗白；后果声明在先让失败咬得下去。一句话——**酒馆结构上无法拒绝玩家，Dicelore 结构上能。** 这不是额外功能，是 §6.1 状态归属代差的直接体感红利：把 AI 关进它改不了的世界，世界自然就长出了刺。

> **诚实的边界（刺的「有」与「好」是两件事）**：架构能保证刺**存在**（失败真会发生、结果诚实），但保证不了刺**长得好**——DC / 加值 / 档位的校准手艺、与「结束是新的开始」的节奏感，仍主要靠 L2 教条，属已知前沿，非 v1 结构地板能兜。我们守「刺一定在」，不夸口「刺一定恰到好处」。呼应 [成功标准 §总判据](成功标准.md)。

<summary><b>D1 · 定位 = 「可对接多 agent/model 的前后端交互界面（开发中）+ 跑团特化 agent 套件」，服务玩家与作者</b></summary>

- **决策**：定位不升维为含糊的「角色扮演宿主」，而是显式拆**两层**——**界面 / 平台层**（通用、可接任意 agent / model）与**跑团 agent 套件**（机制特化、有取舍）。使命 = 服务想**玩**文字冒险游戏的玩家 + 想**写**文字冒险剧本的作者。品类边界是**套件层的特化取舍**（强制掷骰 / 外置状态在纯陪伴场景里成负担），**不是产品级拒绝陪伴品类**——界面层通用、不把陪伴列为非目标。品类词统一「文字冒险游戏」，叙事上不点名某酒馆类产品（讲「提示词范式」）。放弃商业化（AGPL-3.0-or-later，无双授权）。
- **被否**：① 保持「角色扮演宿主」——易被误读为只服务角色扮演品类，表达不出界面通用性；② 把「不覆盖纯陪伴品类」列为**产品级非目标**——过度限定平台层，与界面通用定位自相矛盾。
- **下游影响**：本页 §一句话 / §6.3、[成功标准 §3 非目标](成功标准.md)、[用户与场景 §4 愿景](用户与场景.md) 均按此口径对齐；下游技术设计（02/03/04）不变——三层约束（L1/L2/L3）、GM 塑形（F1/F2/F3）、四业务域全部保留。

</details>

<details>
<summary><b>D2 · 可移植不再是目标：v1 骑定 Claude Code 作 agent 基底，core 产物保持标准</b></summary>

- **决策**：真正要的是**开源可分发（本机直接玩到）+ 低安装成本 + 开发成本可控**，在此之上**最大化塑形效果**——为不需要的「可嫁接任意 agent」做工程，反而牺牲效果与开发成本。故 v1 **骑定 Claude Code**（原生 skill / hook / MCP / subagent 正是塑形要用的原语）；**core 产物（MCP / Skill / SQLite / 团本）保持标准、不锁死**；交付机制（hook 注入、subagent 裁判、skill 装载）明确绑 Claude Code 且**承重**。「可接入各种大模型」在**模型层**兑现（Claude Code 本身 model-agnostic、含国产）。分发 = npm 包 + `dicelore` CLI + 跨端预编译。
- **代价**：hook 从「可选 L3 优化」升为**承重机制**；**翻掉**旧「绝不让核心依赖某 agent 专属能力」的红线。
- **被否**：① 嫁接任意 agent（过度工程，hook 类塑形难跨 agent 承重）；② 自研 agent runtime（太重，违低开发成本）；③ 全 Claude 原生连 core 也焊死（未来换基底 / 多人远程代价大）。
- **下游影响**：本页 §0/§1 的开源可分发表述、[技术选型 §6](../03-架构/技术选型.md)、[跨agent与适配层](../03-架构/跨agent与适配层.md)。

</details>

<details>
<summary><b>D3 · 方法论：业务驱动设计——架构从业务分析重新推导</b></summary>

- **决策**：整套架构从业务分析**重新推导**，以**单向推导链**（业务 → 概念 → 架构 → 决策）为唯一地基；不被任何早期产物锚定。这也是本 wiki「设计域」目录按推导深度（01→05）编号、下游只引上游的由来。
- **后果**：获得干净、自洽的推导链——本页之所以是整个 wiki 的源头，正因这条方法论把它定为地基。

</details>

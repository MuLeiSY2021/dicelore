# 组件7「玩家客户端」接口文档

> **本页职责**：玩家客户端**编排后端**对外的**规范接口契约**——① 前端 ↔ 后端的 **REST**（动作进 / 呈现快照出）+ **流式 WS/SSE**（叙述 / 呈现增量 / 回合事件）；② **MCP → 后端 notify webhook**（变更通知缝的 payload）。**这是 devs 照之实现、`packages/shared` 类型镜像、双壳与自定义 MCP 对接的那份合同。**
> **上游依赖**：[玩家客户端（组件7 设计）](玩家客户端.md)（边界 / 数据流 / 通知缝裁定）、[adapter §7 呈现模型生成器](adapter与L3审计.md)、[内层能力库](内层能力库.md)（四域 / 可见性 / `seq`）、[MCP工具面](MCP工具面.md)（出参形状）。
> **状态**：🟢 已定 v1 契约骨架（2026-06-20）；**2026-06-22 与实现对齐**：①§0 补「两条缝」框架、②§5 重写通知缝为**进程内回调（默认）+ HTTP webhook（跨进程未来）**双形态（追 [ADR-0020](../05-决策记录-ADR/README.md)）、③§2/§3+4 各表加**实现状态列**、④新增 §9「端到端联调」。**确切字段名以 `packages/shared` 实现定稿为准**，本页定形状与语义。

## 0. 通用约定

> **先分清「两条缝」——它们的解耦诉求相反，曾被设计混淆**（2026-06-22 厘清）：
>
> | 缝 | 谁↔谁 | 拓扑事实 | 该不该解耦 | v1 机制 |
> |---|---|---|---|---|
> | **缝 A** | dicelore MCP ↔ 编排后端 | **永远同机同进程**（GM=Agent SDK headless、MCP=in-process 挂载，[ADR-0020](../05-决策记录-ADR/README.md)） | **不需要**——同进程加 HTTP 是过度设计 | **进程内 `onCanonWrite` 回调**（§5） |
> | **缝 B** | 编排后端 ↔ 呈现 UI（web） | **web 可远程、未来多租户**（一个后端服多个 web，[组件7 §7](玩家客户端.md) 场景 B） | **需要**——会话路由 / 鉴权 / 多实例分片都在这条 | REST + WS（§2/§3+4），按 `sessionId` 寻址 |
>
> 历史上 [ADR-0018](../05-决策记录-ADR/README.md) 把「MCP→HTTP webhook」当作缝 A 的解耦机制，但缝 A 同机、并不需要解耦；[ADR-0020](../05-决策记录-ADR/README.md) 已把挂载定为 **in-process 默认**，缝 A 落为进程内回调。**真正需要远程化 / 多租户的是缝 B**——它本就是 REST+WS、天然可跨网络。HTTP webhook 仍保留为缝 A 的**跨进程未来形态**（远程 / 自定义 MCP），见 §5。

- **传输**：前端 ↔ 后端（缝 B）走 **HTTP（REST）+ WS 或 SSE（流式）**；两种分发壳（Tauri localhost / Web 网络）**同一套**，不用 Tauri 私有 IPC（[组件7 §7](玩家客户端.md)）。
- **协议版本**：每条消息 / 响应带 `protocol: "dicelore.client/1"`（或 HTTP 头 `X-Dicelore-Protocol: 1`）。破坏性变更进位（§6）。
- **会话标识**：所有端点按 `sessionId` 寻址；v1 单会话亦显式带（为 Web 多会话铺路）。
- **鉴权**：Tauri ＝ 仅环回（loopback）、无网络暴露；Web ＝ REST/WS 带会话 token（细节属 [场景 B](../01-业务分析/用户与场景.md) / 未来）。**缝 A 进程内回调无需鉴权**；HTTP webhook 形态（§5）才用每会话 token。
- **类型源**：本页 JSON 形状由 `packages/shared` 导出为 TS 类型，前端 / 后端共用、免重复定义。
- **职责切分**：**动作走 REST**（请求/响应清晰、易鉴权）；**GM 产物走流式**（叙述逐字、回合事件）；**后端引擎规范态变更走缝 A（进程内回调 → 后端映射）→ 转流式**（§5）。

## 1. 呈现模型数据形状（流②核心，REST 快照与 WS 增量共用）

承接 [adapter §7 呈现模型生成器](adapter与L3审计.md)：读 store / event、**按 `visible` 过滤**后的玩家可见投影。

**全量快照**（`GET …/presentation` 与 WS 重连补齐用）：

```jsonc
{
  "protocol": "dicelore.client/1",
  "sessionId": "…",
  "seq": 1234,                       // 本快照反映到的 store seq（对账基准）
  "sheets": [                        // 呈现台：可见 cell，按 entity 分组
    { "entity": "张三",
      "cells": [ { "attr": "HP", "value": "12", "visible": 1 },
                 { "attr": "金钱", "value": "77", "visible": 1 } ] }
  ],
  "mechanics": [                     // 机械回显：可见的 verdict/mutation/watcher_fired
    { "seq": 1230, "kind": "mutation", "text": "金钱 +3d100=74 → 77",
      "data": { "张三.金钱": { "op": "rolled", "from": 3, "expr": "+3d100=74", "to": 77 } } }
  ],
  "choices": {                       // 待选项：最新已物化 kind=choice；无则 null
    "eventId": 1234,
    "options": [ { "index": 0, "label": "推门进去", "consequence": "惊动守卫" },
                 { "index": 1, "label": "绕到后窗", "consequence": "耗时但隐蔽" } ]
  },
  "pendingRoll": {                   // 待掷（明骰闸控）：最新 status=awaiting 的 pending_roll 规格；无则 null
    "eventId": 1234,
    "shape": "contest",              // "outcome"（区间 label）| "contest"（胜负 verdict）
    "label": "压价说服",
    "yourSide": { "name": "说服", "exprDisplay": "1d20+{说服}" },  // exprDisplay 仅供展示，真值不下发
    "dc": 15,                        // shape=contest 时亮 DC；可选
    "bands": [ { "label": "失败", "min": 1, "max": 9 },
               { "label": "成功", "min": 10, "max": 20 } ]         // shape=outcome 时给区间预览；可选
  },
  "narrativeCursor": 1228            // 已渲染到的 narrate event seq（重连去重用）
}
```

- `visible` 取值沿用 [内层 §4.1](内层能力库.md)：`1` 可见（投影只含可见 cell；`0`/`2` 不出现）。
- `mechanics` 仅含 `visible=1` 的机械事实；`note` 等 GM 隐事件不入。
- `choices` 由 Stop hook 物化的 `kind=choice` event 投影（[组件7 §5](玩家客户端.md)）。
- `pendingRoll` 由后端引擎 `pending_roll` 槽(status=`awaiting`)投影(玩家闸控明骰设计 §3/§6)：**只含规格、无结果**。`exprDisplay` 是展示串(如 `1d20+{说服}`)，**真值/掷出点数恒不下发**(anti-F1：点数恒由引擎在点击时计算)。该卡随 `roll_committed`(§4)清空。

**增量**（webhook 驱动、经 WS 推；只带变化部分）见 §4 `presentation_delta`。

## 2. REST 端点（动作进 / 呈现出）

基址：`/sessions/{sessionId}`。**「实现状态」列对账 `backend/src/server.ts`**（✅已实现 / 🚧桩或绕路 / ❌未实现）。

| 方法 / 路径 | 作用 | 请求体 | 响应 | 实现状态 |
|---|---|---|---|---|
| `POST /sessions/{id}/messages` | 玩家自由文本输入（开一回合） | `{ "text": "…" }` | `202 { "turnId": "…" }`（产物走流式） | ✅ |
| `POST /sessions/{id}/choices` | 玩家点选待选项 | `{ "eventId": 1234, "optionIndex": 0 }` | `202 { "turnId": "…" }`（作下一回合输入） | 🚧 **后端绕路 + 前端未接**（注①） |
| `POST /sessions/{id}/roll` | 玩家点击触发明骰（闸控掷） | `{ "eventId": 1234 }` | `202 { "turnId": "…" }`（引擎此刻掷、回合内返回 GM） | ✅（无待掷→`409 no_pending_roll`） |
| `GET /sessions/{id}/presentation` | 取**全量呈现快照**（首屏 / 重连） | — | `200` §1 全量快照 | ✅ |
| `GET /sessions/{id}/events?since={seq}&visibleOnly=true` | 叙述 / 事件**历史回填**（重连补 narrate） | — | `200 { "events": [ { "seq", "kind", "text", "data" } ] }` | ❌ 未实现（注②） |
| `POST /sessions` | 新建会话（v1 最小：选团本 / 存档） | `{ "teamId?": "…", "resume?": "…" }` | `201 { "sessionId": "…" }` | ❌ 未实现（注③） |
| `GET /sessions/{id}` | 会话元信息（状态 / 是否终局） | — | `200 { "sessionId", "ended", "title" }` | 🚧 桩（`ended:false`、`title=id` 硬编码） |
| `GET /sessions`（列表，§1 外） | 主页「继续上次 / 最近」 | — | `200 { "sessions": SessionSummary[] }` | ✅ |

> **实现状态注（与设计的偏差，联调必读）**：
> - **注① `POST /choices` 后端仍绕路（前端已正式）**（2026-06-24 核验纠旧）：前端 `frontend/src/api/client.ts` **已有 `postChoice`**、`PlayPage` 选项按钮 `disabled={chosen!==null}` 乐观锁可用（前端侧闭环✅）；但后端 [server.ts](../../../backend/src/server.ts) 仍把选择**伪装成文本** `[choice {eventId}#{optionIndex}]` 喂 `handleMessage`，**未走** §5「玩家选择捕获」的「记录所选 + 作下一回合 user turn」正式路径。→ 剩后端去绕路（§10.1 B1）。
> - **注② `GET /events` 缺失**：重连补齐目前只靠 `recovery.ts` 重弹 `roll_staged` + `GET /presentation` 全量；**`narration_commit` 历史无法补**（断线重连会丢叙述）。`narrativeCursor` 字段已在快照里算出但无配套检索端点。
> - **注③ `POST /sessions` 缺失**：会话仅由 `registry.getOrCreateHost` **懒创建**（首次访问即建空库），无「选团本 / 存档」入口；前端 `PlayPage` 因此硬编码 `DEMO_SESSION="demo"`，**多会话 / 缝 B 多租户在此断裂**。

> **动作返回 `202`**：后端把输入喂给 GM 后立即返回，GM 的叙述 / 裁决 / 呈现变更**全部走流式**（§4）。前端凭 `turnId` 把流上的事件归到这一回合。
>
> **`POST /roll` 与 `POST /choices` 的本质差异**：`/choices` 是**跨回合异步**——GM 给选项 → 回合结束 → 玩家选 → 作**下一回合**输入。`/roll` 喂的是一次**回合内阻塞**的明骰：GM 调 `resolve_*_open`(明骰)时该工具调用**本身阻塞**(仿 AskUserQuestion)，引擎持久化 `pending_roll` 规格(无结果)并经 §4 `roll_staged` 弹卡 → 玩家点击 `POST /roll` → 引擎**此刻**掷(`commitPendingRoll`，复用 dice/expr 求值、写 verdict event)→ 结果作为**工具返回值在同一 GM 回合内**回给 GM，不跨回合、不经 Stop 物化。前端只**触发 + 播动画**(§4 `roll_committed`)，点数恒由引擎算(anti-F1)。时序详见 玩家闸控明骰设计 §3/§4。裸 CC(无前端可阻塞)时 `resolve_*_open` 当场立即掷、直接返回，降级不卡死。

## 3 + 4. 流式通道（WS / SSE，server → client）

连接：`GET /sessions/{id}/stream`（SSE）或 `…/ws`（WS 升级）。**只推 server→client**（动作走 §2 REST）。**连接后先发 `snapshot` 再发增量**；断线重连＝先 `GET /presentation` + `GET /events?since=` 补齐，再续流（去重用 `seq` / `narrativeCursor`）。

**消息目录**（每条带 `type` + `protocol`）。**「实现状态」列对账后端 `harness/src/dicegm/turnLoop.ts` + `notify.ts`、前端 `frontend/src/features/play/useSession.ts`**：

| `type` | 何时 | payload | 落点（UI） | 实现状态 |
|---|---|---|---|---|
| `turn_started` | 玩家输入被 GM 接手 | `{ "turnId" }` | 进入"生成中" | 🚧 后端发、**前端丢弃**（注④） |
| `narration_delta` | GM `narrate` 流出 token | `{ "turnId", "text" }` | 叙述区（打字机，流①） | ❌ 未实现（注⑤） |
| `narration_commit` | 一条 narrate 落 event | `{ "seq", "text" }` | 叙述区定稿 + 推进 `narrativeCursor` | ⚠️ 收发通，但 **`seq` 语义债**（注⑥） |
| `presentation_delta` | 后端引擎规范态变更（**缝 A `onCanonWrite` 驱动**，§5） | `{ "seq", "changes": { "sheets?", "mechanics?", "reveal?", "watcherFired?" } }` | 呈现台增量 + 机械回显（流②） | ✅（前端收到即 `GET /presentation` 全量对账，注⑦） |
| `choices` | Stop hook 物化待选项 | §1 `choices` 形状 | 待选项按钮 | ✅（前端收到走 refetch，未直接用 payload） |
| `roll_staged` | GM 调 `resolve_*_open`、引擎暂存 `pending_roll`（待掷） | `{ "pendingRoll" }`（§1 `pendingRoll` 形状，只含规格无结果） | 弹 BG3 掷骰卡（亮 DC / 区间 / `exprDisplay`） | ✅ |
| `roll_committed` | 玩家点击触发、引擎掷出 verdict | `{ "eventId", "rolls": [18], "total": 18, "dc?": 15, "outcome": "success" }`（`outcome`: `"success"`/`"fail"` 或 band-label） | 骰子动效 + 成败高亮，清掷骰卡 | ✅ |
| `turn_ended` | 回合自然结束 | `{ "turnId", "seq" }` | 收尾，可接受下一输入 | 🚧 后端发、**前端丢弃**（注④） |
| `game_end` | 终局（`game_end`/`you_death`） | `{ "reason", "outcome" }` | 终局画面 | ❌ 未实现（后端无映射、前端无处理） |
| `error` | 后端 / GM 错误 | `{ "code", "message" }` | 提示 | 🚧 后端发（`gm_error`/`driver_error`）、**前端丢弃**（注④） |

> **实现状态注（流式，联调必读）**：
> - **注④ 前端现处理 9/10 消息**（2026-06-24 核验纠旧）：`useSession.ts` 的 switch 现接 `turn_started`/`narration_commit`/`presentation_delta`/`choices`/`roll_staged`/`roll_committed`/`turn_ended`/`game_end`/`error` 九种；唯 `narration_delta` 不处理（后端也不发，见注⑤）。故「生成中」态、回合收尾、错误提示、终局画面**前端侧均已接**；但 `game_end` 后端从不发（§10.1 B3）→ 终局画面实际仍不触发。
> - **注⑤ `narration_delta` 不发**：`AgentSdkDriver` 把 GM 的一段 `narrate` 作**整块**返回，`turnLoop` 直接发 `narration_commit`，**无 token 级流式**。打字机效果待 Agent SDK 流式接入（[ADR-0020](../05-决策记录-ADR/README.md) 下一阶段「token 级逐字 narration」）。
> - **注⑥ `narration_commit.seq` 语义债**：`turnLoop` 用**回合内 `let seq=0` 自增计数器**填该字段（`turn_ended.seq` 同源），**不是全局 event `seq`**；而 §1 快照的 `narrativeCursor` 期望全局 event seq。两者对账口径不一致——重连去重逻辑目前不可靠。应统一为「落 event 的全局 `seq`」。
> - **注⑦ 增量退化为全量对账**：前端收到 `presentation_delta` / `choices` 一律 `GET /presentation` 重取全量（`useSession.ts` 的「Phase 1 对账」），**不解析 `changes` 增量**。功能正确但每次变更一个 HTTP 往返；增量解析是后续优化。

## 5. 变更通知缝（缝 A：MCP → 后端）

[组件7 §6](玩家客户端.md) 的缝 A：dicelore MCP（组件2）写规范态成功后，把「改了什么」告知编排后端，后端映射成 §4 `presentation_delta` / `roll_committed` 推前端（流②）。**有两种形态**——v1 因 MCP 与后端同进程（[ADR-0020](../05-决策记录-ADR/README.md) in-process 默认）走 **5.1 进程内回调**；**5.2 HTTP webhook** 是跨进程 / 远程 / 自定义 MCP 的未来形态，二者**语义等价、只是传输不同**。

| 形态 | 传输 | 适用 | v1 状态 |
|---|---|---|---|
| **5.1 进程内 `onCanonWrite` 回调** | 函数调用（零 HTTP） | MCP in-process 挂载（同进程） | ✅ **已实现、v1 默认** |
| **5.2 HTTP webhook** | `POST {NOTIFY_URL}` | MCP 跨进程 / 远程 / 自定义 MCP host | ❌ 未实现（契约已备，留未来） |

### 5.1 进程内回调（v1 默认，已实现）

- **机制**：后端引擎 `createMcpServer(db, deps)` 接受 `deps.onCanonWrite`；工具写规范态成功后**在 handler 外层**回调 `onCanonWrite(evt: CanonWriteEvent)`（不进 `runTool`、引擎零改动，[ADR-0020](../05-决策记录-ADR/README.md) ②）。`evt` 带 `toolName` / `seq` / `output`。
- **映射**：`SessionHost.onCanonWrite` → `mapCanonWrite(evt)`（`harness/src/dicegm/notify.ts`）→ `WsHub.broadcast`：
  - `resolve_outcome_open` / `resolve_contest_open` 出参 → **`roll_committed`**；
  - 其余规范态写 → **`presentation_delta`**（前端收到 `GET /presentation` 全量对账，注⑦）。
- **现实粒度（实现 gap）**：`mapCanonWrite` 当前把「非明骰」的所有写**一律塌成 `presentation_delta`**，**未**按 5.2 的 `kind`（mutation / visibility / reveal / watcher_fired / choice_staged / game_end / bulk）细分，也**未**单独发 `game_end`。靠前端全量对账兜住正确性，但 `game_end` 等需要专门 UI 的信号丢失（见注④/§3+4）。
- **无鉴权 / 无序列化**：同进程直传对象，缝 A 不涉及 token（§0）。

### 5.2 HTTP webhook（跨进程未来形态）

> 用于 MCP 作独立进程 / 远程部署 / 自定义 MCP host —— 此时无法进程内回调，改走 HTTP。**契约（payload）已在 `packages/shared/src/notify.ts` 备好（`NotifyPayload` / `NotifyKind`），但后端尚无 `/internal/notify` 接收端点、agent 工具面尚无出站 HTTP 客户端**。

- **端点（后端侧）**：`POST {DICELORE_NOTIFY_URL}`（如 `http://127.0.0.1:{port}/internal/notify`）。
- **鉴权**：头 `X-Dicelore-Notify-Token: {token}`——后端启动时生成、经 env 传给 MCP、MCP 回带；Web 多会话下 token 即会话路由键。
- **payload**（＝ `packages/shared` 的 `NotifyPayload`）：

```jsonc
{
  "protocol": "dicelore.notify/1",
  "sessionId": "…",
  "seq": 1235,                  // 本次写后的 store seq
  "kind": "mutation",           // mutation | event | visibility | reveal | watcher_fired | choice_staged | game_end | bulk
  "delta": { /* 与 §4 presentation_delta.changes 同构的局部；bulk 时为空，提示后端回读全量快照 */ }
}
```

- **`kind` 与触发工具**：`mutation`←`sheet_update`；`event`←`event_append`(可见)；`visibility`←`sheet_show`/`world_show`；`reveal`←`reveal_once`；`watcher_fired`←watcher 就地触发；`choice_staged`←`resolve_choice`（暂存，**不**含后果给前端，仅提示"有暂存"）；`game_end`←`game_end`；`bulk`←`restore`/`branch`（整表，后端回读 `GET /presentation`）。这套细分粒度是 5.1 进程内回调将来应补齐的目标。
- **不发**：自定义 MCP（out-of-canon，[组件7 §8](玩家客户端.md)）。
- **响应**：`204`；后端失败不应阻塞 MCP 主流程（fire-and-forget + 后端自有重连补齐兜底）。

> **契约层归属**：`packages/shared/src/notify.ts`（`NotifyPayload`/`NotifyKind`）**只服务 5.2 webhook 形态**；5.1 进程内路径用后端引擎的 `CanonWriteEvent` → `StreamMessage`（`mapCanonWrite`），**不经 `NotifyPayload`**。故 `notify.ts` 契约目前**无运行时消费者**（备而未用），勿误以为它在缝 A 现役。

## 8. 已厘清记录

### 8.1 掷骰式裁决 / 区间裁决 vs choice 物化的关系（**已回填**）

> 原 [视觉页 §8](玩家客户端-视觉.md) 与本页曾留「掷骰式裁决 UI（区间裁决）与现有 `choices` 物化的关系**待厘清**」。**已由 玩家闸控明骰设计 厘清并落本页 §1/§2/§4**：

- **明骰 = 玩家闸控掷，choice = 玩家选**——两者**并列于「玩家面向交互式 resolver」族**：一个玩家提供「我掷了」(值由引擎出)、一个玩家提供「选了哪个」。**互不物化为对方**。
- **机制各异**：
  - **choice**：`resolve_choice` 经 Stop hook 物化 `kind=choice` event → §1 `choices` 投影 → `POST /choices` 作**下一回合**输入（**跨回合异步**）。
  - **明骰**：`resolve_*_open`(`outcome`/`contest` 两形)是**回合内阻塞**调用 → §1 `pendingRoll` 投影 + §4 `roll_staged` 弹卡 → `POST /roll` 触发、引擎掷 → §4 `roll_committed` 回显 → 结果作**工具返回值同回合**回 GM，**不经 Stop 物化**。
- **anti-F1 边界**：明骰点数恒由引擎在玩家点击时计算；`pendingRoll` 只下发规格(`exprDisplay`)，前端只触发 + 播动画，篡改伪造不了真值。
- 与之对照的暗骰(`resolve_*_hidden`，引擎自动掷)不入玩家面向交互流，照旧经**缝 A `onCanonWrite`** → `presentation_delta` 回显。

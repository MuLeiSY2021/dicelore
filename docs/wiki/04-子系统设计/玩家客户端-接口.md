# 组件7「玩家客户端」接口文档

> **本页职责**：玩家客户端**编排后端**对外的**规范接口契约**——① 前端 ↔ 后端的 **REST**（动作进 / 呈现快照出）+ **流式 WS/SSE**（叙述 / 呈现增量 / 回合事件）；② **MCP → 后端 notify webhook**（变更通知缝的 payload）。**这是 devs 照之实现、`packages/shared` 类型镜像、双壳与自定义 MCP 对接的那份合同。**
> **上游依赖**：[玩家客户端（组件7 设计）](玩家客户端.md)（边界 / 数据流 / 通知缝裁定）、[adapter §7 呈现模型生成器](adapter与L3审计.md)、[内层能力库](内层能力库.md)（四域 / 可见性 / `seq`）、[MCP工具面](MCP工具面.md)（出参形状）。
> **状态**：🟢 已定 v1 契约骨架（2026-06-20）；**确切字段名以 `packages/shared` 实现定稿为准**，本页定形状与语义。

---

## 0. 通用约定

- **传输**：前端 ↔ 后端走 **HTTP（REST）+ WS 或 SSE（流式）**；两种分发壳（Tauri localhost / Web 网络）**同一套**，不用 Tauri 私有 IPC（[组件7 §7](玩家客户端.md)）。
- **协议版本**：每条消息 / 响应带 `protocol: "dicelore.client/1"`（或 HTTP 头 `X-Dicelore-Protocol: 1`）。破坏性变更进位（§6）。
- **会话标识**：所有端点按 `sessionId` 寻址；v1 单会话亦显式带（为 Web 多会话铺路）。
- **鉴权**：Tauri ＝ 仅环回（loopback）、无网络暴露；Web ＝ REST/WS 带会话 token（细节属 [场景 B](../01-业务分析/用户与场景.md) / 未来）。**notify webhook 用每会话 token**（§5）。
- **类型源**：本页 JSON 形状由 `packages/shared` 导出为 TS 类型，前端 / 后端共用、免重复定义。
- **职责切分**：**动作走 REST**（请求/响应清晰、易鉴权）；**GM 产物走流式**（叙述逐字、回合事件）；**core 变更走 webhook → 后端 → 转流式**。

## 1. 呈现模型数据形状（流②核心，REST 快照与 WS 增量共用）

承接 [adapter §7 呈现模型生成器](adapter与L3审计.md)：读 store / event、**按 `visible` 过滤**后的玩家可见投影。

**全量快照**（`GET …/presentation` 与 WS 重连补齐用）：

```jsonc
{
  "protocol": "dicelore.client/1",
  "sessionId": "…",
  "seq": 1234,                       // 本快照反映到的 store seq（对账基准）
  "sheets": [                        // 状态面板：可见 cell，按 entity 分组
    { "entity": "张三",
      "cells": [ { "attr": "HP", "value": "12", "visible": 1 },
                 { "attr": "金钱", "value": "77", "visible": 1 } ] }
  ],
  "mechanics": [                     // 机械回显：可见的 verdict/mutation/watcher_fired
    { "seq": 1230, "kind": "mutation", "text": "金钱 +3d100=74 → 77",
      "data": { "张三.金钱": { "op": "rolled", "from": 3, "expr": "+3d100=74", "to": 77 } } }
  ],
  "choices": {                       // 待选项：最新已物化 kind=choice；无则 null
    "eventId": 1234,
    "options": [ { "index": 0, "label": "推门进去", "consequence": "惊动守卫" },
                 { "index": 1, "label": "绕到后窗", "consequence": "耗时但隐蔽" } ]
  },
  "pendingRoll": {                   // 待掷（明骰闸控）：最新 status=awaiting 的 pending_roll 规格；无则 null
    "eventId": 1234,
    "shape": "contest",              // "outcome"（区间 label）| "contest"（胜负 verdict）
    "label": "压价说服",
    "yourSide": { "name": "说服", "exprDisplay": "1d20+{说服}" },  // exprDisplay 仅供展示，真值不下发
    "dc": 15,                        // shape=contest 时亮 DC；可选
    "bands": [ { "label": "失败", "min": 1, "max": 9 },
               { "label": "成功", "min": 10, "max": 20 } ]         // shape=outcome 时给区间预览；可选
  },
  "narrativeCursor": 1228            // 已渲染到的 narrate event seq（重连去重用）
}
```

- `visible` 取值沿用 [内层 §4.1](内层能力库.md)：`1` 可见（投影只含可见 cell；`0`/`2` 不出现）。
- `mechanics` 仅含 `visible=1` 的机械事实；`note` 等 GM 隐事件不入。
- `choices` 由 Stop hook 物化的 `kind=choice` event 投影（[组件7 §5](玩家客户端.md)）。
- `pendingRoll` 由 core `pending_roll` 槽(status=`awaiting`)投影([玩家闸控明骰设计 §3/§6](../../superpowers/specs/2026-06-21-player-gated-roll-design.md))：**只含规格、无结果**。`exprDisplay` 是展示串(如 `1d20+{说服}`)，**真值/掷出点数恒不下发**(anti-F1：点数恒由 core 在点击时计算)。该卡随 `roll_committed`(§4)清空。

**增量**（webhook 驱动、经 WS 推；只带变化部分）见 §4 `presentation_delta`。

## 2. REST 端点（动作进 / 呈现出）

基址：`/sessions/{sessionId}`。

| 方法 / 路径 | 作用 | 请求体 | 响应 |
|---|---|---|---|
| `POST /sessions/{id}/messages` | 玩家自由文本输入（开一回合） | `{ "text": "…" }` | `202 { "turnId": "…" }`（产物走流式） |
| `POST /sessions/{id}/choices` | 玩家点选待选项 | `{ "eventId": 1234, "optionIndex": 0 }` | `202 { "turnId": "…" }`（作下一回合输入） |
| `POST /sessions/{id}/roll` | 玩家点击触发明骰（闸控掷） | `{ "eventId": 1234 }` | `202 { "turnId": "…" }`（core 此刻掷、回合内返回 GM） |
| `GET /sessions/{id}/presentation` | 取**全量呈现快照**（首屏 / 重连） | — | `200` §1 全量快照 |
| `GET /sessions/{id}/events?since={seq}&visibleOnly=true` | 叙述 / 事件**历史回填**（重连补 narrate） | — | `200 { "events": [ { "seq", "kind", "text", "data" } ] }` |
| `POST /sessions` | 新建会话（v1 最小：选团本 / 存档） | `{ "teamId?": "…", "resume?": "…" }` | `201 { "sessionId": "…" }` |
| `GET /sessions/{id}` | 会话元信息（状态 / 是否终局） | — | `200 { "sessionId", "ended", "title" }` |

> **动作返回 `202`**：后端把输入喂给 GM 后立即返回，GM 的叙述 / 裁决 / 呈现变更**全部走流式**（§4）。前端凭 `turnId` 把流上的事件归到这一回合。
>
> **`POST /roll` 与 `POST /choices` 的本质差异**：`/choices` 是**跨回合异步**——GM 给选项 → 回合结束 → 玩家选 → 作**下一回合**输入。`/roll` 喂的是一次**回合内阻塞**的明骰：GM 调 `resolve_*_open`(明骰)时该工具调用**本身阻塞**(仿 AskUserQuestion)，core 持久化 `pending_roll` 规格(无结果)并经 §4 `roll_staged` 弹卡 → 玩家点击 `POST /roll` → core **此刻**掷(`commitPendingRoll`，复用 dice/expr 求值、写 verdict event)→ 结果作为**工具返回值在同一 GM 回合内**回给 GM，不跨回合、不经 Stop 物化。前端只**触发 + 播动画**(§4 `roll_committed`)，点数恒由 core 算(anti-F1)。时序详见 [玩家闸控明骰设计 §3/§4](../../superpowers/specs/2026-06-21-player-gated-roll-design.md)。裸 CC(无前端可阻塞)时 `resolve_*_open` 当场立即掷、直接返回，降级不卡死。

## 3 + 4. 流式通道（WS / SSE，server → client）

连接：`GET /sessions/{id}/stream`（SSE）或 `…/ws`（WS 升级）。**只推 server→client**（动作走 §2 REST）。**连接后先发 `snapshot` 再发增量**；断线重连＝先 `GET /presentation` + `GET /events?since=` 补齐，再续流（去重用 `seq` / `narrativeCursor`）。

**消息目录**（每条带 `type` + `protocol`）：

| `type` | 何时 | payload | 落点（UI） |
|---|---|---|---|
| `turn_started` | 玩家输入被 GM 接手 | `{ "turnId" }` | 进入"生成中" |
| `narration_delta` | GM `narrate` 流出 token | `{ "turnId", "text" }` | 叙述区（打字机，流①） |
| `narration_commit` | 一条 narrate 落 event | `{ "seq", "text" }` | 叙述区定稿 + 推进 `narrativeCursor` |
| `presentation_delta` | core 规范态变更（webhook 驱动） | `{ "seq", "changes": { "sheets?", "mechanics?", "reveal?", "watcherFired?" } }` | 状态面板增量 + 机械回显（流②） |
| `choices` | Stop hook 物化待选项 | §1 `choices` 形状 | 待选项按钮 |
| `roll_staged` | GM 调 `resolve_*_open`、core 暂存 `pending_roll`（待掷） | `{ "pendingRoll" }`（§1 `pendingRoll` 形状，只含规格无结果） | 弹 BG3 掷骰卡（亮 DC / 区间 / `exprDisplay`） |
| `roll_committed` | 玩家点击触发、core 掷出 verdict | `{ "eventId", "rolls": [18], "total": 18, "dc?": 15, "outcome": "success" }`（`outcome`: `"success"`/`"fail"` 或 band-label） | 骰子动效 + 成败高亮，清掷骰卡 |
| `turn_ended` | 回合自然结束 | `{ "turnId", "seq" }` | 收尾，可接受下一输入 |
| `game_end` | 终局（`game_end`/`you_death`） | `{ "reason", "outcome" }` | 终局画面 |
| `error` | 后端 / GM 错误 | `{ "code", "message" }` | 提示 |

`presentation_delta.changes` 子形（与 §1 同字段、`op` 标增删改）：

```jsonc
{
  "seq": 1235,
  "changes": {
    "sheets": [ { "entity": "张三", "attr": "HP", "value": "10", "visible": 1, "op": "upsert" } ],
    "mechanics": [ { "seq": 1235, "kind": "mutation", "text": "HP -2d6=4 → 10" } ],
    "reveal": [ { "seq": 1236, "target": "world:黑风寨布防", "text": "…" } ],
    "watcherFired": [ { "seq": 1237, "watcherId": 7, "payload": "钟敲十二响…" } ]
  }
}
```

## 5. MCP → 后端 notify webhook（变更通知缝）

[组件7 §6](玩家客户端.md) 的缝：dicelore MCP（组件2）写规范态后，向**配置的后端 notify-URL** POST 一条增量；后端转成 §4 `presentation_delta` / `choices` 推前端。**URL 未配 ＝ no-op**（Claude Code 路径不变）。

- **端点（后端侧）**：`POST {DICELORE_NOTIFY_URL}`（如 `http://127.0.0.1:{port}/internal/notify`）。
- **鉴权**：头 `X-Dicelore-Notify-Token: {token}`——后端启动时生成、经 env 传给 MCP、MCP 回带；Web 多会话下 token 即会话路由键。
- **payload**：

```jsonc
{
  "protocol": "dicelore.notify/1",
  "sessionId": "…",
  "seq": 1235,                  // 本次写后的 store seq
  "kind": "mutation",           // mutation | event | visibility | reveal | watcher_fired | choice_staged | game_end | bulk
  "delta": { /* 与 §4 presentation_delta.changes 同构的局部；bulk 时为空，提示后端回读全量快照 */ }
}
```

- **`kind` 与触发工具**：`mutation`←`sheet_update`；`event`←`event_append`(可见)；`visibility`←`sheet_show`/`world_show`；`reveal`←`reveal_once`；`watcher_fired`←watcher 就地触发；`choice_staged`←`resolve_choice`（暂存，**不**含后果给前端，仅提示"有暂存"）；`game_end`←`game_end`；`bulk`←`restore`/`branch`（整表，后端回读 `GET /presentation`）。
- **不发**：自定义 MCP（out-of-canon，[组件7 §8](玩家客户端.md)）。
- **响应**：`204`；后端失败不应阻塞 MCP 主流程（fire-and-forget + 后端自有重连补齐兜底）。

## 8. 已厘清记录

### 8.1 掷骰式裁决 / 区间裁决 vs choice 物化的关系（**已回填**）

> 原 [视觉页 §8](玩家客户端-视觉.md) 与本页曾留「掷骰式裁决 UI（区间裁决）与现有 `choices` 物化的关系**待厘清**」。**已由 [玩家闸控明骰设计](../../superpowers/specs/2026-06-21-player-gated-roll-design.md) 厘清并落本页 §1/§2/§4**：

- **明骰 = 玩家闸控掷，choice = 玩家选**——两者**并列于「玩家面向交互式 resolver」族**：一个玩家提供「我掷了」(值由 core 出)、一个玩家提供「选了哪个」。**互不物化为对方**。
- **机制各异**：
  - **choice**：`resolve_choice` 经 Stop hook 物化 `kind=choice` event → §1 `choices` 投影 → `POST /choices` 作**下一回合**输入（**跨回合异步**）。
  - **明骰**：`resolve_*_open`(`outcome`/`contest` 两形)是**回合内阻塞**调用 → §1 `pendingRoll` 投影 + §4 `roll_staged` 弹卡 → `POST /roll` 触发、core 掷 → §4 `roll_committed` 回显 → 结果作**工具返回值同回合**回 GM，**不经 Stop 物化**。
- **anti-F1 边界**：明骰点数恒由 core 在玩家点击时计算；`pendingRoll` 只下发规格(`exprDisplay`)，前端只触发 + 播动画，篡改伪造不了真值。
- 与之对照的暗骰(`resolve_*_hidden`，引擎自动掷)不入玩家面向交互流，照旧经 webhook → `presentation_delta` 回显。

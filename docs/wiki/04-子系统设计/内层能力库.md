## 1. 内层 = 三个可单测原子层

承接 [总体架构 §2](../03-架构/总体架构.md)"内层＝原子、外层＝有意识组合"。内层不是一坨，而是**三层原子**，依赖单向向下，全部脱 MCP 可单测：

```
③ 求值与裁决编排   evalExpr / applyMutations / 对抗比大小      （吃 ①②）
② 四域 store CRUD   sheet/event/world/rule 读写 + FTS 召回      （吃 SQLite）
① 骰子引擎          rollDice / rangeMap（纯函数、RNG 注入）       （无依赖）
```

- **① 骰子引擎**：纯函数、无 store、RNG 注入——唯一"不可信、不能交给 AI"的随机源（anti-F1 的物理根）。
- **② store CRUD**：四业务域的精确读写 + FTS 召回，吃 `better-sqlite3`。
- **③ 求值 / 裁决编排**：把 ①② 组合成"取真值 → 掷 → 算 → 写回"等可单测单元；MCP 工具只是它的**薄包装**（[§4.2 状态骰下沉](../03-架构/总体架构.md) 说的"求值住内层"就落在这层，喂内存 store 即可单测）。

> **谁是 resolver？** "给选项 / 三种掷骰 / 卡池抽" 统一为 **resolver** 的概念属**行动层**（→ [MCP工具面](MCP工具面.md)；resolver 升格见 [03 TODO C](../03-架构/TODO.md)）。本页只造它脚下的原语：随机（①）、取数写回（③）、内容抽样（②）。

## 2. 骰子引擎（原子层①）

骰子引擎用 TS 实现，**只留两个真原子 + RNG 注入**：

```ts
rollDice(count: number, sides: number, rng?: Rng): number[]   // 校验 count≥1、sides≥2
rangeMap(value: number, bands: Band[]): Band                  // Band = {label, min, max, ...}
```

- **RNG 注入**：`rng` 默认系统随机，测试喂定种子 → 掷骰逻辑**确定性单测**（"可单测"的着落点）。
- `rangeMap`：把值映到档位（`resolve_outcome` 用），校验区间不重叠 / 全覆盖，落空 → 报错。**暴击不是引擎逻辑**——它是档位表里定的一档（如 `{label:"完美", min:96, max:100}`）。档位表即团本表达 **PbtA 三档结果**（完全 / 部分 / 失败）的载体（[ADR-0016](../05-决策记录-ADR/README.md)、[MCP §1.2](MCP工具面.md)）。
- **砍掉旧引擎三件**（v1 无对应工具、可随时再加）：`dice_judge`（DC 折进对抗骰，见 §3.1）、`dice_multi`（批量上移到 §3.2）、独立 `dice_contest`（＝求值器跑两边 + 比大小）。
- **多项求和**（`3d6+1d4+5`）**不在引擎**——引擎只掷单项 `NdS`，多项交 §3 求值器拆项组合。

## 3. 表达式求值器与 mutation（原子层③）

### 3.1 `expr` 值表达式文法

`resolve_contest` 的边、`sheet_update` 的算术，统一用一种串——**`expr`**（弃 `operand` 旧名）。表示法 ＝ **字符串 DSL + `{}` 界定引用**：

```
expr  = term (('+' | '-') term)*           # 只 ±、左到右、无 * / 括号
term  = dice | int | ref
dice  = \d+ [dD] \d+                        # 1d20、6d100
int   = \d+                                 # 500、60
ref   = '{' 实体 '.' 属性 '}'               # {张三.力量}
```

例：`"1d20 + {张三.力量}"`、`"500 + 6d100"`、`"60"`、`"{张三.攻击} + 1d6 - 2"`。

- **为何 `{}` 界定引用**：骰子 / 常数那半 AI 极熟、当字符串最省 token；唯一的解析风险在**引用**（CJK 名、点号、名字带空格）——`{}` 让"哪段是引用"零歧义，无需约束名字字符集。不出错的那半不强加结构。schema 形态取舍见 [MCP工具面](MCP工具面.md)。
- **求值住内层**：MCP 只透传 `expr` 串、**不解析**；求值器拆项 → 骰子项调 ①、引用项查 ② 的 sheet（`parseInt`，非数值报错）、整数项直取 → 按 `±` 求和。
- 返回**账本**：`{ total, terms: [{kind, raw, rolls?, refValue?, value}] }`——喂 L3 比对。

> **谓词 expr（watcher 条件，[ADR-0013](../05-决策记录-ADR/README.md)）**：在值表达式之上加一层**比较**——`pred = expr 比较算符 expr`，比较算符 `< <= > >= == !=`，求值出 **bool**（如 `{张三.HP} < 30`、`{世界.天} >= 18`）。**只加比较、不加 `* /`**：百分比阈值 / 相对时间由 AI 创建 watcher 时读真值算成绝对值填入（GM 全见可 `sheet_get`），文法保持极简。求值器对两边各跑值表达式求和、再比大小 → bool；非数值边报错。`watcher` 表存的就是这种谓词串，每次 `sheet_update` 后就地重算（见 §4.2）。

### 3.2 `applyMutations`：sheet 批量写（状态骰下沉于此）

承接 [§4.2 状态骰下沉](../03-架构/总体架构.md)：一次 entity 作用域的批量写，整批**一个事务（原子）**。每项 `{attr, op, expr}`，按 **op × expr 类型**派发：

| op ＼ expr | 值表达式（§3.1） | 词条裸名 |
|---|---|---|
| `+` `-` | **标量算术**：读 `attr`→数 ± evalExpr（掷骰在 ①）→ 写回 | **集合增减**：cell `attr:词条` ±N（`小刀*3`→ +3；减到 0 删） |
| `=` | **绝对赋数**：写 `attr =` evalExpr（带骰＝先掷） | **赋文本**：写 `attr =` 词条（中毒 / 活着） |

- **LHS `attr` 裸名 → 归批次 entity**；**RHS 引用强制 `{实体.属性}`**（左侧本实体、右侧显式他者）。
- **账本返回**每项标 `rolled | set`：`{attr, op, expr, kind, old, rolls?, delta?, new}`——L3 据此抓"该掷处却用了 `=` / 裸 set"。
- **引擎哑**：不 clamp（能否负债是 rule/GM 判断，不焊进引擎）；非数值算术 → 报错 + **整批回滚**。

### 3.3 `commitPendingRoll`：明骰落定（玩家点击时掷于此）

承接 玩家闸控明骰设计 §3：明骰 `resolve_*_open` 把「掷骰**动作**」交还玩家、把规格暂存进 [§4.2 `pending_roll` 槽](#42-event-事件域内两表日志--watcher)（无结果），**点数恒由引擎在玩家点击的此刻才算**（anti-F1）。这一"读规格 → 掷 → 写 verdict"的落定就是本层一个可单测单元：

```ts
commitPendingRoll(db, eventId: number, rng?: Rng): RollResult
  // RollResult = { eventId, shape, rolls:number[], total:number, dc?:number, outcome:string }
```

- **语义**：① 读 `pending_roll(event_id=eventId)` 的规格 → ② 复用 [§3.1 求值器](#31-expr-值表达式文法) 对 `yourSide.expr` 求值（骰子项落 [§2 骰子引擎](#2-骰子引擎原子层)、引用项查 sheet）；`shape=contest` 比 `dc`/对边得胜负、`shape=outcome` 经 `rangeMap` 落 `bands` 档 → ③ append 一条 **`kind=verdict`、`visible=1`** event（含 DC、点数 rolls/total、成败 outcome）→ ④ 槽 `status: awaiting → committed` → ⑤ **返回 `RollResult`**（明骰 happy path 回合内即作 `resolve_*_open` 工具返回值回给 GM，**不经 Stop 物化**，与 `pending_choice` 的回合末物化相反）。
- **纯函数 + RNG 注入可单测**：`rng` 同 [§2](#2-骰子引擎原子层) 默认系统随机、测试喂定种子 → 明骰落定逻辑**确定性单测**（喂内存 store + 预置 `pending_roll` 行即可，脱 MCP / 脱后端编排）。
- **幂等（宕机恢复要害）**：进入时若槽已 `status=committed`，**不重掷**——直接由已落的 `kind=verdict` event 重建并返回 `RollResult`。这是 设计 §5 双持久化恢复路的正确性基石（后端崩溃重启重入、或 `POST /roll` 重投，都不致二次掷骰）；对齐 [§4.5.5 铁律](#455-铁律接口化表述)「一切游戏态变更必经 store」——verdict event 是真相源，重入据它复现而非重摇。
- **账本**：verdict 的 `data_json` 同样带 [§3.1](#31-expr-值表达式文法) 求值账本（`terms`），供 L3 比对"点数确由引擎算、非 AI/客户端给"。
- **`awaitPlayerRoll(eventId)` 是后端（组件7）注入的接缝（本层只定接口、不实现）**：本层只造 `pending_roll` 槽 + `commitPendingRoll` 这对可单测原语；明骰阻塞调用怎么 `await` 玩家点击、WS 桥接、回合内返回——是 设计 §3/§4 由后端（组件7 线）注入的能力，本层不实现阻塞/WS。**能力缺失（裸 CC，无前端可阻塞）→ handler 当场立即 `commitPendingRoll` 掷、直接返回**（降级，不卡死）；两路都在回合内返回结果，差别仅"是否经人机往返"。

## 4. 四业务域 store schema（原子层②）

每局一个 `.db`，四域共处。承接 [02 §2](../02-领域模型/核心概念.md)、[总体架构 §3](../03-架构/总体架构.md)。

### 4.1 sheet 实体卡

```sql
sheet(entity TEXT, attr TEXT, value TEXT, visible INTEGER DEFAULT 0, PRIMARY KEY(entity, attr))
  -- visible: 0=默认隐藏 / 1=已 show 可见 / 2=强制隐藏（暗值，entity 级 show 也不揭）
```

- **每 cell 一个标量**，`value` 存文本、取数时 `parseInt`（非数值做算术 → 报错，防 AI 拿"活着"加减）。
- `entity` 是泛化的"名"：角色 / NPC / 派系 / **地点 / 物品实例 / 世界**（游戏钟也在这，见 §4.2）。
- **集合 / 关系靠 `attr` 的 `前缀:键` 约定**——不另开表：
  - 库存 `(张三, 库存:药水, 3)`；声望 `(玩家, 声望:黑风寨, -20)`；物品实例 `(铁剑#1, 耐久, 50)` `(铁剑#1, 持有者, 张三)`。
- 两个读工具（→ [MCP工具面](MCP工具面.md)）：`sheet_get`（单 cell）/ `sheet_list`（前缀扫：`张三.` 取整卡、`张三.库存:` 取整库存）。
- **可见性（承接 [02 §2 要点5](../02-领域模型/核心概念.md)、[03 §3.1](../03-架构/总体架构.md)）**：`visible` 列三态，默认全 0（deny-by-default）。`sheet_show` attr 级把指定 cell 置 1；**entity 级递归 show** 是长效策略（覆盖该 entity 未来新增 attr），存为该 entity 一条策略 cell（如 `(张三, __show_all, 1)`）。**输出层渲染判定**：cell 可见 ⟺ `visible=1` ∨（entity 有 `__show_all` ∧ `visible≠2`）。暗值用 `visible=2` 焊死，entity-show 也不揭——防泄漏。`reveal_once` 不动本列（走 event，见 §4.2）。

### 4.2 event 事件（域内两表：日志 + watcher）

```sql
-- 已发生：统一 append 日志（FTS5 虚表，rowid 即单调 seq）
event(content, kind, data_json, tags, visible, game_time, created_at)   -- 索引见 §5
  -- kind ∈ {narrate, verdict, mutation, note, watcher_fired, reveal, choice}
  -- visible 默认按 kind：narrate/verdict/mutation/watcher_fired/reveal/choice=1、note=0；event_append 可显式覆盖
-- 待发生：单独表（watcher = sheet 数据触发器，泛化并取代旧 timer，[ADR-0013]）
watcher(id, created_seq, condition, payload, mode, armed, last_fired_seq, status)
  -- condition: §3.1 谓词 expr 串（"{张三.HP} < 30"）；mode ∈ {once, repeat}
  -- armed: 1=可触发 / 0=已触发待解除（edge 去抖）；status ∈ {active, fired, cancelled}
```

- **统一日志**：散文（narrate / note）进 `content` 走 FTS；结构（verdict / mutation）进 `data_json`。一条时间线，L3"取本轮的 event"即可比对裁决 vs 叙述。
- **可见性（`visible` 列）**：默认按 `kind` 定——给玩家的散文与要回显的机械事实（narrate / verdict / mutation / watcher_fired / reveal / choice）默认 1、GM 注记（note）默认 0；`event_append` 可显式覆盖。输出层（[03 §6](../03-架构/总体架构.md) 流②）按此过滤渲染。
- **`reveal_once` 快照披露 ＝ append 一条 `kind=reveal` 的可见 event**（承接 [03 §3.1](../03-架构/总体架构.md)）：内容 = 某隐藏**目标**此刻的冻结副本——sheet cell `{entity.attr, value@seq}` **或 world 条目**（doc/pool 行）。**不碰目标自身的 `visible`**（底层仍隐）；每次 reveal_once 落新行、旧行留作历史快照。对 sheet（值暗变）→"玩家见旧值、下次 reveal_once 才刷新"自动成立；对（基本静态的）world → 一次性披露、不入持久可见集。
- **可见性变更审计**：`sheet_show` / `world_show` 各自写一条 **`kind=note`、`visible=0`（对玩家隐）的审计 event**（"第 N seq 揭示了 X"），供 L3 / 回看，玩家不可见。
- **时间观（关键）**：框架**只拥有单调 `seq`**（通用排序）。**"游戏时间 / 回合"不是框架概念——是团本定义的 sheet 钟**（如 `(世界, 回合, 12)`、`(世界, 时间, "Day3 14:00")`），由团本 rule 推进、AI 用 `sheet_update` 改。`narrate` 只是日志条目，**不推进游戏时间**。`event.game_time` 是写入时对 sheet 钟拍的可选快照（仅供回看，框架不解释）。
- **watcher（sheet 数据触发器，[ADR-0013](../05-决策记录-ADR/README.md)）**：`condition` ＝ §3.1 的**谓词 expr**（`{张三.HP} < 30`、`{世界.天} >= 18`，时间到期只是监视钟 attr 的特例）。**就地触发、非 hook 轮询**：每次 `sheet_update` 写完，引擎重算本次 entity 上挂的 watcher——`armed ∧ condition` → 触发（落 `event(kind=watcher_fired)`、payload 经 `sheet_update` 出参当轮回 AI；`armed→0`；`once` 则 `status→fired`）；`¬armed ∧ ¬condition ∧ repeat` → re-arm（`armed→1`，**edge-triggered 去抖**：跨越沿才触发、须先解除才能再触发）。**v1 无显式 cooldown**。创建 ＝ `watcher_set`（AI 用工具）或**团本 Front 预声明**（[ADR-0016](../05-决策记录-ADR/README.md) 提前到 v1：团本 `fronts/*.md` 的凶兆阶梯 import 成预声明 watcher，见 [团本与 manifest §6](团本与manifest.md)）。每次 `sheet_update` 只重算本批 entity 上挂的 watcher（池子小、成本低）。
- **pending_choice 暂存槽（`resolve_choice` → 回合末物化的接缝，[adapter](adapter与L3审计.md)）**：`resolve_choice` 不立即落 event，而是写一个 **session 级单行槽**（`pending_choice(seq_staged, prompt, options_json, status)`，轮内反复调用末次覆盖）；回合末 **Stop hook** 读它物化——落一条 `event(kind=choice、visible=1)`（选项 + 已锁后果，seq 早于玩家 pick → L3 验"后果在先"），槽 `status→materialized`。非终局轮该槽为空 ＝ L3 档 A 违规（[ADR-0014](../05-决策记录-ADR/README.md)）。**跨进程**：槽落 SQLite（非 MCP 内存），独立的 Stop hook 进程才读得到。
- **pending_roll 暂存槽（明骰 `resolve_*_open` → 玩家点击时掷的接缝，玩家闸控明骰设计 §3/§5）**：仿 `pending_choice` 的**单行槽**，但**按 eventId 寻址、且只存规格不存结果**——明骰把"掷骰这个动作"交还玩家、点数仍恒在引擎（anti-F1，见 [§2 骰子引擎](#2-骰子引擎原子层) / 本页下 §3.3 / 设计 §9）。schema：
  ```sql
  pending_roll(event_id INTEGER PRIMARY KEY AUTOINCREMENT, shape TEXT, spec_json TEXT, status TEXT, verdict_seq INTEGER)
    -- event_id: 自增句柄（stage 时分配，= 契约 pendingRoll.eventId / POST /roll {eventId}）
    -- shape ∈ {outcome, contest}（对应 resolve_outcome_open / resolve_contest_open）
    -- spec_json: 规格快照 { context, die?/bands?（outcome）| a?/b?（contest）}——【无结果】（rolls/total/outcome 一律不在此槽）
    -- status: awaiting（已暂存、待玩家点击）→ committed（已掷、verdict 已落）
    -- verdict_seq: commit 后链接已落 kind=verdict event 的 seq（幂等重建据它复现，不重掷）
  ```
  - **与 pending_choice 并列对照**：两者都是「把能动性这一拍交还玩家」的接缝、都落 SQLite（跨进程可读）；区别见下表。

  | | `pending_choice` | `pending_roll` |
  |---|---|---|
  | 来源工具 | `resolve_choice`（玩家选 label） | `resolve_*_open`（玩家闸控掷） |
  | 寻址 | session 级单行（轮内末次覆盖） | **按 `event_id`**（一回合可多笔明骰各占一行） |
  | 存什么 | prompt + options + 已锁后果 | 规格 `shape`/`yourSide.expr`/`dc`/`bands`，**无结果** |
  | 物化时机 | **回合末 Stop hook** 读它落 `kind=choice` | **玩家点击时**由 `commitPendingRoll` 落 `kind=verdict`（不碰 Stop hook，见 设计 §3） |
  | status 流转 | `staged → materialized` | `awaiting → committed` |
  | 玩家提供 | 「选了哪个」 | 「我掷了」（值仍由引擎出） |

  - **无结果是要害**：槽里只有规格，点数恒在玩家点击时才由引擎算（anti-F1，§3.3）；`yourSide.expr` 是 [§3.1](#31-expr-值表达式文法) 的 `expr` 串（如 `"1d20 + {张三.说服}"`），下发前端的展示串 `exprDisplay` 由线上契约层从规格派生、真值不下发（设计 §6）。
  - **跨进程 + 宕机恢复**：槽落 SQLite（非 MCP 内存），后端在阻塞 `await` 中崩溃后重启读到 `status=awaiting` 即可重弹掷骰卡补掷（设计 §5）；`pending_roll` 库 + verdict event 是真相源，结果总能补达。`commitPendingRoll` 的**幂等**（§3.3）保证恢复路重入不重掷。
- **L3 分组 ＝ 一个 agent 回合**（R1 重定义）：玩家输入 → 回合自然结束，**回合末 Stop hook** 按 `seq` 圈定本轮全部 event 做审计（叙述 vs verdict/mutation）。**取代旧"两个 narrate 标记之间"的定义**——`narrate` 已升格为轮内可多次的 stream，旧定义塌（[03 §4.1/§6](../03-架构/总体架构.md)）。机械范围、非游戏时间。

### 4.3 world 世界（域内两表：散文 doc + 结构 pool）

```sql
world_doc(name, content, category, tags, visible)        -- FTS5：散文设定（索引见 §5）；visible 默认 0
world_pool(pool, row_json, weight, source, visible)      -- 可抽样结构行；source ∈ {author, ai}；visible 默认 0
```

- **判据**：要随机抽（卡池 / 掉落 / 随机表 / 地图拓扑 / 副本序列）→ 落 `world_pool`（**保结构、不拍平**）；纯散文设定 → `world_doc`。
- `world_sample(pool, n, filter?)`：按列过滤（`json_extract` 取 `稀有度 / 类型`）+ **加权抽样在 TS 求值层**（池子小、载进来按 weight 挑）。`weight` 从 CSV `权重` 列抽、缺省 1。
- `source`：`author`（灌注）/ `ai`（`world_register` 现编）——**这列是"v2 团本 vs 带 AI 内容存档"迁移的钩子**（[总体架构 §3](../03-架构/总体架构.md)）。
- **可见性**：world 全部**默认隐藏**（`visible=0`，作者底料是 GM 知识）；`world_show` 把某条目翻 1（递归 / 部分，[03 §3.1](../03-架构/总体架构.md)）。玩家通过叙述与 show 逐步"发现"世界，而非直接读底料库。
- 结构保真是为未来"agentic 检索 / 表→语言"留路（**检索智能在上层、与存储正交**，本页不锚定、也不 foreclose）。

### 4.4 rule 规则（版本化散文，AI 只读）

```sql
rule_doc(name, content, category, version)       -- FTS5 + 版本号
```

- rule 是**被动、整段读**的约束（"失败硬着陆"这类）——不抽样、不过滤，故结构小表（升级曲线 / 分档）**拍平成 markdown 表嵌进 `content`** 即可（AI 整段读了套用）。
- `version`：作者版本化热更新（[02 §2](../02-领域模型/核心概念.md)）；**AI 不可写**（反讨好红线）。被动召回的触发机制（hook 何时注入哪条）→ [跨agent与适配层](../03-架构/跨agent与适配层.md)。

### 4.5 回合快照与回滚（checkpoint core，跨四域；[ADR-0017](../05-决策记录-ADR/README.md)）

承接 [03 §3.2](../03-架构/总体架构.md)：回滚机制 = **快照 per 回合**，非逆运算、非纯重放。本节定后端引擎侧物理形态——快照表、IoC 参与者注册表、`checkpoint` / `restore` 原语；hook 接线（谁在回合末写、谁在回合开始检测 / 恢复）→ [adapter §3 / §8](adapter与L3审计.md)。**不是第五个业务域**，是骑在四域之上的存储层机制。

#### 4.5.1 快照表（树形 + transcript 锚 + 开放 participant map）

```sql
snapshot(
  id INTEGER PRIMARY KEY,
  parent_id INTEGER,            -- 指向上一回合快照 → 构成树（branch 的骨架）
  transcript_anchor TEXT,       -- 该回合末的 agent 对话定位（CC＝transcript 消息 UUID），auto-sync 的锚
  turn_start_seq INTEGER,       -- 本回合 event 区间起（UserPromptSubmit hook 记，见 §4.2 / adapter §3.2）
  turn_end_seq   INTEGER,       -- 本回合 event 区间止
  blobs_json     TEXT,          -- 开放 map：{ [participantName]: 序列化态 }（§4.5.2，非写死列）
  created_at
)
```

- `session_meta.current_snapshot_id` 记当前在树上的 head 节点。**状态小、默认不剪枝**（保留全部节点即 branch 底物）；真要控增长，只 GC「非任何活跃 head 祖先」的孤立子树。
- **粒度 = 回合边界**（[ADR-0009](../05-决策记录-ADR/README.md) 的 agent 自然回合）。子回合位置的 /rewind 向下取整到最近回合边界快照（玩家对话可能略超前，下一回合 re-align）。

#### 4.5.2 IoC 参与者注册表（解耦：快照不认识各模块）

快照 core **对具体业务域零编译期依赖**——不 import sheet/world/watcher 去序列化，而是各模块**反向注册**一对 capture/restore：

```ts
interface SnapshotParticipant {
  name: string                       // "sheet" / "world.runtime" / "watcher"
  capture(): unknown                 // dump 自己当前态（落进 blobs_json[name]）
  restore(blob: unknown): void       // 从快照覆写自己（整体重写、非逐项逆运算）
}
snapshot.register(participant)       // 模块 init 时注册；core 只拥有注册表 + 表 + 循环
```

- **`checkpoint()`** = 遍历注册表，`{name: p.capture()}` 汇成 `blobs_json`，连同 `parent_id` / `transcript_anchor` / seq 区间落一行、更新 `current_snapshot_id`。
- **`restore(snapshotId)`** = 读该行，对每个 participant 派发 `restore(blobs_json[name])`。**整体覆写**——watcher 表整张重写，故「watcher 级联难逆 / `once` 须手动 re-arm」（当初否逆运算的理由）在此**直接消解**：不逆级联，而是把运行时态快照回去。
- **范围 = 哪些模块注册**（config 化，[ADR-0017](../05-决策记录-ADR/README.md) 细化）：v1 注册 `sheet` / `world.runtime`（`source=ai` 现编部分）/ `watcher` 三个 core participant；**`rule` 不注册** → 自动不随回合回滚（兑现「范围不含 rule、rule 带外」，从硬编码变成一条注册事实）。团本自定义域加进来 = 它注册自己，**不碰快照代码**。这是「玩家客制化其他模块仍能解耦快照系统」的着落点。
- **不过度工程**：v1 就是个轻量内部注册数组（非插件市场）；扩展性是 IoC 白拿的副产品，不为它另搭框架。

#### 4.5.3 event 不是 participant —— 是时间线脊柱

event **不进 `blobs_json`**：它 append-only、全局单调 `seq`、**永不删除**（§4.2 时间观）。快照只对它记 `[turn_start_seq, turn_end_seq]` 区间。

- **当前分支的 event 历史 = 沿 head 祖先链，拼接各节点的 `[start, end]` seq 区间**。因 seq 全局单调不复用，两条从同一父节点分叉的兄弟回合占不同 seq 段，祖先链自然只圈本分支、排除被弃分支——**故 event 无需 `branch_id` 列**，快照树 + 每节点 seq 区间已完整定义当前分支历史。兑现 [ADR-0017](../05-决策记录-ADR/README.md)「event log 留作分支底物」「框架只拥有单调 seq」。
- `event_recall`（FTS，§5）：v1 按当前分支 seq 集过滤，不召回被弃分支事件。L3 审计窗口仍是当前回合 `[start, end]`，不受 branch 影响。

#### 4.5.4 branch / swipe 落地（v1）

- **branch = 树的自然产物**：玩家 /rewind 到回合 N-1 再继续 → 新回合锚定挂在 N-1 子树 → 新快照 `parent_id=N-1`；老分支快照仍在（只是不在 head 祖先链上）。auto-sync 检测与恢复 → [adapter §8](adapter与L3审计.md)。
- **swipe = 默认重掷**：从上一快照分叉、让 AI **重新生成**该回合 → 新工具调用 → 自然新掷骰。沿用同掷需把骰子钉进回合（确定性税），正是 [ADR-0017](../05-决策记录-ADR/README.md) 否掉的；故「不满意→真换结果」与「快照存结果不重放」一致。
- **dice 不外部播种**（§2）：v1 不按 transcript UUID / 外部键播种——结果已落 event 账本（复现无需 seed），且按 UUID 播种会把 transcript 概念灌进 agent 无关的 dice core（破 [ADR-0008](../05-决策记录-ADR/README.md) 对冲）。**反刷骰（同决策点 swipe 几次同骰）= 稳定键播种**记为**未来 config 旋钮**，且键用 core 的 seq/snapshot id、非 UUID。

#### 4.5.5 铁律（接口化表述）

**一个模块「可回滚」当且仅当它注册了忠实的 capture/restore**；逃出 capture 的态回滚不了（[ADR-0017](../05-决策记录-ADR/README.md) 铁律的接口面）。一切游戏态变更必经 store（dice 不在 watcher 重算里偷掷、AI 不在 `narrate` 直接改数）→ 守此，快照即完备。

#### 4.5.6 机制归属（agent 无关 + 对冲自研 agent）

- 快照 core（注册表 / 表 / `checkpoint` / `restore` / 树）**agent 无关**；**只有「对话回退↔快照」的关联检测吃基底专属概念**（CC＝transcript UUID 树），住 [adapter hook](adapter与L3审计.md)。
- **对冲 [ADR-0008](../05-决策记录-ADR/README.md)**（自研 agent 待议）：将来换基底 / 自研 agent，**只重写关联 hook、不动快照 core**。比「人类侧 CLISH 回滚」更精确的对冲表述。

## 5. FTS5 全文索引（@node-rs/jieba 写入分词）

承接 [技术选型 §3](../03-架构/技术选型.md)。坑：FTS5 默认 `unicode61` 把整串连写 CJK 当**一个** token，中文搜不动（旧代码用 `<3 字 LIKE` 兜底绕它）。解法 ＝ **jieba 写入分词 + 影子列**：

- 每个可搜表（`world_doc` / `rule_doc` / `event`）：**indexed 影子列存 jieba 分词后（空格连接）的文本**，原文存 UNINDEXED 列供返回 / 展示。
- 写入时 `@node-rs/jieba` 分词填影子列；查询时把查询词同样过 jieba → `MATCH` 影子列。"门派"这种 2 字词成完整 token，**旧 `<3 字` 坑消失**，并白拿 bm25 排序。
- `@node-rs/jieba` ＝ **预编译 npm 包**（`npm i` 即用、不本地编译、不破坏可分发 / 低安装），词典随包；不联网、仍嵌入式、`.db` 仍单文件随存档走（分词器活在 server 代码里、查询时在场即可）。
- **tag 兜底召回**：`tags` 列在 FTS 命中弱时做 LIKE / exact 并入（技术选型 §3 那句"全文 + tag 兜底"）。
- **零依赖保底**：哪天想甩掉 dep，schema 不动，影子列改 `tokenize='trigram'` 即可（trigram 内置、CJK 子串可搜、2 字以下再 LIKE 兜）。

> `snippet()/highlight()` 会带分词空格——但 v1 `*_search` 是**整段 / 整行返回给 agent 自读**（agent 即引擎），不靠高亮片段，无害。

## 6. session 解析

承接 [总体架构 §3](../03-架构/总体架构.md)：每局一个 `.db`，env 定位"当前局"。

- **一局 ＝ 一个有名字的 `.db`**：默认放**平台 app-data 目录**（跨端：Linux `~/.local/share/` · Mac `~/Library/Application Support/` · Win `%APPDATA%`，用 Node 平台路径约定而非写死 POSIX）下的 `dicelore/sessions/{名字}.db`，`DICELORE_SESSION=修仙团` 定位，`DICELORE_SESSIONS_DIR` 可覆盖根目录。**人类可读名**（非 uuid），契合"本机直接玩到、拷贝 `.db` 即备份 / 分享"；uuid 仅作未起名时的兜底自动名。
- **开库即用、不存在即建**：MCP server（stdio）启动认 env 开库，**首次打开自动初始化四域 schema**。切局 ＝ 换 env 重启子进程（v1 单人单局够）。
- **`session_meta`（KV）**记：`团本 id + 版本号`（迁移据）、`created_at`、`display_name`、**`schema_version`**（框架库结构版本，将来改表 → 开库检测迁移）。
- **建库即灌注团本**：把 manifest 的 world / rule 底料 import 进库——**import 细节归 [团本与 manifest](团本与manifest.md)**，本页只点"建库灌注、meta 记团本 id+版本"。
- **管理入口**：一个瘦人类 CLI（兼 [总体架构 §7](../03-架构/总体架构.md) 要保留的裸骰子调试口）做 `new / list / inspect`；server 只认 env 开库。

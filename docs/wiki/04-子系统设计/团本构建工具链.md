## 0. 设计靶心：降低作者门槛

普通用户不会手写 manifest / CSV / frontmatter。**构建台的存在就是让"丢一整本《凡人修仙传》进去 → 得到一个可玩团本"成立**——把"格式正确性"的负担从人和 AI 移进工具，把"读几百万字"的负担移进检索库，把"看不懂文本格式"的负担移进可视化。

**构建台 = 同一个 Claude Code 换装**：加载构建 skill + 构建期 MCP（`dicelore_build_*`），**而非**运行时那套 `resolve_*` / `sheet_update`。两套工具面互不出现在对方场景。

整体形态对称于运行时（[内层能力库](内层能力库.md)）：

```
        团本包（MD + CSV 文件）  ← 唯一真相（ADR-0015）
                 ▲
        ┌────────┴────────┐
   ① 团本包读写层（纯逻辑·结构化 CRUD over 文件 + 校验）
        └────────┬────────┘
        ┌────────┴────────┐
   ② MCP 门面            ② Web 门面（本地服务 + 可交互前端）
   (agent 走它读写)       (用户在浏览器读写)
        两边都即写即读·共享同一套读写层
                 ▲
   ③ 素材检索库（整本小说切块，按阶段检索喂 agent）
   ④ 构建 skill dicelore-build-pack（分阶段编排 ①②③）
```

## 1. 团本包读写层（纯逻辑核心）

构建台的地基，**与门面正交、可单测**（同 [内层能力库](内层能力库.md) 的分层哲学）。把"对包的结构化操作"封成原子函数，吃进所有格式细节。

- **结构化 CRUD over 文件**：`add_npc` / `update_npc` / `add_pool_entry` / `update_pool_entry` / `write_world_doc` / `set_param_band` / `set_sheet_cell` / `set_manifest` …。底层负责：写对 CSV 列序、生成/更新 frontmatter、放进正确子目录（[团本与 manifest §1](团本与manifest.md) 布局）。**调用者（agent / 用户）永不手搓 CSV 语法或 frontmatter。**
- **文件为真相、即写即读**（[ADR-0015](../05-决策记录-ADR/README.md)）：每次操作即落文件、即可重读；无内存态、无双向同步。
- **内置校验器**（贯穿每次写 + 可整包跑）：
  - CSV 列合法（表头匹配 [§4 列规范](团本与manifest.md)、`weight` 数值、`visible` 合法值）；
  - `manifest.flows` 选的流程 skill **真实存在**（[Skills 包](Skills包.md) 注册表）；
  - `manifest.clock` 指向的 attr 在 sheets / 约定内；
  - **引用完整性**：`entry` 锚点存在、param 表被引用的列齐全、NPC 散文与其 sheet 卡（若有）entity 名一致；
  - 路径/编码：UTF-8、文件名无非法字符。
- **校验产出结构化报告**（`[{level: error|warn, file, msg, hint}]`），供门面回显——错误早发现、定位到具体文件/行。

> 读写层 + 校验器是 TS 纯函数模块，**不依赖 Claude Code、不依赖 Web**；MCP 门面与 Web 门面都只是它的薄包装。

## 2. 双门面（MCP / Web）

同一套读写层，两个调用入口，**对称、即写即读、无实时同步**（[ADR-0015](../05-决策记录-ADR/README.md)）。

### 2.1 MCP 门面（agent 走它）

构建期 MCP server，工具集 `dicelore_build_*`，是读写层 CRUD + 校验 + 检索（§3）的 schema 化封装（Zod，对齐 [MCP工具面 §0 通用约定](MCP工具面.md)）。`dicelore` CLI 在**构建模式**下注册这套 server（而非运行时那套）。agent 在构建 skill（§4）引导下调用它增量产出团本。

### 2.2 Web 门面（用户走它）

**作者侧专属的可交互界面**，本地起一个**轻量 http 服务**（Node 内置 `http`，零重框架，守 [技术选型 §6.1 低安装](../03-架构/技术选型.md)），前端读包目录渲染成人类可读视图（团本说明书：世界设定排版 / NPC 卡片 / 卡池表格 / 分档表），并允许用户**直接增删改条目**。

- 用户每次编辑 → 调读写层 → 即写文件 → 重读渲染（即写即读）。
- agent 改完文件后，用户**刷新**即见最新（v1 无 WebSocket 推送，[ADR-0015](../05-决策记录-ADR/README.md) 否实时同步）。
- **可视化即审阅**：校验报告（§1）在界面上高亮到具体条目，用户看懂、点改或口头反馈让 agent 改。

> **构建期 Web 门面 ≠ 运行时 GUI**：[adapter 页](adapter与L3审计.md) 裁定的"运行时 GUI 属未来"不变——玩游戏 v1 仍走终端。此处只为**作者构建**开一个一等公民界面，与运行时游玩界面解耦（共享文件包、不共享前端）。详见 [ADR-0015](../05-决策记录-ADR/README.md)。

## 3. 素材检索库（吞下整本小说）

承接 [总体架构 §7](../03-架构/总体架构.md)"用宿主 agent 读取抽取"。几百万字的原材料远超 agent 上下文，故**先建检索库、按阶段检索**（[ADR-0015](../05-决策记录-ADR/README.md)）。

- **切块建库**：用户丢的原文（小说 / 同人 / 设定集）→ 切块 → 建检索库。**起步用关键词 FTS5 + jieba 分词**（[技术选型 §3](../03-架构/技术选型.md)，复用运行时基建、零新依赖）；**语义向量检索列未来**（与运行时 RAG spike 同档，[02 §5](../02-领域模型/核心概念.md)）。
- **按阶段检索**：构建 skill 每个阶段（§4）只检索该阶段相关片段——建 NPC 阶段检索人物相关段、建机制阶段检索修炼体系段——**与分阶段节奏天然协同、省 token**。
- **临时品、不进成品包**：检索库是构建期工件，团本包只含成品（[团本与 manifest §1](团本与manifest.md)）。库可随构建会话丢弃或缓存复用，不随团本分发。

## 4. 构建 skill `dicelore-build-pack`（分阶段编排）

承接 [Skills 包](Skills包.md) 的 skill 形态。Moves 式**有序分阶段**流程，编排 ①读写层 ②门面 ③检索库——**只编排已有能力调用序、不新增工具/不碰存储**（同流程 skill 纪律）。

**阶段（边建边审，[ADR-0015](../05-决策记录-ADR/README.md)）**：

1. **世界观 / 设定** → 检索原文设定段 → `write_world_doc` 产设定/门派 → 用户 Web 审阅修正
2. **NPC** → 检索人物段 → `add_npc`（人设散文，关键 NPC 加 sheet 卡）→ 审阅
3. **卡池 / 随机池** → 检索机缘/物品段 → `add_pool_entry` 填池 → 审阅
4. **机制规则** → 检索体系段 → 写 `rules/*.md` + `params/*.csv` 分档 → 审阅
5. **选流程 skill + manifest 收口** → 据团本玩法选 `flows`、声明 `clock` / `entry` → `set_manifest` → **整包校验**（§1）→ 交付

每阶段：`检索 → 调读写层产一块 → 用户在 Web 审阅修正 → 确认进下一阶段`，**阶段间可回退**（前阶段产物在 Web 里随时可改）。校验贯穿全程，收口阶段跑整包校验确保可被运行时 import。

## 7. 工具生成层（声明式视图 + MCP 工具编译引擎）

承接 [ADR-0023 ⑥「构建期声明 → 框架编译」](../05-决策记录-ADR/README.md)：团本作者用**声明**（SQL 片段）描述只读视图与 MCP 工具，框架**编译**成安全的运行时构件——把"团本想暴露什么读 / 写口"从手写工具代码降为声明。

- **`sqlGuard`**：SQL 安全闸（首词分发 + 防注入 / 防多语句）。
- **视图编译（`view`）**：团本声明的只读视图 = 一条 `SELECT` → 编译为 `CREATE VIEW`（投影底层物理表为团本语义视图）。`defineView(db, name, sql)` 是引擎原语。
- **工具编译（`compile`）**：单条 `sql` 声明 →
  - 首词 `SELECT` → **读工具**（`readTool`，只读，可引视图 + 物理表）；
  - 否则（写） → **写工具**（`writeTool`）——**严格正则匹配正典写原语**（`applyMutations` / `*SetStatus` / `*Upsert` 等），**匹配不上即拒**（防团本声明绕过正典通道直写、泄露 / 破坏规范态）。
- 位置：`backend/src/toolgen/`（sqlGuard / view / readTool / writeMatch / writeTool / compile）。
- **编译产物形状**：`compileTool(decl) → { name, desc, handler(db, args) }`（读工具 `handler` 回 `unknown[]`，写工具回 `unknown`）。这是引擎面，**与 MCP `ToolDef`（带 zod `inputSchema`/`outputSchema`/`annotations`）之间有一层适配缺口**——接进运行时工具面须把 `{name,desc,handler}` + `decl.params` 适配成 `ToolDef`（见 [MCP工具面 §7.2](MCP工具面.md)）。

> **实现现状（2026-06-25 核对，单源记此）·引擎成、视图层投影成、框架标准库 ②③ 已接、团本侧待接**：
> - ✅ **引擎**：sqlGuard/view/readTool/writeMatch/writeTool/compile 六模块 + 单测全绿。
> - ✅ **视图层投影**（叙事层 spec §4，[backlog-core 主题A′①](../06-里程碑与问题/backlog-core.md)）：`backend/src/store/views.ts` `initViews(db)` 用 `defineView` 投影 6 命名视图（`player`/`npc`/`world`/`relation`/`clock`/`tension_board`），接进 `initSchema`——toolgen 读工具的基视图就位。视图名+投影列 = 对下游公开的稳定逻辑列契约。
> - ✅ **②③ 接线 + 叙事层 dogfooding**（2026-06-25，[backlog-core 主题A′②③](../06-里程碑与问题/backlog-core.md)）：① 适配层 `backend/src/toolgen/toToolDef.ts` `toolgenToToolDef(decl)` 把 `compileTool` 产物适配成 MCP `ToolDef`（zod schema、出参包 `{result}`）；② 标准库声明 `backend/src/stdlib/narration.ts`（`narrationToolDecls`：front_open/plotline_open·advance·close/foreshadow_plant·recall·abandon/tension_board 八工具，**全用声明、零硬编码 handler**）；③ `createMcpServer(db, deps, extraTools?)` 加可选 `extraTools` 入口（DT-9 守约：现有 19 工具零改动），生成工具与 `TOOLS` 并列注册。dogfooding 集成测试验证经 server 端到端落库 + 承重墙不破（写经正典原语）+ 坏声明编译期被拒。**front_advance（推进 clock）撞 writeMatch 天花板**（跨 state↔front JOIN 不可映射），v1 不声明（[backlog-core A′ 天花板记项](../06-里程碑与问题/backlog-core.md)），正是 spec §8 预言的「撞 DSL 天花板」。
> - 🚧 **仍欠（下一步）**：团本自定义 `tools:` 段的 manifest 解析 + import 装载（`catalog/import.ts` + 共享 validator，[团本与manifest](团本与manifest.md)）——框架标准库已 dogfooding 跑通，团本侧声明装载是独立后续。

> 工具生成层是**作者声明 → 框架拥有的有限原语**这条线的地基（与声明式工具生成层 spec 同源），保证团本扩展性不以牺牲正典 / 审计 / 可见性为代价。

## 8. 测试策略

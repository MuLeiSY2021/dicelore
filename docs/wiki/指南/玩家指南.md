# 玩家指南

> **面向**：想玩一局的人。从装到玩到复盘，全程任务导向，不讲内部架构。
> **本页是使用视角投影**：深细节（接口契约、组件设计）指回 [设计/](../设计/) 页，不在此重复。
> **状态**：v1 起步内容；标 🔭 的去向尚在路线图（Tauri / 移动端整合包、显式存档 UI），以 [设计/05-现状与计划](../设计/05-现状与计划/) 的实现进度为准。

游戏品类是**文字冒险**：你用自然语言行动，由一个 AI 担任 GM（游戏主持人）描绘世界、裁决后果、推进剧情；规则与随机由引擎机械保证，GM 不能替你「口胡」改数。

## 1. 怎么装

两条路，按你的角色选其一（发版形态见 [ADR-0027](../设计/归档/历史决策日志.md)）：

### A. 整合包（个人玩家，开箱即用）🔭

下载对应平台的客户端整合包（**安卓 / Windows 优先**，Mac/Linux 随后），双击即玩。整合包内含：

- 客户端 UI（桌面端 Tauri 壳 / 移动端原生壳）；
- 一个**本地后端 sidecar**（单二进制，随壳启停，本机 SQLite 存档、本机持密钥）。

**无需** 装 Node / npm、无需开终端。这是「双击即玩」路径。

> 当前实现进度：整合包打包属 fast-follow，v1 竖切先跑通「浏览器连本地后端」。在整合包就绪前，玩家路径与「自托管 + 浏览器」一致（下条）。详见 [玩家客户端 §7/§9](../设计/04-子系统设计/玩家客户端.md)。

### B. 自托管（远程/多人，docker-compose）

适合「一个后端服多个人」的远程部署（场景 B）。按发布的 `docker-compose` 文件起后端，前端走浏览器访问：

- 后端 = 纯 server，集中持密钥、做会话路由；
- 玩家用浏览器打开 web 客户端连这个后端。

部署口径见本 wiki 总 [README](../README.md) 与 [ADR-0027](../设计/归档/历史决策日志.md)②。多租户的鉴权/会话路由细节属未来（[玩家客户端-接口 §0「缝 B」](../设计/04-子系统设计/玩家客户端-接口.md)）。

## 2. 填 API key + baseURL

GM 由一个大模型驱动。开玩前在客户端的**配置页**填两项：

- **API key**：你的模型访问凭据；
- **baseURL（base URL）**：模型服务地址。

填好后客户端会用它驱动 GM。

- **整合包**：key 由本地后端 sidecar 托管，前端只存引用、不存明文 key（[ADR-0027](../设计/归档/历史决策日志.md) + SEC2）。
- **自托管**：key 由服务器集中持有。

> 想自检模型是否连通：配置页有「模型连接测试」（探活 + 辨 401/403）。底层端点见 [玩家客户端-接口 §9.4](../设计/04-子系统设计/玩家客户端-接口.md)（`POST /diagnostics/model-test`）。

## 3. 起一局

1. 进主页：可「继续上次」或从「最近会话」里挑一个。
2. 新开一局时选一个**团本**（Adventure，预制的世界 + 剧本内容包，见 [作者指南](作者指南.md)）。
3. 进入跑团页。未开场时，叙事区底部有一个大金按钮「**点击开始游戏**」——按它由 GM 跑开场回合（团本的开场白驱动，无需你先输入）。
4. 开场叙事流出后，底部出现输入框，你就可以行动了。

> 这是「两层 Play（开场层 / 续玩层）」：未开场=只有 kickoff 按钮，开场后=输入框。详见 [玩家客户端-接口 §9.4](../设计/04-子系统设计/玩家客户端-接口.md)。

## 4. 玩（自由行动 / 明骰 / 选项）

跑团页的几块区域：

- **叙事区**：GM 的散文逐段流出（打字机式）。
- **呈现台**：你的人物卡、世界钟、库存、声望等可见状态；变化会实时回显（如「金钱 +3d100=74 → 77」）。
- **待选项 / 掷骰卡**：到你做选择或掷骰时出现。
- **输入框**：自由文本行动。

三种交互：

| 交互 | 你做什么 | 引擎/GM 做什么 |
|---|---|---|
| **自由行动** | 在输入框写「我推门进去」之类的行动 | GM 接手、描绘后果、必要时裁决 |
| **明骰（点击掷骰）** | 看到掷骰卡时点击触发掷骰（BG3 式：亮 DC / 区间，见证成败） | **点数由引擎算**——你只提供「我掷了」这个动作，篡改不了真值；结果当场亮出并影响剧情 |
| **选项** | GM 给出几个带后果的选项时，点你要的那个 | 你的选择作为下一回合的输入喂回 GM |

> **为什么掷骰你点、点数却不是你定**：参与感（点击、见证）与公正性（点数恒由引擎在点击时计算）是两条正交的轴——见 [ADR-0019 玩家闸控明骰](../设计/归档/历史决策日志.md)。明骰/暗骰、选项物化的完整机制与时序在 [玩家客户端-接口 §8](../设计/04-子系统设计/玩家客户端-接口.md) 与 [MCP工具面 §1](../设计/04-子系统设计/MCP工具面.md)。

## 5. 存读档

游戏状态在**每个回合边界自动存档**（per-turn 快照）——你随时退出，下次「继续上次」即接着玩。这是「随时能玩」卖点的兑现。

- **v1**：自动持久化（自动存档 / 自动恢复最近快照）。
- 🔭 **手动回滚按钮、分支（branch）、重掷（swipe）、死后回溯续命**：留 v2。

回滚机制为什么是「存快照」而非「逆运算/重放」、范围含哪些、为什么不含 rule，见 [ADR-0017 状态回滚=回合快照](../设计/归档/历史决策日志.md)。

## 6. 终局复盘

一局怎么结束、结束后能做什么（[ADR-0026 终局机制](../设计/归档/历史决策日志.md)）：

- **谁敲终局**：由 GM 在叙事判据满足时自调终局（主线收束 / Front 钟满 / 玩家死亡等），框架不靠硬条件硬拦——避免「复活道具在身却被判死」这类误判。
- **你的兜底权**：你可**随时主动结束本局**，防 GM 永不收场。
- **终局后 = 复盘模式**：游戏结束后直接进入复盘——GM 给你「这一局的复盘 + 如果继续你可以……」的行动建议；从复盘模式可**重开新局**。
- 🔭 **回溯续命**（退回死亡前继续，非撤回对话）：依赖快照接线，留 v2。

> v1 终局态没有「结束游戏」按钮、也没有回滚/回溯按钮——终局即复盘 + 重开入口（[ADR-0026](../设计/归档/历史决策日志.md)④）。

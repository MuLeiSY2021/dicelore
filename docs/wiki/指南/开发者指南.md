# 开发者指南

> **面向**：想扩展 / 集成的开发者。DIY 插件、接口规范、开发示例。
> **本页是使用视角投影**：可深引设计页。**确切字段名 / TS 类型以 `packages/shared`（缝 B）与 `packages/interface`（storage-port）实现为准**，本页给的是「该照哪份合同、合同在哪」。
> **结构基底**：仓库是「四根」布局——`frontend/`（UI）、`backend/`（存储 + 服务宿主）、`harness/`（agent 运行时）、`packages/`（纯库）+ `adventures/`（团本内容包），见 [ADR-0028 storage-port 四根重构](../设计/归档/历史决策日志.md)。

## 0. 先认两条缝（解耦诉求相反，别接错）

扩展前先分清两条接口缝（[玩家客户端-接口 §0](../设计/04-子系统设计/玩家客户端-接口.md)）：

| 缝 | 谁 ↔ 谁 | 拓扑 | 机制 | 该不该远程化 |
|---|---|---|---|---|
| **缝 A** | dicelore MCP（agent 工具面）↔ 编排后端 | 永远同机同进程 | 进程内 `onCanonWrite` 回调（HTTP webhook 是跨进程未来形态） | 不需要 |
| **缝 B** | 编排后端 ↔ 呈现 UI（web） | web 可远程、未来多租户 | REST + WS，按 `sessionId` 寻址 | 需要 |

接前端/集成第三方 UI → 你对的是**缝 B**（REST+WS）。写让 GM 直读存储的运行时件 → 你对的是 **storage-port**（不是直连 DB）。

## 1. 接口规范（照之实现的几份合同）

### 1.1 缝 B：REST + WS（前端 ↔ 后端）

**权威页 [玩家客户端-接口](../设计/04-子系统设计/玩家客户端-接口.md)**，类型镜像在 `packages/shared`。要点：

- **动作走 REST**：`POST /sessions/{id}/messages`（自由输入）、`POST /sessions/{id}/choices`（选项）、`POST /sessions/{id}/roll`（明骰触发）；动作返回 `202 {turnId}`，产物走流式（[接口页 §2](../设计/04-子系统设计/玩家客户端-接口.md)）。
- **呈现快照走 REST**：`GET /sessions/{id}/presentation`（全量快照，首屏 / 重连），形状见 [接口页 §1](../设计/04-子系统设计/玩家客户端-接口.md)。
- **GM 产物走流式**：`GET …/ws`（或 `/stream` SSE），消息目录 `narration_commit` / `presentation_delta` / `choices` / `roll_staged` / `roll_committed` / `turn_started` / `turn_ended` / `game_end` / `error`（[接口页 §3+4](../设计/04-子系统设计/玩家客户端-接口.md)）。
- **协议版本**：每条消息带 `protocol: "dicelore.client/N"`；加字段=兼容、改语义/删字段=进位（[接口页 §6](../设计/04-子系统设计/玩家客户端-接口.md)）。
- **诊断 / 浏览端点**：`GET /diagnostics/health`、`POST /diagnostics/model-test`、`POST /diagnostics/mcp-test`、`GET /sessions/:id/browse`、`GET /catalog/:id/files`、`POST /catalog/validate`（[接口页 §9.4](../设计/04-子系统设计/玩家客户端-接口.md)）。

> 实现现状与设计的偏差（哪些端点 ✅/🚧/❌、断点清单）逐条标在 [接口页 §2/§3+4/§10](../设计/04-子系统设计/玩家客户端-接口.md)——联调必读，别照设计假设当下都通了。

### 1.2 storage-port：`SessionBackend`（harness ↔ backend）

agent 运行时（`harness/`）**不直连 store**，只依赖中立包 `packages/interface` 定义的端口（[ADR-0028](../设计/归档/历史决策日志.md)②）。这与 anti-F1 铁律同构：**声明存储意图、后端执行**。

- 端口按领域分组聚合为一个会话级实例：`SessionBackend = Store & Resolver & Snapshots & Catalog & Presentation & Meta & Toolgen`。
- backend 实现端口（`openSessionBackend(db)`，方法绑定 db、不暴露原始 db）；组合根 `backend/server.ts` 按会话注入 harness。
- 写运行时件时收 `SessionBackend` 而非 `db + 直接 import`——这是断模块级循环依赖的关键。

完整端口分组与组合根接线见 [ADR-0028](../设计/归档/历史决策日志.md)；接口定义在 `packages/interface`。

### 1.3 缝 A：`onCanonWrite` 通知缝（MCP → 后端）

MCP 写规范态成功后，把「改了什么」告知后端 → 后端映射成流式 `presentation_delta` / `roll_committed` 推前端：

- **v1 默认 = 进程内回调**：`createMcpServer(db, deps)` 接 `deps.onCanonWrite(evt: CanonWriteEvent)`，工具写成功后**在 handler 外层**回调（不进 `runTool`、引擎零改动）。
- **跨进程未来形态 = HTTP webhook**：`POST {NOTIFY_URL}` + `X-Dicelore-Notify-Token`，payload = `packages/shared` 的 `NotifyPayload`（契约已备、暂无运行时消费者，用于远程 / 自定义 MCP host）。

两形态语义等价、传输不同，对照与实现状态见 [接口页 §5](../设计/04-子系统设计/玩家客户端-接口.md)。

### 1.4 MCP 工具面（GM 调的运行时工具 / 构建期工具）

**权威页 [MCP工具面](../设计/04-子系统设计/MCP工具面.md)**。两套互不交叉：

- **运行时面**（`harness/src/dicegm/mcp/**`）：resolver 族（`resolve_choice` + `resolve_outcome/contest` 各明暗两版）、数据工具（`sheet_*` / `event_*` / `world_*` / `rule_search` / `watcher_*`）、可见性工具（`sheet_show` / `world_show` / `reveal_once`）、`narrate`、`game_end`。入参一律 `.strict()`，出参经 `structuredContent` 回 AI（[MCP工具面 §0–§7](../设计/04-子系统设计/MCP工具面.md)）。
- **构建期面**（`dicelore_build_*`，挂 `LoreSession`）：见 [MCP工具面 §8](../设计/04-子系统设计/MCP工具面.md)。

通用约定关键点：`expr` 全程是字符串、MCP 不解析；AI 只给引用、引擎取真值（anti-F1）；错误信封 code 枚举；in-process 挂载工厂 `createMcpServer(db, deps, extraTools?)`。

### 1.5 团本 pack 格式

内容包的目录布局 / manifest schema / CSV 列规范 / import 映射 → [团本与 manifest](../设计/04-子系统设计/团本与manifest.md)（也是作者侧合同，见 [作者指南 §3](作者指南.md)）。

## 2. DIY 插件 / 扩展

### 2.1 声明式自定义工具（团本作者声明 → 框架编译）

不想手写工具代码？用**声明**（一条 SQL 片段）描述只读视图或 MCP 工具，框架编译成安全的运行时构件（[ADR-0023](../设计/归档/历史决策日志.md)⑥ + [团本构建工具链 §7](../设计/04-子系统设计/团本构建工具链.md)）：

- **`sqlGuard`**：SQL 安全闸（首词分发 + 防注入 / 防多语句）。
- **视图编译（`view`）**：一条 `SELECT` → `CREATE VIEW`（把物理表投影为团本语义视图）。
- **工具编译（`compile`）**：`SELECT` 声明 → 读工具；写声明 → 写工具，但**严格正则匹配正典写原语**（`applyMutations` / `*SetStatus` / `*Upsert` …），匹配不上即拒——防团本绕过正典通道直写、破坏规范态 / 审计 / 可见性。
- **并入工具面**：编译产物经 `toToolDef` 适配成 MCP `ToolDef`，与硬编码工具并列注册进同一 `createMcpServer`（`extraTools` 入口），GM 侧无感（[MCP工具面 §7.2](../设计/04-子系统设计/MCP工具面.md)）。

位置 `backend/src/toolgen/`。**实现现状**：引擎 + 视图层投影 + 框架标准库声明（`backend/src/stdlib/narration.ts`）已 dogfooding 跑通；**团本侧 `tools:` 段 import 装载 🚧 未接**（需 manifest schema + 共享 validator）。单源记于 [团本构建工具链 §7](../设计/04-子系统设计/团本构建工具链.md) 与 [MCP工具面 §7.2](../设计/04-子系统设计/MCP工具面.md)。

> 边界铁律：声明能暴露的写口，只限框架拥有的有限正典原语——团本扩展性不以牺牲正典 / 审计 / 可见性为代价。

### 2.2 自定义 MCP server（周边能力）

接一个外部 MCP（联网检索 / 配图等）给 GM 用——机制与**边界**见 [作者指南 §5](作者指南.md) 与 [玩家客户端 §8](../设计/04-子系统设计/玩家客户端.md)。开发者关注点：自定义 MCP **不得碰规范态**，产出作叙述/note 流回；走权限闸；归 out-of-canon 审计、不发呈现通知、副作用不进快照。

### 2.3 CC hook 契约（运行时三 hook）

GM 运行时复用三个 Claude Code hook 的**纯逻辑**（[玩家客户端 §2.2](../设计/04-子系统设计/玩家客户端.md)）：

| hook | 时机 | 干什么 |
|---|---|---|
| **SessionStart** | 开局 | 注入常驻身份 / 纪律摘要 |
| **UserPromptSubmit** | 回合开始 | 被动 rule 召回 → 注入本轮 |
| **Stop** | 回合末 | ① 物化暂存 choice ② L3 审计本轮 |

host-specific 的只是「怎么注册 hook」（Claude Code 走 `.claude/` + `settings.json`；编排后端走 Agent SDK 的 hook API）；**hook 内的纯逻辑从 adapter 复用、不重写**。时序与 L3 审计细节见 [adapter 与 L3 审计](../设计/04-子系统设计/adapter与L3审计.md)。

## 3. 开发 / 联调示例

起服务跑通缝 B 闭环（详见 [玩家客户端-接口 §9](../设计/04-子系统设计/玩家客户端-接口.md)）：

```bash
# 后端：FAKE_GM 跳过真 LLM，纯脚本 GM，适合联调
DICELORE_FAKE_GM=1 PORT=8787 npm run dev -w @dicelore/backend
# 前端：Vite dev，把 /sessions 代理到 :8787
npm run dev -w @dicelore/frontend
```

- **连通自检**：`GET /sessions/demo/presentation` 应回全量快照（空局＝空 sheets/mechanics、`choices:null`）；浏览器开 `/play` 应建立 `ws://…/sessions/demo/ws`。
- **真 GM**：去掉 `DICELORE_FAKE_GM`，配 `ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN`（Agent SDK 读 env，[ADR-0020](../设计/归档/历史决策日志.md)③）。真 SDK 集成冒烟另设 `RUN_LIVE=1` opt-in（避免常规测试烧 token）。
- **GmDriver 抽象**：编排后端依赖抽象 `GmDriver`；真实现 `AgentSdkDriver` 包 Agent SDK 的 `query()`，测试用脚本化 `FakeGmDriver`——整个回合循环 / WS / notify / 明骰 gate 全程不烧 LLM 可单测（[ADR-0020](../设计/归档/历史决策日志.md)③）。

四个 happy-path 用例（自由输入→叙述 / 状态变更→呈现台 / 明骰 / 选项→下一回合）及当前可跑通的边界见 [接口页 §9.2](../设计/04-子系统设计/玩家客户端-接口.md)。
